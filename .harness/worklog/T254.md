# T254 — per-repo commit counts on /api/github/repos

## What changed
- `src/pages/api/github/repos.js`:
  - Added exported pure helper `parseCommitCountFromLinkHeader(linkHeader)` — parses the
    `rel="last"` page number out of a GitHub `Link` header (per_page=1 → page number == commit
    count), returning `null` when missing/unparseable.
  - After the existing filter+sort step, fan out one `GET /repos/{full_name}/commits?per_page=1`
    call per active (post-filter) repo, using `Promise.all`. Commit count resolution order:
    Link header page number → response body array length (0/1 commits, no Link header) → `null`
    on non-OK response or thrown error (each repo degrades independently; the whole request never
    fails because of one repo's commit-count fetch).
  - Final mapped repo objects now include `commitCount` alongside existing fields.
  - No cache-key/TTL change — commit counts ride along in the same `withApiCache` 4h payload.

- `src/pages/api/github/repos.test.js`:
  - Replaced the single-shape `global.fetch` mock with `mockFetchWithCommits(repoListData,
    commitsResponder)`, which inspects the requested URL and answers either the repos-list call
    or a `/commits?per_page=1` call (with a stubbed `headers.get('link')`).
  - Added unit tests for `parseCommitCountFromLinkHeader` (multi-page header → 42; null/empty →
    null; header with no `rel="last"` → null).
  - Added integration tests: multi-page Link header → `commitCount: 7`; no Link header + 1-item
    body → `commitCount: 1`; failed (403) commits fetch → `commitCount: null` and route still
    returns 200 with the rest of the payload intact.
