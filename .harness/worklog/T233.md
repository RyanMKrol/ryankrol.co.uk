# T233 — fix storeWorkoutInDynamoDB's early-return skipping Exercises sync

## What changed

`src/lib/workoutBackfill.js`:
- In `storeWorkoutInDynamoDB`'s `ConditionalCheckFailedException` branch, the
  `existingItem && existingItem.exercises` case now calls `await storeExercises(workout)`
  before returning `false`. Previously it returned early without ever writing/backfilling
  the corresponding `Exercises` table rows, so a Workouts item with a populated `exercises[]`
  field was permanently exempt from having its per-exercise rows (re)synced by this live
  write path — the confirmed root cause of Workouts/Exercises rows going out of sync.
- Return value (`false`) and all other behavior in that branch unchanged, so
  `backfillWorkouts()`'s `newWorkouts`/`consecutiveExisting` counters are unaffected (the
  workout record itself genuinely didn't change).
- Exported `storeWorkoutInDynamoDB` and `storeExercises` (previously module-private) so the
  sibling one-off scripts (`wdi-consolidate-audit`, `wdi-consolidate-migrate`) can import and
  reuse them instead of maintaining their own reimplementations.

`src/lib/workoutBackfill.test.js`:
- Added a new test case ("still writes missing Exercises rows for an already-existing
  complete workout, without healing the Workouts item") that mocks the workout `PutCommand`
  to reject with `ConditionalCheckFailedException`, the follow-up `GetCommand` to resolve an
  item with a non-empty `exercises` field, and asserts a per-exercise `PutCommand` for the
  Exercises table still fires (i.e. `storeExercises` ran) while `UpdateCommand` is never
  called and the result is not counted as a new workout.
- Existing test cases were left as-is (still pass unmodified).
