# T232 — Decide whether to still pursue a full DynamoDB workout-data wipe + reset

## Status: done. Decision made by the owner interactively — option (a), no wipe.

## Decision

Owner chose **(a): accept the targeted, additive repair as sufficient and drop the full-wipe idea
entirely.** No wipe happens.

## Why this was the right call

By the time this decision was made, the additive repair path had already run to completion and been
verified against production:

- **T233** (done) — `workoutBackfill.js` refactored to export `storeWorkoutInDynamoDB`/
  `storeExercises` and self-heal missing `Exercises` rows on already-existing workouts.
- **T236** (done) — built `backfillMissingExerciseRows.js` to reconstruct missing `Exercises` rows
  from each `Workouts` item's own denormalized `exercises[]` data.
- **T237** (done) — ran the backfill against production, then, on finding the reported symptom
  (`Iso-Lateral High Row (Machine)`) still unresolved, investigated further and found a second, deeper
  root cause: 225 of 1,314 `Exercises` rows (~17%, across 72 workouts) were written in a legacy schema
  shape that DynamoDB's `exercise_name-workout_date-index` GSI silently refused to project. Built and
  ran `migrateLegacyExerciseRows.js` to rebuild those 225 rows from the Workouts table's own data.
  **Verified live**: `GET /api/exercises/history/Iso-Lateral%20High%20Row%20(Machine)` now returns 28
  sessions (was 3), matching the Workouts-table denormalized count exactly.

T237 also actively checked whether a full wipe + Hevy re-fetch would have been safe, and found it
would **not** have been: 3 real, well-formed workouts exist in DynamoDB that were later deleted on
Hevy's side. A full wipe-and-refetch would have permanently lost those 3 — the additive path avoided
that entirely.

## Resolution

No action needed beyond what T233/T236/T237 already did. The reported symptom is confirmed fixed in
production. The full-wipe idea is dropped.
