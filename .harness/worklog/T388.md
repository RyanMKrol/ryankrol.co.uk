# T388 — Decide the fix direction for single-newline markdown spacing

2026-07-15

## Decision

Owner picked **option (b)** — a narrower, `Markdown.js`-scoped preprocessing patch, structurally
similar to T382's `preserveBlankLines` helper.

**What changes:** In `src/components/Markdown.js`'s **block mode only**, detect single newlines
that are NOT already part of a blank-line paragraph break, and convert them into an explicit line
break (e.g. a trailing double-space + newline, or a raw `<br>`) before handing the string to
ReactMarkdown — so a single Enter renders as a visible line break, matching the plain-text/textarea
mental model, without adopting a global `remark-breaks` plugin.

**What stays untouched:**
- **Inline mode** (used by `ReviewCard.js` for album track highlights) is explicitly NOT touched —
  keeps its existing `INLINE_ELEMENTS`/`unwrapDisallowed` behaviour exactly as today.
- T382's existing `preserveBlankLines` blank-line-run preprocessing is NOT undone or duplicated —
  the new single-newline pass must compose correctly with it (no double-counting gaps between the
  two preprocessing steps).
- No `remark-breaks` dependency is added; this is a bespoke, scoped preprocessing function local to
  `Markdown.js`, easy to unit test the same way `preserveBlankLines` already is.

**Rejected alternatives:** (a) site-wide `remark-breaks` — rejected for its blast radius across
every one of the ~25 `<Markdown>` call sites, including already-published review text; (c)
leave rendering as-is and only fix `/tom`'s editor UX/help text — rejected since it doesn't actually
fix the behaviour the owner called wrong, only better explains it.
