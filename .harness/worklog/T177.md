# T177: Fix gym-panel sparkline

## Root cause
`getWorkoutStats()` in `src/lib/workoutQueries.js` filtered `workouts` to a rolling 30-calendar-day
window BEFORE sorting/slicing to the last 10. With a multi-year logging history, any recent gap in
gym attendance collapsed `recentActivity` to ~1-2 items, which combined with `.home-sparkline-bar`'s
`flex: 1` stretched each bar to fill 50-100% of the row — the "two flat solid-orange boxes" bug.

## Fix
- `src/lib/workoutQueries.js`: extracted `selectRecentActivity(workouts, limit = 10)` — a pure
  helper that filters out entries missing `workoutDate`, sorts desc, and slices to `limit`, with no
  calendar-window filter. `getWorkoutStats()` now calls it against the full scanned `workouts` array.
  Removed the dead `thirtyDaysAgo`/`thirtyDaysAgoStr` variables.
- `src/styles/globals.css`: added `max-width: 32px` to `.home-sparkline-bar` as a defensive cap for
  low-count degenerate cases, plus `:nth-child(2)`-`(10)` `color-mix()` rules stepping from
  `var(--accent-albums)` (oldest, left) to `var(--color-chart-coral)` (most recent, right) — all
  theme-token-driven so it still re-themes correctly under `html.matrix-active`.
- No change needed to `src/pages/index.js` — the render/map logic was already correct; only its
  upstream data source was broken.

## Test
Added `src/lib/workoutQueries.test.js` covering: more-than-limit truncation + ordering,
fewer-than-limit pass-through, a date range with nothing in the last 30 days (the exact regression),
and workouts missing `workoutDate` being filtered rather than crashing the sort.
