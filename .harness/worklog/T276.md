# T276: Wire the shared Markdown editor into all 5 review 'edit' forms

## What changed

Like-for-like swap of the plain `<textarea className="collection-form-textarea" ...>` for
`MarkdownEditor` (built in T274) across the 5 review edit detail pages:

- `src/pages/reviews/movies/edit/[id].js` (field `gist`)
- `src/pages/reviews/tv/edit/[id].js` (field `gist`)
- `src/pages/reviews/books/edit/[id].js` (field `overview`)
- `src/pages/reviews/albums/edit/[id].js` (field `highlights`)
- `src/pages/reviews/perfumes/edit/[id].js` (field `description`)

Each swap is import + tag rename only — same `id`/`name`/`value`/`onChange`/`className`/
`placeholder`/`required` props as before, no change to `formData` state, the fetch-then-populate
effect, the `GetCommand`-then-`PutCommand` update flow (handled server-side, untouched here), or
the delete button.

## Tests added

One co-located test per edit page (`src/__tests__/pages/reviews/<type>/edit.test.js`), mirroring
the T275 add-form test pattern but also mocking `next/router`'s `query.id` and `global.fetch` so
the async fetch-by-id effect resolves and the form renders populated. Each test asserts the
MarkdownEditor toolbar (Bold/Italic/Bullet list/Link) is present and that the review-text field is
pre-populated with the fetched value — this covers the "editor loads pre-populated with existing
review text" requirement from Done-when without touching real DynamoDB.
