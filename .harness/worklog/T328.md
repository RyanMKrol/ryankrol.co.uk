# T328 — unify the Edit-header/card box on all 5 review edit list pages

2026-07-08

## What I did
`.review-edit-card` (`src/styles/globals.css`) previously only laid out its children in a column —
it drew no border/background of its own, so the `.review-edit-header` strip (with its own
`border-radius` top corners + `margin-bottom: 0.75rem`) visually floated above the variant card
(`.spine-cover-card` / `.square-cover-card` / `.poster-banner-card` / `.perfume-v6-card`) below it,
which drew its own competing full border/radius/shadow — two separate boxes with a gap and a
mismatched-corner seam between them.

Fixed in `src/styles/globals.css`:
- `.review-edit-card` now owns the single visible box: added `border`, `border-radius`, `background`,
  and `overflow: hidden` (matching the tokens the variant cards themselves use — `var(--color-card)`,
  `var(--border-width) solid var(--color-ink)`, `var(--radius-card)`).
- `.review-edit-header` lost its `margin-bottom` (the parent's `overflow: hidden` now clips it flush)
  and its own `border-radius` (redundant — the parent's radius + `overflow: hidden` clips the header's
  top corners). Its `border-bottom` stays, now reading as an internal divider between the header strip
  and the card content rather than the bottom edge of a separate box.
- Added a scoped override — `.review-edit-card .spine-cover-card, .review-edit-card .square-cover-card,
  .review-edit-card .poster-banner-card, .review-edit-card .perfume-v6-card { border: none;
  border-radius: 0; box-shadow: none; }` — so the nested variant card stops drawing its own competing
  border/radius/shadow inside the edit page's unified box. Scoped to `.review-edit-card` descendants
  only, so the public (non-edit) list pages — which render the same variant card classes directly,
  with no `.review-edit-card` ancestor — are completely unaffected.

In `scripts/_visual-harness.mjs`: added 5 new `PAGES` entries (`reviews-movies-edit`,
`reviews-tv-edit`, `reviews-books-edit`, `reviews-albums-edit`, `reviews-perfumes-edit`) for
`/reviews/<type>/edit`, `waitFor: ['.review-edit-card']`. None of these routes had a `PAGES` entry
before this task — they're not password-gated to view, and their `useEffect` fetch hits the same
`/api/reviews/<type>` fixture endpoints already wired for the public list pages, so no new fixture
data was needed.

No other files touched.
