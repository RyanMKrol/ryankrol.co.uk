# T267 — loop.sh auto-deploy on empty backlog

## Status: done. Built interactively (needs-human, harness-scoped), per the "never let the
autonomous loop modify itself" convention.

## What changed

- `.harness/loop.sh`: added `site_touched_since_last_deploy()` (resolves the highest-numbered
  `done` "Deploy pending site changes to production" task, finds its `mark done` commit via
  `git log --all --grep`, diffs `src`/`public` against it — fails open, returning "deploy
  warranted", if no prior deploy task or its commit can't be resolved) and `run_auto_deploy()`
  (mirrors T193's sanctioned `vercel --prod` + exit-code + `curl` verification procedure). The
  empty-backlog branch (where `select_task` returns nothing) now calls
  `site_touched_since_last_deploy` before exiting and runs `run_auto_deploy` when it returns true.
- `.harness/loop-auto-deploy.test.sh` (new): hermetic test, throwaway `mktemp -d` git repo,
  `LOOP_SOURCE_ONLY=1 source .harness/loop.sh` to load just the functions — never touches this
  real checkout, never invokes `vercel`/`curl`. Covers: no prior deploy task → fails open; no
  src/public diff since the last deploy commit → skip; a real src/ change since then → deploy
  warranted; a change scoped only to `.harness/` → correctly ignored (bonus case beyond the
  spec's minimum, validating the pathspec scoping).
- `.harness/CLAUDE.md`: added a paragraph to the deploy-task convention section documenting this
  as a belt-and-suspenders fallback — explicitly not a replacement for the existing
  authoring-time convention.

## Verification

- `bash -n .harness/loop.sh` — passes.
- `DRY_RUN=1 .harness/loop.sh` — still runs and prints its normal dry-run output
  (`DRY-RUN → would build: T257`), proving the script parses/loads cleanly end to end.
- `bash .harness/loop-auto-deploy.test.sh` — all 4 cases pass (3 required + 1 bonus).
- Diff confirmed scoped to exactly `.harness/loop.sh` + `.harness/CLAUDE.md` (task's declared
  scope) plus the new test file (always-allowed).

## Note

`run_auto_deploy()` itself (the real `vercel --prod` + `curl` call) was not exercised live as
part of this task — it only fires in production when the loop's backlog genuinely empties out,
and the spec explicitly scopes verification to the hermetic test + dry-run/syntax checks, not a
live deploy trigger.
