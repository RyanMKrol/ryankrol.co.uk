# T368: Fix Last.fm thumbnail never saving for albums/vinyl (images shape bug)

## What was done
1. **Added new helper `pickLargestFromImageMap(images)`** to `src/lib/lastfm.js` (exported) that reads from the object-keyed images shape (`{ extralarge: "...", large: "...", ... }`) instead of the old array shape, following the same `IMAGE_SIZE_ORDER` priority list.

2. **Replaced broken expressions** at 3 call sites:
   - `src/pages/api/reviews/albums/add.js` line 32
   - `src/pages/api/reviews/albums/update.js` line 47
   - `src/pages/api/vinyl/add.js` line 33
   
   Changed from: `lastfm?.images?.[lastfm.images.length - 1]?.['#text'] || ''`
   Changed to: `pickLargestFromImageMap(lastfm?.images) || ''`

3. **Fixed test mocks** in both `add.test.js` files:
   - Changed `images: [{ '#text': 'https://...' }]` (old array shape) to `images: { extralarge: 'https://...', large: 'https://...' }` (new object shape)
   - Added assertion: `expect(putCall[0].input.Item.thumbnail).toBe('https://last.fm/cover.jpg')`

## Verification
- ✅ `npm run lint` — no new errors
- ✅ `npm test` — all 480 tests pass, including updated album/vinyl add tests
- ✅ `npm run build` — succeeds
- ✅ Manual test: `pickLargestFromImageMap()` correctly prioritizes extralarge > large > medium > small, returns null for empty/null input
- ✅ Visual check (hermetic): `reviews-albums` + `vinyl` pages render album/vinyl cover thumbnails correctly (9 album captures + 3 vinyl captures all PASS with no broken-image regressions)

## Root cause (confirmed)
`mapAlbumInfo()` builds `images` as an object keyed by size (lines 123-129 of `lastfm.js`). The three thumbnail-save call sites were using `.length - 1` (array syntax) against this object, which returns `undefined - 1 = NaN`, so `images[NaN]` was always undefined → thumbnail always saved as `''`. Confirmed 0/48 AlbumRatingsV3 and 0/190 VinylCollectionV2 records had non-empty thumbnails from this path in production.
