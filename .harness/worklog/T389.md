# T389: Audit all <Markdown> call sites for rendering inaccuracies

## What was done

Audited every real `<Markdown>` render path and every `<MarkdownEditor>` import site in the repo.

**Finding: the spec's "~25 call sites" figure counts files that import either `Markdown` or
`MarkdownEditor`, but the two components are very different in scope.** `MarkdownEditor` is just a
raw textarea + B/I/List/Link toolbar — it never calls `ReactMarkdown` and renders no markdown
itself. Grepping for actual `<Markdown ...>` renders (the thing this task is about) turns up only
**4 files**, not ~17-25:

1. `src/components/ReviewCard.js` — 4 render paths: `SpineCoverCard` (block), the `poster-banner`
   variant (block), `SquareCoverCard` highlights (inline), and the default/unstyled card (block,
   currently unreachable in production — every review type sets an explicit `styleVariant`).
2. `src/components/perfumeVariants/Variant6Hybrid.js` — perfume description (block).
3. `src/pages/tom.js` — Top-of-mind editor preview pane (block).
4. `src/pages/index.js` — homepage Top-of-mind panel (block).

The 10 review add/edit forms (`src/pages/reviews/**/{add.js,edit/[id].js}`) and the hot-takes
add/edit forms only render `<MarkdownEditor>` with **no** live `<Markdown>` preview pane — contrary
to the spec's Overview, which described them as having "live preview panes." Only `tom.js` actually
pairs the editor with a preview. This is a spec inaccuracy, not a code bug; nothing to fix there,
and there's no markdown *rendering* to audit on those 10+2 pages since none is being rendered.
Logged as idea #1 below (a real, but out-of-scope, feature gap: those forms could plausibly benefit
from adding a preview, mirroring tom.js).

### Real inaccuracies found and fixed (small, self-contained)

1. **Inline mode silently dropped GFM strikethrough.** `src/components/Markdown.js`'s inline
   `ReactMarkdown` allowed `del` in `INLINE_ELEMENTS` (implying strikethrough support) but never
   passed `remarkPlugins={[remarkGfm]}`, so `~~text~~` never parsed as strikethrough in inline mode —
   it rendered as the literal string `~~text~~`. Block mode already had `remarkGfm` and worked
   correctly. **Fix:** added `remarkPlugins={[remarkGfm]}` to the inline `ReactMarkdown` call too.
   Confirmed with a scratch test before the fix (`~~struck~~` rendered as plain text in inline mode,
   as `<del>` in block mode) — the fix makes both modes consistent.
2. **Inconsistent list styling across block-mode call sites.** T373 previously added a scoped
   list-indent fix (`ol`/`ul` `padding-left: 1.4rem` + explicit `list-style-type`) for
   `.spine-cover-snippet` only (the books card). `.poster-banner-snippet` (movies/tv cards),
   `.perfume-v1-description` (perfume description), and `.review-text` (the default/unreachable
   card) had no such rule, so a Markdown list there would fall back to the browser's default 40px
   indent instead of the design system's own list styling — a real, visible inconsistency between
   call sites, exactly the class of issue the spec calls out as an example finding. **Fix:** applied
   the identical T373 pattern to all three remaining containers in `src/styles/globals.css`.

### Non-trivial issues found — NOT fixed here, logged as ideas instead

Per the task's own "if a found issue is non-trivial... capture as a new idea" instruction, appended
to `.harness/tracking/IDEAS.jsonl` (gitignored, so not part of this commit):

- **Idea #1** — Add a live Markdown preview pane to the 5 review add/edit forms (10 files) and
  hot-takes add/edit (2 files), mirroring the recipe already used by `tom.js`. A real UX gap (users
  can't see rendered markdown until they save), but it's a new-feature addition touching 12 page
  files entirely outside this task's `scope`.
- **Idea #2** — GFM tables (enabled by `remarkGfm` in block mode) have no CSS styling anywhere in
  `globals.css`, so a Markdown table in any block-mode call site would render with unstyled
  browser-default borders, inconsistent with the Collection design tokens. Not currently known to be
  exercised by real content (no toolbar affordance nudges toward tables). Needs a genuine new CSS
  design decision (Collection-themed table styling), not a small copy of an existing pattern like the
  list-indent fix was — so left as an idea rather than fixed here.
