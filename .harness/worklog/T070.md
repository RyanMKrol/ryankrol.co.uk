# T070 — Backfill-from-API button on the movies edit page

## What changed
- `src/pages/reviews/movies/edit/[id].js`: added a `MetadataBackfillModal` (from T066) below the
  title field, before rating. `handleBackfillSearch` calls
  `GET /api/tmdb/search?query=...&type=movie` using the form's current title. `handleBackfillConfirm`
  maps the selected TMDB result onto `formData.tmdbId`, `mediaType`, `posterPath`, `tmdbOverview`
  (from the result's `overview`), and `tmdbDate` (from the result's `date`) — `rating` and `gist`
  (the user's own review text) are never touched. A poster `<img>` renders when `posterPath` is set,
  via `tmdbPosterUrl` from `src/lib/tmdb.js`. The existing Save button still submits `formData` (now
  including the extra metadata keys) via `POST /api/reviews/movies/update`, which already accepts
  them (T067). No auto-submit on confirm.

## Verified
- `npm run lint` — 0 errors, 3 warnings (2 pre-existing in unrelated files; 1 new
  `@next/next/no-img-element` warning on the new poster `<img>`, tolerated as a warning not an
  error — matches the spec's requirement of a visible poster image).
- `npm test` — 139/139 passed.
- `npm run build` — succeeded, `/reviews/movies/edit/[id]` compiles (1.93 kB).
- Manual UI click-through not run headless (would require a live TMDB call / real movie record).
  Reviewed code paths against `mapTmdbResult` shape (`src/lib/tmdb.js`) and the books edit page's
  established `MetadataBackfillModal` calling convention to confirm field names line up.

## Scope
Only `src/pages/reviews/movies/edit/[id].js` was modified (plus this worklog). No files under
`src/components/**` needed changes — `MetadataBackfillModal` from T066 was reused as-is.
