# T105 — client-side search-button cooldown

## What changed
Added a `cooldown` boolean (`useState(false)`) to each of the three search-trigger components.
On click, `search()` now sets `cooldown` true and clears it via `setTimeout(..., 2000)`, in
addition to (not instead of) the existing `searching` in-flight state. The button's `disabled`
expression now includes `cooldown` alongside its existing conditions:

- `src/components/TmdbSearch.js` — `disabled={searching || cooldown || !query.trim()}`
- `src/components/BookSearch.js` — `disabled={searching || cooldown || !title.trim()}`
- `src/components/LastfmAlbumSearch.js` — `disabled={searching || cooldown || loadingInfo || !titleQuery.trim()}`
  (preserved the existing `loadingInfo` condition)

Button label logic is unchanged — only `searching` drives the "Searching…" text, per spec.

Added a Conventions bullet to root `CLAUDE.md` documenting the 2s client-side cooldown layer
alongside the existing per-route backend rate limit (`src/lib/rateLimit.js`).
