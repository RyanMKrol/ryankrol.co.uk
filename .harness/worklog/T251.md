# T251 — perfumes update: preserve original date via GetCommand, add editedDate

## What changed
- `src/pages/api/reviews/perfumes/update.js`: mirrored the `movies/update.js` pattern.
  - Added a `GetCommand` against `PERFUME_RATINGS_TABLE` keyed by `originalId`, run BEFORE the
    `idChanged` delete branch so the original item is still readable.
  - `preservedDate = existingReview.Item?.date || <today>` and `editedDate = <today>` computed
    unconditionally on every update.
  - `reviewData.date` now uses `preservedDate` (server-derived, no longer trusts `req.body.date`)
    and a new `editedDate` field is stored alongside it.
  - Dropped the now-unused `date` field from the destructured `req.body`.
- Added `src/pages/api/reviews/perfumes/update.test.js`, mirroring `movies/update.test.js`'s
  structure (mocked `docClient`/`clearApiCache`, `mockRes()`, `todayDDMMYYYY()`), with a
  realistic perfume request body satisfying `src/lib/perfumes.js` validators. Two cases:
  preserves an existing `date` from the `GetCommand` mock + stamps `editedDate` to today; and
  falls back to today for `date` when no `Item` is returned, still stamping `editedDate`.
- Docs: `README.md`'s `PerfumeRatings Table` schema block now documents `editedDate` after
  `date`. `CLAUDE.md`'s Data model row for `PerfumeRatings` appends `, editedDate`.

## Verification (observed, no live external calls)
- `npm run lint` — 0 errors, 6 pre-existing warnings (img/exhaustive-deps), none new.
- `npm test` — 49 suites / 310 tests passed, including both new perfumes update tests.
- `npm run build` — succeeded; `/reviews/perfumes/edit/[id]` page size unaffected (that page was
  intentionally left untouched — it still sends `date: originalData.date` in the request body,
  which the server now ignores in favour of its own `GetCommand`-derived value).
