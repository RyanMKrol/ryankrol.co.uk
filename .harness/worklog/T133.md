# T133 — Throttle/cooldown loop.sh's pushes to main to avoid tripping Vercel's build rate limit

Built interactively (owner-directed session), per `gate: "needs-human"` — never routed through
`.harness/loop.sh`.

## What was built

- New shared helper `throttled_push()` in `.harness/loop.sh`, replacing every direct
  `git push origin "HEAD:$MAIN_BRANCH"` call site except one deliberate exemption:
  - `mark_done()`, `reconcile_overlays()`, `block_task()`, and the main per-task integration push
    now all route through `throttled_push()`.
  - The CI-red auto-revert push is deliberately left as a direct `git push` (with a one-line
    comment explaining why): restoring a broken `main` is more urgent than pacing, and reverts
    only fire after a red CI — not during a successful-task burst — so they don't meaningfully add
    to the burst risk this helper guards against. This is a 5th call site the original spec didn't
    anticipate (it named 4); the spec explicitly permits an exempt site with a one-line comment,
    which this satisfies.
- `throttled_push()` enforces a minimum wall-clock gap (`PUSH_COOLDOWN_SECONDS`, default 300s) since
  the previous SUCCESSFUL push, sleeping out the remainder if called too soon. Only a push that
  actually succeeds updates the timestamp. The first push of a fresh checkout (no prior timestamp)
  is never delayed.
- Cooldown state persists to a new gitignored file, `.harness/worklog/.last-push-ts`, so it survives
  a `loop.sh`/`supervise.sh` restart mid-burst (added to `.gitignore` alongside `.failures.buf`).
- New knob `PUSH_COOLDOWN_SECONDS` (default 300s) documented in `.harness/harness.env`.
- Documented in `.harness/HARNESS.md` (new §7a, after the usage-limit-backoff section) and
  cross-referenced from the CLAUDE.md incident log this task originated from.

## Verification performed

- `bash -n .harness/loop.sh` and `bash -n .harness/harness.env` — both pass.
- `DRY_RUN=1 bash .harness/loop.sh` — still runs end to end and selects the next eligible task
  (T080), confirming nothing broke in the surrounding script.
- Hermetic smoke test (ad hoc, not committed — this task is `expectsTest: false`, and the spec
  itself notes this can't be exercised end-to-end without a live task burst): built a scratch git
  repo + bare remote in `mktemp -d`, sourced `loop.sh` with `LOOP_SOURCE_ONLY=1`, and called
  `throttled_push` directly:
  - First push (no prior timestamp): 0s delay, as expected.
  - Second push immediately after, `PUSH_COOLDOWN_SECONDS=3`: correctly slept ~3-4s before pushing;
    log line printed; `.last-push-ts` written.
  - Restart-survival: killed the process, started a FRESH one 1s later with `PUSH_COOLDOWN_SECONDS=5`
    — correctly slept the remaining ~4s, confirming the persisted-file state (not an in-process
    variable) is what's actually being read.
- `grep -n "push origin" .harness/loop.sh` — confirms only 2 direct `git push origin` calls remain:
  `throttled_push()`'s own internal implementation, and the deliberately-commented revert exemption.
  Every other call site routes through `throttled_push`.

## Design choices made while building (per the spec's "use judgement, record the choice")

- `PUSH_COOLDOWN_SECONDS` default: 300s (5 min), matching the spec's suggested starting point.
- State file: `.harness/worklog/.last-push-ts`, mirroring the existing `.failures.buf` gitignored
  scratch-state pattern exactly.
- `throttled_push()` suppresses git's own push stderr internally (`2>/dev/null`), matching the
  majority (3 of the original 4 named) call sites' existing convention — the one call site that
  previously let git's raw stderr through (the main integration push) now also has it suppressed;
  this is a minor, deliberate behavior unification, not a diagnostic regression, since that site
  already logs an explicit `WARN`/`push to main failed` message on failure via the `if !` branch.

Not built: a post-push Vercel-deploy-status check (that was the alternative option the owner
explicitly rejected in favor of this cooldown approach) and any change to the CI-gate/revert
DECISION logic (only its push call site's participation in throttling was considered, per the
spec's "do not touch wait_ci_green / red-CI revert" instruction).
