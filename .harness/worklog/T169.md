# T169 — cross-cutting visual audit + Matrix regression check

## Method

Read every CSS custom property referenced anywhere in `src/styles/globals.css` and
diffed it against what's actually defined in `:root`. Also grepped `src/components`
and `src/pages` for hardcoded hex values and stray `font-family` declarations.
Dev server had no browser tooling available in this headless environment (no
chromium/electron binary) — verified visually via `npm run build`, code tracing of
the CSS cascade for every changed selector, and `curl` 200-checks on the main
routes, rather than screenshots. Flagging this per the harness's "verify by
reading code + build" fallback.

## Inconsistencies found and fixed

**Root cause: ~50 CSS custom properties used by real selectors (`.form-input`,
`.form-button`, `.error-message`, `.success-message`, `.pagination-page`,
`.filter-button`, `.now-playing`, scrollbar/selection styling, `.container`,
`MetadataBackfillModal`) were only ever defined inside `html.matrix-active`,**
left over from the pre-Collection multi-theme system that CLAUDE.md says was
"fully retired." Outside Matrix mode these vars resolved to nothing, so every
one of those selectors silently lost its border-radius/color/shadow/font and
fell back to browser defaults — sharp corners, invisible/default-black text,
missing borders. This is exactly the "bolted-on legacy form in an otherwise
redesigned site" the task asked to check for, and it affected far more than
just the 6 add/edit forms:

- `MetadataBackfillModal` (the "Backfill from API" modal used by movies/tv/books
  reviews) — completely unstyled in Collection mode (`--color-text`/`--color-border`/
  `--color-accent`/`--color-bg` all undefined outside Matrix).
- `.form-button`/`.form-input`/`.error-message`/`.success-message` — used by
  `PerfumeCharacteristics`, `TmdbSearch`, `BookSearch`, `LastfmAlbumSearch`, and
  the workouts/exercises "load more" buttons.
- Pagination active-state, filter-button active-state, the Now Playing widget,
  page scrollbars/selection color, and `.container`/`.review-container` padding.
- `--font-heading` (used by `.review-title`, home-nav-link-large, etc.) and
  `--spacing-container-padding` were referenced but **never defined anywhere**,
  Matrix mode included — pure dead/broken references.

### Fix (shared tokens, not per-page patches)

Added a "Legacy multi-theme variable names (T169)" block to `:root` in
`globals.css` that aliases every one of these old names to the Collection
tokens from T142 (e.g. `--color-text: var(--color-ink)`, `--radius-base:
var(--radius-pill)`, `--btn-bg: var(--color-ink)` + white `--btn-color`,
`--font-heading: var(--font-display)`). `html.matrix-active` already redefines
the subset it cares about, so Matrix mode is unaffected — the fix only changes
what non-Matrix pages resolve to.

Three selectors reuse the same var for both an accent background AND the text
color sitting on top of it (`.pagination-page.active`, `.filter-button.active`,
`.now-playing`). Aliasing `--color-text` straight to ink would have put dark
ink text on a dark ink/purple background in these three spots. Introduced
dedicated `--color-pagination-active-text` / `--color-filter-active-text` /
`--color-now-playing-text` tokens (white at root, kept at the existing neon
green in `html.matrix-active`) instead of reusing `--color-text` for the
inverted case.

Rewrote `MetadataBackfillModal.js`'s inline `<style jsx>` to use the Collection
tokens directly (`--color-card`, `--radius-card`, `--color-ink`, etc.) rather
than the legacy names, matching the card/border/radius treatment used by
`.collection-form-card` and its siblings.

### Checked and left alone (not bugs)

- `Badge.js`/`Pill.js`/`StatBlock.js` hardcode `color: '#FFFFFF'` for text on a
  dynamic `accentColor` background — no such "always-white-on-accent" token
  exists in the palette, and all three do it identically, so this is
  consistent, not drifted.
- `CoverTile.js`'s `GRADIENT_PAIRS` hardcodes hex — the "from" stop matches
  each section's `--accent-*` value exactly, but the "to" (lighter tint) has
  no corresponding token to reference, so hardcoding both as a self-contained
  literal pair is reasonable here.
- `MatrixRain.js`'s canvas `fillStyle` hex values — canvas 2D drawing can't
  consume CSS custom properties as directly/cheaply per-frame; these are the
  easter egg's own rendering constants, out of scope for the Collection
  token system.
- `_document.js`, `projects/index.js`, `dev/cache.js` hex hits were out of my
  scope (pages, not components/styles/hooks) — spot-checked and none looked
  like new drift from T149-T164; left alone.

## Matrix-mode regression check

Could not click through a live browser (no chromium/electron in this headless
box). Verified by reading the cascade instead:
- `html.matrix-active` still defines its own `--color-text`, `--color-border`,
  `--color-accent`, `--radius-base`, `--btn-bg`, etc. — none of the new `:root`
  aliases can leak through, since `html.matrix-active` selectors have higher
  specificity than `:root` for the same custom property and are declared later
  in the cascade (existing, already-working pattern — this file already relies
  on it for `--color-bg`/`--font-display`/etc).
- The three new inverted-text tokens (`--color-pagination-active-text` etc.)
  were given matching neon-green matrix values, so Matrix mode's pagination/
  filter/now-playing active states render exactly as before.
- `MatrixRain.js`/`CRTOverlay.js`/`useMatrixActive`/Konami-code activation and
  the `sessionStorage('matrix-active')` persistence in `_document.js` were not
  touched by this task at all (out of scope, and no reason to touch them).
- `npm run build` succeeded, confirming no CSS/JS syntax errors from the edit.

**This verification is code-level, not a browser click-through** — flagging
per harness rule 2b since no browser tooling was available in this run.
