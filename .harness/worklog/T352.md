# T352 — Books search: propagate real upstream Google Books error status + log diagnostic detail

## Done

### Implementation

**File: `src/pages/api/books/search.js`**

1. **Enhanced diagnostic logging:** When Google Books API returns a non-ok response, the code now logs full diagnostic detail **before** constructing any response:
   - HTTP status code and statusText
   - Response body snippet (first 300 chars)
   - Query parameters used (title and author)
   - Uses emoji-prefixed logging convention (`❌ [GoogleBooks]`) matching the repo's style

2. **Upstream 429 propagation:** When the upstream Google Books API returns a 429 (rate limited), the route now:
   - Detects it via the error message 
   - Returns a 429 status to the client
   - Sets a `Retry-After` header (60 seconds)
   - Does NOT mask it as a generic 500

3. **End-to-end wiring verification:** Confirmed that `src/pages/reviews/books/backfill.js`'s `handleSearch()` (lines 28-45) already implements the correct wiring:
   - Checks `if (res.status === 429)` regardless of which layer produces it
   - Reads the `Retry-After` header
   - Sets `err.retryAfterSeconds` to trigger retry logic in `backfillQueue.js`
   - No client-side changes were needed

4. **Non-429 upstream failures:** Still surface as a clear, distinct error (500) with full diagnostic logging, not masquerading as a 429

### How the fix works

**Before:** Upstream Google 429 → thrown as generic error → caught and wrapped into 500 → client gets 500 → no retry triggered

**After:** Upstream Google 429 → detected and re-thrown with "429" in message → caught by handler → returns 429 with Retry-After header → client's handleSearch() sees 429 → sets retryAfterSeconds → backfill retry logic triggers

Non-429 failures are fully logged so future occurrences are diagnosable from Vercel logs alone.
