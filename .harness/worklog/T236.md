# T236 — backfill missing Exercises-table rows

## What was built

`src/scripts/backfillMissingExerciseRows.js`:
- Scans the full `Workouts` table.
- For every item with a non-empty `exercises[]`, queries the `Exercises` table's
  `workout_id-index` GSI (same pattern as `backfillIncompleteWorkoutMetrics.js`'s
  `getExerciseRowsForWorkout`) and compares against the expected `exercise_id`s
  (`${workout.id}_${index}`) via the pure, exported `missingExerciseIndexes(exerciseCount,
  existingRows)`.
- Dry run (default): reports total workouts scanned, total missing exercise-row count, and a
  per-workout breakdown for any workout missing > 0 rows. No writes.
- `LIVE=1`: for every workout with at least one missing index, calls the shared
  `storeExercises` (from `../lib/workoutBackfill.js`) with the Workouts item's own shape
  (`{id, start_time, end_time, title, exercises}`). `storeExercises` already does a
  per-exercise conditional `PutCommand` (`attribute_not_exists(exercise_id)`), so existing rows
  are left untouched and only genuinely missing rows get created. `sleep(150)` between
  workouts, matching the existing rate-limiting convention.
- No `DeleteCommand` anywhere — purely additive, confirmed via `grep`.

Added npm script `workout:backfill-missing-exercises`, matching the `workout:backfill-metrics`
naming convention.

## Tests

`src/scripts/backfillMissingExerciseRows.test.js` covers `missingExerciseIndexes` with: no rows
missing, some rows missing, all rows missing, and a workout with zero exercises.
