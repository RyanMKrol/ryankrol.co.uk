# T132 — hide TMDB-match error on TV add form until first submit attempt

## What changed
`src/pages/reviews/tv/add.js`:
- Added `hasAttemptedSubmit` state (default `false`).
- `handleSubmit` now sets `hasAttemptedSubmit(true)` as its first statement (right after
  `e.preventDefault()`, before the `!tmdbMatch` guard), so any submit attempt — success or
  failure — flips the flag.
- Inline error paragraph next to `TmdbSearch` now renders on
  `hasAttemptedSubmit && !tmdbMatch` instead of `!tmdbMatch` alone.

## Not touched (by design)
- Top-of-form `message`/`messageType` banner — already gated correctly via `setMessage` calls.
- `handleSubmit`'s own `if (!tmdbMatch) { ... return; }` validation guard — unchanged.
- Submit button's `disabled={loading || formData.rating === 0 || !tmdbMatch}` — unchanged.
- `src/pages/reviews/movies/add.js` — out of scope, covered by a separate task.

## Verification
- `npm run lint`: 0 errors, 3 pre-existing warnings (unrelated files), no new warnings.
- `npm test`: 34 suites / 243 tests passed.
- `npm run build`: succeeded, `/reviews/tv/add` compiled as a static route.
- Manual/code-read check: on initial mount `tmdbMatch` is `null` and `hasAttemptedSubmit` is
  `false`, so the inline error paragraph's condition is `false` — it does not render. After a
  submit attempt with no match selected, `hasAttemptedSubmit` becomes `true` and the paragraph
  renders alongside the existing message banner. Selecting a match via `TmdbSearch`'s `onSelect`
  sets `tmdbMatch`, making the condition `false` again, so the paragraph disappears.
- Did not run `npm run dev` interactively (headless environment) — verified via code reading
  and the build output instead, per harness rule against unnecessary live/manual steps beyond
  what's needed to confirm the JSX condition.
