# T104 — rate-limit GitHub repos route

Added `checkRateLimit`/`getClientIp` gate to `src/pages/api/github/repos.js`, mirroring the
pattern already used on the Last.fm/TMDB/Books routes: 20 req/60s per IP, keyed
`github-repos:<ip>`, checked right after the method guard and before the try block. On block,
sets `Retry-After` and returns 429 with a JSON `message`; the GitHub `fetch` loop is never
entered (verified in the new test).

Created `src/pages/api/github/repos.test.js` (didn't exist before) using the
`jest.mock('../../../lib/rateLimit', ...)` + `mockRes()` pattern from
`src/pages/api/lastfm/album-search.test.js` (closer analogue than `reviews/movies/add.test.js`
since both are GET-only external-API proxies). Two cases: under-limit GET succeeds (200, fetch
called), rate-limited GET returns 429 + Retry-After header and never calls fetch.

Updated CLAUDE.md's file map row for `src/pages/api/github/repos.js` to note the rate limit.
