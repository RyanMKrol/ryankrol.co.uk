# T095 — Per-spot spray count UI — make it joyful

## Approach

Replaced the six application-spot checkboxes with a per-spot "spray bottle" tap counter
(`ApplicationSpotsSprayer` in `src/components/PerfumeCharacteristics.js`, shared by add and
edit forms — consistent with this repo's convention of keeping perfume-form widgets in that
component file, alongside `LongevitySlider` / `ProjectionSlider` / `SeasonsCheckboxes`).

Each spot renders as a card with a 🧴 button. Tapping it:
- increments that spot's spray count and shows it ("2 sprays" / "no sprays"),
- triggers a `💦` emoji that pops up and fades out (`spray-poof` keyframe) plus a small
  squeeze animation on the bottle itself (`spray-squeeze` keyframe), so every tap gets a
  quick, tactile bit of feedback rather than a static number changing,
- highlights the card border/background in the theme's accent colour once a spot has at
  least one spray, so it's visually obvious at a glance which spots are in use.
- right-click removes a spray (kept as a discoverable but non-primary affordance — decrementing
  isn't the common path, so it doesn't get its own button).

This stays consistent with the site's existing look: reuses `.form-label`/`.form-group`,
reads colours from the same CSS custom properties (`--color-accent`, `--color-surface-alt`,
`--color-border-strong`, `--color-text-muted`) so it respects the Matrix theme swap, and uses
`styled-jsx` (already used implicitly via Next) for the two small keyframe animations, scoped
to the component only.

Two pure helpers, `applicationSpotCounts` (array → `{spot: count}` map, defaulting missing/old
reviews to all-zero) and `countsToApplicationSpots` (map → `{spot, sprays}[]`, omitting zero
counts per T094's validator), do the shape conversion and are unit tested in the new
`src/components/PerfumeCharacteristics.test.js`.

`add.js` and `edit/[id].js` both swapped their checkbox block + `toggleApplicationSpot` for
`<ApplicationSpotsSprayer value={formData.applicationSpots} onChange={...} />` — no other
behaviour changed. The edit form already fell back to `perfume.applicationSpots || []`, and
`applicationSpotCounts` further defaults any spot not present in that array to 0, so it
degrades correctly for reviews that predate this field.

## Verification

- `npm run lint` — 0 errors (3 pre-existing warnings, none new).
- `npm test` — 22 suites / 167 tests pass, including the new
  `PerfumeCharacteristics.test.js` covering both helpers (empty/undefined input, known spots,
  omission of zero-count spots).
- `npm run build` — succeeds; `/reviews/perfumes/add` and `/reviews/perfumes/edit/[id]` both
  build as static pages at their previous rough size (+~0.5kB for the new component).
- Started `npm run dev` and confirmed `/reviews/perfumes/add` renders the new sprayer markup
  ("Application spots — tap the bottle to spray 🧴💦") via a plain HTTP fetch of the page.
  Did **not** perform a live form submission — that would write a real perfume review row to
  production DynamoDB via `/api/reviews/perfumes/add`, which is a live/paid backend write this
  harness must not trigger just to "verify" a UI change (see harness rule on live third-party
  calls). The submit → reload → re-populate round trip was instead verified by code review:
  `add.js` sends `formData.applicationSpots` (already in `{spot, sprays}[]` shape via
  `countsToApplicationSpots`) unchanged to the existing `/api/reviews/perfumes/add` route, and
  `edit/[id].js` seeds `formData.applicationSpots` straight from the fetched review's
  `applicationSpots`, which `ApplicationSpotsSprayer`/`applicationSpotCounts` then render as
  the correct per-spot counts.
