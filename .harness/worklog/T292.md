## History

The autonomous loop attempted this task earlier and correctly self-blocked: `backfillPersonalBests.js`
crashed under `LIVE=1` on rows with zero working sets — a raw `-1` "not found" sentinel from
`calculateExerciseMetrics` (`src/lib/workoutMetrics.js`) was leaking through `detectPersonalBests`
into `applyFlagsToSets`, which then tried `sets[-1].isWeightPR = true` (`undefined` in JS, unlike a
real negative-index language). The loop's own diagnosis was correct and out of this task's declared
scope (`src/lib/workoutMetrics.js` isn't in `T292`'s scope) — it needed a fix task first, exactly as
recorded.

The owner separately ran the script manually, hit the identical crash on the same row
(`df393308-0d85-4f3d-8420-42f7ce6b1e97_7`, "Cable Twist (Down to up)" — weight logged with 0 reps, so
`hasWeightData` was true but no set ever satisfied `volume > 0`, leaving `heaviestWeightSetIndex` etc.
stuck at their `-1` initial value), and asked for the root cause diagnosed and fixed.

## Fix (this session, interactive)

Root-caused and fixed at the source — `calculateExerciseMetrics` now coalesces `heaviestWeightSetIndex`/
`bestEstimated1RMSetIndex`/`bestSetVolumeSetIndex` from `-1` to `null` before returning, matching how
every other "not applicable" case in this function is already represented. This is a shared function
also consumed by the live Hevy-sync write path (`workoutBackfill.js`), which had the identical latent
crash risk — fixing it here fixes both call sites.

Added regression tests: `src/lib/workoutMetrics.test.js` (unit level, plus updated two pre-existing
tests that had baked in the buggy `-1` expectation) and `src/scripts/backfillPersonalBests.test.js`
(integration level, reproducing the exact production row shape that crashed). Full suite: 394/394
passing, 0 lint errors. Committed as `29e04bb`.
