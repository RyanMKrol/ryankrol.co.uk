## Resolved 2026-07-03 (owner-directed interactive session)

Root cause was more specific than the marshaller symptom suggested: `reshapeExerciseRows` in
`src/scripts/backfillIncompleteWorkoutMetrics.js` read `row.exercise_index`/`row.exercise_name` off
Exercises-table rows, but the real (current) schema uses `index`/`title` — confirmed by diffing a
broken record's raw Exercises rows against an already-complete workout's `exercises[]` entries. Both
fields came back `undefined` for every row, which is what the DynamoDB marshaller choked on. The test
fixtures (`backfillIncompleteWorkoutMetrics.test.js`) used the same wrong field names, so they were
self-consistently wrong and never caught this — a fixture-only test that doesn't reflect the real
schema is a real gap here, not just in this one script.

Fixed `reshapeExerciseRows` to read `title`/`index` first. A second live-run crash on a different
workout revealed the table actually has **two Exercises-row schemas mixed together**: the current
`title`/`index` shape, and an older `exercise_name`/`exercise_index` shape (some rows also carry
`data_source: "hevy_api"`, `backfilled_at`, `created_at` — likely pre-dating a schema change, or from
the external writer flagged in T134's investigation). Some individual workouts have ONE exercise row
in each shape. Made the reshape resilient with `??` fallbacks to both field-name pairs, added tests
covering the old-shape and mixed-shape cases, then re-ran.

**Outcome:** dry run + live run in two passes (41 workouts on the first pass before hitting the
second bug, 18 more after the fallback fix) — all 59 originally-incomplete workouts updated
successfully, zero writes lost. Final production verification scan: 368/368 workouts now have
`exercises` populated, 0 incomplete remaining. `npm test`/`npm run lint` clean throughout (no new
errors, only pre-existing warnings).
