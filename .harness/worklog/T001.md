# T001 — Fix workouts-page pagination + filtering

## Summary

Implemented client-side filter + pagination for the workouts page. Previously the API
paginated server-side then the page filtered client-side, so Push/Pull/Legs only showed
matching workouts from the current 10-item page, not across all history.

## Changes

- **src/lib/workoutPagination.js** — new pure module: `filterWorkouts`, `getPageCount`,
  `paginateWorkouts`. All logic is title-case-insensitive substring match; out-of-range
  pages clamp to the last valid page.
- **src/lib/workoutPagination.test.js** — 11 tests covering all 4 spec requirements (a–d).
- **src/lib/workoutQueries.js** — added `getAllWorkouts()`: same scan + sort as the
  existing paginated variant but returns the full array without slicing.
- **src/pages/api/workouts.js** — added `?mode=all` path: calls `getAllWorkouts()`,
  wraps in `withApiCache('api-workouts-dynamo-all', ...)`, triggers `triggerBackfillAsync`
  on cache miss (equivalent to the old page-1 trigger). Legacy `?page=` path is preserved.
- **src/pages/workouts/index.js** — refactored to fetch `/api/workouts?mode=all` once,
  then call `filterWorkouts` + `paginateWorkouts` purely in the browser. Filter changes
  reset to page 1. Pagination indicator shows correct total for the active filter.

## Verification

- `npm test` (scoped to workoutPagination): 11/11 passed.
- `npm run build`: succeeded cleanly. No new lint errors (pre-existing warning in
  dev/cache.js — unrelated to this task).
- Live call to DynamoDB NOT made (would require Vercel env credentials).
