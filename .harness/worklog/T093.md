# T093 — Fix dashboard auto-refresh wiping checkbox selections

## What changed

`.harness/dashboard/server.js`:

- Added `let selectedIds = new Set();` at script scope (alongside the existing `window.__openIds`
  pattern already used for expanded-row state), plus `window.toggleSelect(cb)` which adds/removes
  `cb.dataset.id` from `selectedIds` and calls `updateBulk()`.
- `task()`'s checkbox markup (`.sel-done` / `.sel-rev`) now uses `onchange="toggleSelect(this)"`
  instead of the bare `onchange="updateBulk()"`, and appends a `checked` attribute when
  `selectedIds.has(t.id)` — so a freshly re-rendered checkbox comes back pre-ticked from the
  persisted set rather than relying on DOM state the `innerHTML` replace destroys.
- `toggleAllDone`/`toggleAllRev` (the "select all" bulk toggles) now also add/remove ids from
  `selectedIds` as they flip each checkbox, so bulk-select stays in sync with the persisted set.
- `render()` now calls `updateBulk()` at the end, so the "Mark N reviewed/done" button labels
  reflect the restored selection immediately after a re-render (previously only wired to the
  `onchange` event).

This mirrors `local-jobs/dashboard/app/backlog/page.tsx`'s `useState<Set<string>>` pattern
translated to plain JS: selection lives in a variable declared outside the render function, is
updated only by the checkbox handlers, and drives each row's `checked` state on every re-render —
so a data poll (still a full re-render, same as React) never touches selection.

- Started the dashboard locally (`node .harness/dashboard/server.js`) and curled the served page —

  report and `local-jobs` reference describe. Recommend the owner do a quick eyeball confirmation
  in-browser.
