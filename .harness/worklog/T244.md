# T244: fix masonry reading order on /workouts

## What changed
- `src/styles/globals.css`: `.workout-session-grid` switched from `column-count: 2` (CSS multi-column,
  which reads top-to-bottom-then-left-to-right) to a flex row; added `.workout-session-grid-col` as the
  per-column flex container. Removed the `max-width: 900px` `column-count: 1` override — column count is
  now driven by `useResponsiveColumnCount(2, 900)` in JS. Removed the now-redundant `margin-bottom` on
  `.workout-session-card` (columns provide spacing via `gap`); kept `break-inside: avoid`.
- `src/pages/workouts/index.js`: replaced the raw `<div className="workout-session-grid">{pageWorkouts.map(...)}</div>`
  with `<MasonryColumns items={pageWorkouts} columnCount={columnCount} className="workout-session-grid"
  columnClassName="workout-session-grid-col" renderItem={...} />`, reusing `src/lib/masonryColumns.js` +
  `src/hooks/useResponsiveColumnCount.js` + `src/components/MasonryColumns.js` (already built by a prior
  task, used as-is per spec). `columnCount` is computed once via `useResponsiveColumnCount(2, 900)` at the
  top of the component (not inline in the ternary) to avoid a conditional-hook-call lint error
  (`react-hooks/rules-of-hooks`) that surfaced when first placed inline. All per-card rendering logic
  (`getExercisePreview`, `getTotalVolume`, split badge, exercise list, stats) is unchanged — only the
  wrapping/columnizing mechanism changed.
