# T176 — Build log/audit `<details>` snaps shut on every 5s poll

## What was built

`.harness/dashboard/server.js`'s page template already tracked outer task-row expansion
(`window.__openIds`, reapplied as a CSS class after each `render()`), but the nested
`<details class="buildlog">` used for the "Build log" / "Audit" sections had no tracking at all —
`render()` replaces `#main.innerHTML` wholesale every 5s poll, so every `<details>` is a fresh DOM
node with no `open` attribute, and it always collapsed on the next poll regardless of row state.

Added a second Set, `window.__openLogIds`, mirroring `__openIds`:
- `window.onLogToggle(el)` — wired via `ontoggle` on each buildlog/audit `<details>` — adds/removes
  the element's id from the set based on the browser's native `open` IDL property.
- Each `<details>` now gets a stable id (`log-<taskId>-build` / `log-<taskId>-audit`) instead of no
  id at all.
- `render()` now has a second reapply loop alongside the existing `__openIds` one, setting
  `el.open = true` for any id present in `__openLogIds` after the innerHTML replace.

The existing `__openIds`/`toggle(id)` row-level mechanism is untouched.
