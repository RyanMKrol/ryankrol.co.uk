# T381 — Vinyl edit flow (re-search Last.fm)

2026-07-13

## What was built

- `src/pages/vinyl/edit.js` — list of vinyl records (cover grid, `CoverTile` + gradient
  fallback), each card with an "Edit" link, mirroring `reviews/albums/edit.js`.
- `src/pages/vinyl/edit/[id].js` — per-record edit detail form: editable title/artist,
  `LastfmAlbumSearch` to re-run a Last.fm search (title-query driven, same component
  `vinyl/add.js` uses), a cover preview (current cover, or the new one once a match is
  confirmed), password field, single "Update Record" submit. No delete action (out of
  scope per spec). Leaving the search unused and just editing title/artist and saving is
  supported — `lastfm` is only sent if a fresh match was confirmed.
- `src/pages/api/vinyl/update.js` — POST, gated on `RYANKROL_SITE_KEY`, `UpdateCommand` by
  `id` writing `title`/`artist` always, and `thumbnail` (via `pickLargestFromImageMap`) +
  full `lastfm` object only when a `lastfm` payload is present. Clears
  `clearApiCache('api-vinyl-collection')` — verified this matches the read route's
  `generateCacheKey('vinyl-collection')` key (same as `add.js`), not `api-vinyl`.
- `src/pages/vinyl/index.js` — added an "Edit records" link (`/vinyl/edit`) in the header,
  next to the title/count, styled with `--color-accent-secondary` to match the albums
  edit-page "Backfill metadata" link convention.
- Docs: `CLAUDE.md` file map updated with the two new vinyl edit pages and the new
  `api/vinyl/update.js` row (documenting the `api-vinyl-collection` cache-key gotcha).
  README's `VinylCollectionV2` schema section needed no changes (edit doesn't alter the
  stored shape).
- `scripts/_visual-harness.mjs`: added `PAGES` entries `vinyl-edit` and
  `vinyl-edit-detail` (using fixture record `vn-in-rainbows`).
