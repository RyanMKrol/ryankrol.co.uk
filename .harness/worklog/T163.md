# T163: Restyle books + albums add/edit forms to Collection tokens

## What changed

Followed the exact precedent set by T162 (movies/TV forms):

- `src/pages/reviews/books/add.js`, `src/pages/reviews/books/edit/[id].js`
- `src/pages/reviews/albums/add.js`, `src/pages/reviews/albums/edit/[id].js`

Swapped legacy classes (`form-container`, `form-group`, `form-label`, `form-input`,
`form-textarea`, `form-button`, `success-message`/`error-message`, `btn-danger`) for the
Collection form classes already defined in `globals.css` from T162
(`collection-form-card`, `collection-form-group`, `collection-form-label`,
`collection-form-input`, `collection-form-textarea`, `collection-form-button`,
`collection-form-button-danger`, `collection-form-actions`, `collection-form-message[-success|-error]`).
Added `hasAttemptedSubmit` gating for the "select a match" hint (matches the movies/TV pattern,
only shows the warning after a submit attempt instead of unconditionally). Inline `style jsx`
blocks (Last.fm cover preview, `mbm-*` backfill card rows) were re-pointed from hardcoded
colors/radii to Collection tokens (`--color-hairline`, `--radius-cover`, `--font-body`,
`--color-ink`).

- `src/components/BookSearch.js`, `src/components/LastfmAlbumSearch.js` — restyled in place
  (shared by add + edit flows) following `TmdbSearch.js`'s existing Collection treatment: search/
  select buttons now use `collection-form-button`, inline errors use `collection-form-message
  collection-form-message-error`, and the scoped `style jsx` blocks use `--accent-books` /
  `--accent-albums` for the confirmed-match border/label color, `--color-ink` for the clear
  button, and `--radius-cover`/`--radius-stat`/`--radius-pill`/`--color-hairline(-strong)` in
  place of hardcoded hex/px values.

No `onSubmit`/`onSearch`/API-calling logic was touched. `src/pages/reviews/books/edit.js` and
`src/pages/reviews/albums/edit.js` (the list pages) needed no changes — they already use the
shared `review-container`/`reviews-wrapper`/`ReviewCard` chrome, identical to
`movies/edit.js`, which T162 also left untouched. `globals.css` needed no changes — the
`collection-form-*` classes it added for T162 are shared/reused as-is.

## Verify

- `npm test` — 35 suites / 241 tests passed.
- `npm run build` — succeeded; `/reviews/books/{add,edit/[id]}` and
  `/reviews/albums/{add,edit/[id]}` all built as static pages.
- Manual `run-app` verification (live Google Books / Last.fm calls, password-gated submit) was
  not performed in this headless run per the harness rule against live third-party/metered calls;
  verified instead by code review against the working T162 movies/TV pattern this mirrors exactly.
