# T365 — worklog

## 2026-07-11 — reopened after a spec-defect block (recovery + re-scope)

The loop previously blocked this task: the builder built the script + tests + docs and ran the
read-only dry run against production (4 rows), but reported it could not autonomously perform the
`LIVE=1` irreversible production data deletion. Root cause was NOT the code (the audit found the
script/tests/docs solid) — it was a **self-contradictory spec**: the Overview said to defer the
`LIVE=1` run until Hardcover-from-production was confirmed, while `## Done when` demanded that same
`LIVE=1` run be executed and verified. No competent build could satisfy both, so it audit-failed
twice and then the agent correctly reported it could not proceed autonomously.

**Fix applied (owner-directed interactive session):** the spec was rewritten to descope the
destructive live run out of this task's acceptance criteria — T365 now delivers only the migration
script, its unit tests, its read-only dry run, and the docs (all loop-buildable). The actual
`LIVE=1` production run + post-run re-scan verification is now a separate human-gated task,
**T370** (`dependsOn: T365`). This task was reopened (status back to pending), and the spec-defect
ledger rows (the two `audit-fail` rows and the `agent-blocked` row in failures.jsonl, plus the
`blocked` outcome row in outcomes.jsonl) were pruned so the `db × data-migration` calibration cell
isn't skewed by an unsatisfiable-spec block rather than genuine difficulty.

Status: ready for the loop to rebuild against the corrected spec.
