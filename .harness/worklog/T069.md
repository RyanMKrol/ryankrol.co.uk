# T069 — Backfill-from-API button on the albums edit page

## What changed
- `src/pages/reviews/albums/edit/[id].js`: added a "Backfill from Last.fm" button using the
  shared `MetadataBackfillModal` (from T066).
  - `handleBackfillSearch` calls `GET /api/lastfm/album-search?query=...` seeded from the
    form's current `title`/`artist`.
  - `handleBackfillConfirm` fetches full details via `GET /api/lastfm/album-info` (by `mbid`
    when the search result has one, else `artist`+`album`), then sets `formData.lastfm` to
    the full `mapAlbumInfo` shape and a separate `backfillThumbnail` state (not persisted
    directly — used only for the client-side cover preview) for visible confirmation.
  - `rating` and `highlights` are never touched by the backfill flow.
  - Save (existing `handleSubmit`) is unchanged — it already spreads `...formData`, so
    `lastfm` rides along to `POST /api/reviews/albums/update` once present. No auto-submit
    on confirm.

## Verified
- `npm run lint` — 0 errors (2 pre-existing warnings, unrelated files).
- `npm test` — 139 passed.
- `npm run build` — succeeds, `/reviews/albums/edit/[id]` builds fine.
- Did not make live Last.fm calls (per harness rules) — verified by code reading against
  `src/pages/api/lastfm/album-search.js`, `album-info.js`, and `src/lib/lastfm.js`'s
  `mapAlbumSearchResult`/`mapAlbumInfo` shapes, and cross-checked against the existing
  `LastfmAlbumSearch` component (used on the add page) which does the same two-step
  search→info flow.

## Scope
Only touched `src/pages/reviews/albums/edit/[id].js` (no changes needed under
`src/components/**` — `MetadataBackfillModal` already supports this use case as-is).
