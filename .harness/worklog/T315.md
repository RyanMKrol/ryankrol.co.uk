# T315 — visible axis-specific PR badges on /workouts + /workouts/[id]

## What changed

- `src/components/Badge.js`: added a named `PrBadge({ label })` export — the
  shared medal + visible-label pill, built on top of the existing default
  `Badge`. Added `className`/`tabIndex` passthrough to `Badge` so `PrBadge`
  can attach its own hover/focus CSS hook and be keyboard-focusable.
- `src/styles/globals.css`: added `.pr-badge` — a subtle `:hover`/
  `:focus-visible` background/border shift (via `color-mix(in srgb, currentColor
  22%, transparent)`, so it tracks each badge's own accent colour rather than a
  hardcoded value) as the only interactive affordance. No new information is
  revealed on hover — the label is already always visible.
- `src/pages/workouts/index.js`: `getExercisePreview` now returns a per-exercise
  `prAxes` array (deduped `weight`/`1rm`/`volume` PR axes across that
  exercise's sets in the session) instead of a single `hasPersonalBest`
  boolean. Rendering now maps `prAxes` to one `PrBadge` per axis, replacing the
  old single generic badge + screen-reader-only clip-rect span.
- `src/pages/workouts/[id].js`: kept the existing axis-specific `prBadges`
  array as-is; swapped its rendering to `PrBadge` (visible label) and deleted
  the clip-rect hidden span.

## Verification

- `npm run lint` — 0 errors (6 pre-existing warnings, all unrelated: `no-img-element`
  on other pages, one `exhaustive-deps` on `dev/cache.js`).
- `npm test` — 396 passed. Updated `test/pages/workouts/index.test.js` (a
  pre-existing test file for `getExercisePreview` I didn't spot before writing
  a duplicate — removed the duplicate, updated the real one) to assert on
  `prAxes` shape, including a new case for an exercise whose different sets
  each PR'd a different axis in the same session (dedup + combine).
- `npm run build` — succeeds, `/workouts` and `/workouts/[id]` both build clean.
- Did not run the app in a browser (headless environment, no live DynamoDB
  data available to eyeball a real PR'd exercise) — verified the rendering
  logic by reading the diff + the passing unit tests instead, per the
  DoD's "verify by reading the code and `npm run build`" allowance when a
  live check isn't practical. `PrBadge` always renders both the medal and a
  normal (non-clipped) `<span>` label side by side for every axis present in
  `prAxes`/`prBadges`, and renders nothing when that array is empty — matching
  the required "no badge at all for the non-PR case" behaviour structurally.
