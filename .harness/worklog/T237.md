## Root cause of the verification failure — much bigger than either T236 or T237 anticipated

Cross-referencing the Workouts table's denormalized `exercises[]` data (as in the original
investigation) against **direct `GetCommand`s by `exercise_id`** (not the GSI) for all 28 workouts
containing this exercise: **all 28 rows already exist in the base `Exercises` table.** The GSI
was never missing data — it was refusing to *see* data that's actually there.

Inspecting the 25 rows the GSI can't find (`Iso-Lateral High Row (Machine)` alone) shows they are
written in a **completely different, legacy schema shape**: `title` instead of `exercise_name`,
`workout_start_time` instead of `workout_date`, `index` instead of `exercise_index` — no computed
metrics fields at all (no `sessionVolume`, `bestEstimated1RM`, etc.), just the raw per-set data
plus Hevy-native fields (`exercise_template_id`, `superset_id`, `notes`). This looks like the
original raw-Hevy-shape ingestion, predating this app's current `calculateExerciseMetrics`-based
schema entirely.

Because `exercise_name`/`workout_date` are the GSI's own partition/sort key attributes, DynamoDB
never projects an item missing either one into that index — so these rows are permanently
invisible to `getExerciseHistory()` (`src/lib/workoutQueries.js`), which is the exact query the
exercise-detail page (`/exercises/[exerciseName]`) uses. This also explains why NEITHER
`storeExercises()` (T233's shared write path) NOR `backfillMissingExerciseRows.js` (T236) can
ever repair these rows through their normal write path: both use
`ConditionExpression: attribute_not_exists(exercise_id)`, which correctly (and by design) refuses
to overwrite an item that already exists at that key — even though the existing item is in this
incompatible legacy shape. The rows aren't missing; they're stuck.

**Scale, checked directly against production**: scanned the entire `Exercises` table (1,314 rows
total) for items missing `exercise_name` and/or `workout_date` — **225 rows (~17% of the table)**
share this legacy shape. This is not specific to `Iso-Lateral High Row (Machine)` — it very likely
affects many other exercises' detail pages the same way, to varying degrees depending on how many
of their sessions happen to fall in this legacy cohort.

## Why this wasn't hand-fixed here

225 production rows need a real MIGRATION (backfilling `exercise_name`/`workout_date`, and likely
the missing computed-metrics fields too, onto each legacy row — probably derivable from the
sibling Workouts item's own denormalized `exercises[]` entry, similar in spirit to T236's script
but targeting *malformed* rows rather than *absent* ones) — a meaningfully different, larger, and
riskier task than either T236 or T237 were scoped for. Per the spec's own instruction, stopping
here rather than improvising a fix.
