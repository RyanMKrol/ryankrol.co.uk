## Part 2 — the real root cause: 225 legacy-schema Exercises rows invisible to the GSI

Cross-referencing the Workouts table's denormalized `exercises[]` data against **direct
`GetCommand`s by `exercise_id`** (not the GSI) for all 28 workouts containing
`Iso-Lateral High Row (Machine)`: all 28 rows already existed in the base `Exercises` table. The
GSI was never missing data — it was refusing to *see* data that's actually there.

25 of those 28 rows were written in a **legacy schema shape** predating this app's current
`calculateExerciseMetrics`-based schema: `title` instead of `exercise_name`, `workout_start_time`
instead of `workout_date`, `index` instead of `exercise_index`, no computed metrics fields at all —
just raw per-set data plus Hevy-native fields (`exercise_template_id`, `superset_id`, `notes`).
Because `exercise_name`/`workout_date` are the GSI's own partition/sort key attributes, DynamoDB
never projects an item missing either one into that index, so these rows were permanently invisible
to `getExerciseHistory()` (`src/lib/workoutQueries.js`) — the exact query the exercise-detail page
(`/exercises/[exerciseName]`) depends on. Neither `storeExercises()` (T233) nor
`backfillMissingExerciseRows.js` (T236) could ever repair these, since both correctly refuse to
overwrite an `exercise_id` that already exists via `ConditionExpression: attribute_not_exists`.

**Full-table scan confirmed scale**: 225 of 1,314 Exercises rows (~17%) shared this legacy shape,
across 72 unique workouts — not specific to one exercise.

**Investigated before acting**: also checked whether a full table wipe-and-refetch-from-Hevy (the
owner's first suggestion) was safe. It was NOT — 3 workouts exist in DynamoDB (all real, well-formed,
canonical-shape records with genuine dates/volumes/`data_source: hevy_api`) that no longer exist on
Hevy's side (deleted by the owner at some point). A full Hevy-based re-import would have permanently
lost those 3. Also confirmed the Workouts table itself has **zero** schema drift on any field the app
depends on (`workoutDate`, `totalVolume`, etc. — 0 missing across all 378 rows; its "2 shapes" differ
only in harmless provenance metadata like `data_source`/`description`).

**Fix built and run** (`src/scripts/migrateLegacyExerciseRows.js`, new): rebuilds the Exercises table
from the Workouts table's own already-present `exercises[]` data — NOT from Hevy — so it's lossless
even for the 3 orphan workouts. For each of the 72 affected workouts: delete its legacy-shaped
Exercises row(s) (`exercise_id` is deterministic — `${workout.id}_${index}` — so this is a like-for-
like replacement, not an addition), then call the shared `storeExercises(workout)` to regenerate all
of that workout's exercise rows fresh in canonical shape. Rows already in canonical shape are left
untouched entirely (surgical, not a full-table wipe).

- Dry run: 225 legacy rows / 72 workouts, 0 skipped (workout missing or with no `exercises[]`).
- Live run (`LIVE=1 npm run workout:migrate-legacy-exercises`): all 225 rows deleted and regenerated,
  0 errors.
- **Verified**: full re-scan of Exercises table afterward — 1,314 rows total (unchanged, 1:1
  replacement), **0 legacy-shaped rows remain**. `Iso-Lateral High Row (Machine)` GSI count: **28**
  (was 3) — matches the Workouts-table denormalized count exactly.
- **Verified live on production**: flushed the API cache (`POST /api/dev/cache-bust`), then
  `GET /api/exercises/history/Iso-Lateral%20High%20Row%20(Machine)` returns 28 sessions.

New `src/scripts/migrateLegacyExerciseRows.test.js` covers the pure `isLegacyShape()` predicate.
