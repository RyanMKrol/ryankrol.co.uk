# T268 ‚Äî consolidate-ideas.mjs deploy-task enforcement

## Status: done. Built interactively (needs-human, harness-scoped), per the "never let the
autonomous loop modify itself" convention.

## What changed

- `.harness/consolidate-ideas.mjs`: inserted a new step (5.5) between building `newTaskObjects`
  and merging into `TASKS.json`. Determines which of the sweep's new tasks are "site-touching"
  (scope has an entry not under `.harness/`). If any: finds the existing pending
  "Deploy pending site changes to production" task and unions their ids into its `dependsOn`, or
  ‚Äî if none is pending ‚Äî allocates a new one via the same `allocId()` used for everything else,
  builds it in the same shape as `T193`/`T216`/`T238` (`gate: null`, `scope: []`, matching
  facets), writes its spec via a new `buildDeploySpec()` template function (parameterized by the
  sweep's site-touching ids), and pushes it into both `newTaskObjects` (so the merge picks it up)
  and `allocatedTasks` (so the shell wrapper's existing `NEW_MD_FILES` derivation stages its spec
  file automatically ‚Äî confirmed by reading `consolidate-ideas.sh`, no shell change needed).
  Step 8's summary now records `deployTask: { action, id, siteTouchingTaskIds }` and folds one
  line into `suggestedCommitMessage` when a deploy task was created or updated.
- `.harness/consolidate-ideas.test.sh` (new): hermetic test, throwaway `mktemp -d` git repo,
  invokes `consolidate-ideas.mjs` directly (it does no git operations itself ‚Äî only its shell
  wrapper does ‚Äî so no lock/remote setup needed for this unit test). Covers: (a) a site-touching
  task + no existing pending deploy task ‚Üí a new one is created with the correct `dependsOn` and
  spec file; (b) a site-touching task + an existing pending deploy task ‚Üí its `dependsOn` gains
  the new id while preserving what was already there, no second deploy task created; (c) a task
  scoped entirely under `.harness/` ‚Üí no deploy task created; plus a bonus case for `scope: []`.
- `.harness/CLAUDE.md`: added a paragraph documenting this as authoring-time enforcement,
  explicitly framed as coexisting with (not replacing) T267's loop.sh-level fallback and the
  original manual authoring convention.
