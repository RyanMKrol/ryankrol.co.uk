# T261: bulk metadata-backfill page for albums

## What was built
- `src/pages/reviews/albums/backfill.js` — new page mirroring the movies/tv/books bulk-backfill
  pattern, using the existing `needsAlbumBackfill` from `src/lib/backfillEligibility.js` to filter
  albums missing `lastfm.mbid`. Single page-level password field, `BulkBackfillList` with
  `pageSize=15`.
  - `onSearch` hits `GET /api/lastfm/album-search?query=<title-or-artist>`, reads `.results`,
    attaches `retryAfterSeconds` from the `Retry-After` header on a 429 (mirrors
    `handleBackfillSearch` in `edit/[id].js`).
  - `renderCandidate`/`getCandidateKey` reuse the thumbnail/title/artist card JSX from
    `edit/[id].js:263-282`.
  - `onConfirm` does the two-step flow per the task's `design` field: `GET
    /api/lastfm/album-info` (by `mbid` if present, else `artist`+`album`), maps the result into
    the same `lastfm` object shape as `handleBackfillConfirm`, then `POST
    /api/reviews/albums/update` with the full required field set (`title`, `artist`, `rating`,
    `highlights`, `originalId`, `password`, `lastfm`) since the update endpoint requires all of
    them every time. Errors from either the info fetch or the update POST are thrown and surfaced
    on that row only (`BulkBackfillList` already isolates per-row errors) — including 429
    `retryAfterSeconds` from the album-info call, which hits a separate rate-limit bucket per the
    design note.
- `src/pages/reviews/albums/edit.js` — added a "Backfill metadata" link to
  `/reviews/albums/backfill`, matching the existing pattern in movies/tv/books edit pages.
