# T141 — Retire the legacy multi-theme AppearancePicker system

## What happened

The loop's builder attempted this task first and correctly reported `failed:blocked` (see the
original blocked record preserved below): `src/pages/_app.js` directly imports and calls
`useTheme()` (a call site the task's original spec missed — it only accounted for `MatrixLayout`'s
`active` prop, not `_app.js`'s own direct `useTheme()` call), so deleting `src/hooks/useTheme.js` per
the spec would break the build, and `_app.js` wasn't in the task's `scope`. The builder refused to
either leave `useTheme.js` in place (failing the Done-when grep) or edit a file outside its scope —
correct behavior per this repo's scope-is-a-hard-boundary convention.

This was completed interactively (owner-directed session) rather than re-queued through the loop:
1. Fixed the task definition first — added `src/pages/_app.js` to T141's `scope` in `TASKS.json`,
   and updated `.harness/tasks/T141.md` to explicitly describe removing `_app.js`'s `useTheme` import
   + call.
2. Built the (now correctly scoped) task.

## What was built

- Deleted `src/components/AppearancePicker.js`, `src/hooks/useTheme.js`, `src/lib/appearance.js`,
  `src/lib/appearance.test.js`.
- `src/components/MatrixLayout.js` — removed the `pickerOpen` state, the 🎨 button, and the
  `<AppearancePicker>` render. Now just renders `MatrixRain`/`CRTOverlay`/children off the `active`
  prop, unchanged otherwise.
- `src/pages/_app.js` — removed the `useTheme` import and its bare `useTheme();` call. Left
  `useKonamiCode()` and `MatrixLayout` usage untouched.
- `src/pages/_document.js` — the pre-paint script no longer sets `data-theme`/`data-mode`/
  `data-font`/`data-motion`; it now only restores the `matrix-active` class from `sessionStorage`
  pre-hydration (unchanged logic, just stripped of the theme-system lines around it). Also had to
  touch the font `<link>` here even though the original spec didn't call for it: dropped **VT323**
  (confirmed via `grep -rn "VT323" src/` — zero remaining references, it was terraria-only) but had
  to **keep Share Tech Mono** (the surviving default `:root` block in `globals.css` — which this task
  deliberately leaves alone for T142 to repopulate — still actively declares
  `--font-body/--font-heading/--font-mono: 'Share Tech Mono', monospace`, so dropping its font link
  would have silently broken the current site's rendered font) and **JetBrains Mono** (still used by
  `useChartTheme.js` and `dev/cache.js`). Nunito/Space Mono links were dropped (confirmed unused
  anywhere in `src/` right now — T142 adds them back fresh with the new Collection weights).
- `src/styles/globals.css` — deleted the "Theme Palette Overrides" block (all 5
  `html[data-theme="X"]` blocks + their light-mode sub-overrides, ~875 lines) and the "Font
  Overrides" block (`data-font` overrides + the `[data-motion="reduced"]` rules, now unreachable
  since nothing sets `data-motion` anymore) and the "Appearance Picker Modal" block (`.global-
  appearance-btn` + all `.ap-*` classes, ran to EOF). Left the "Default Theme" `:root` block (lines
  1-123) and the "Matrix Theme Override" block completely untouched, per spec — T142 repopulates
  `:root` with the Collection tokens next. Also left the unrelated "Cyberpunk Enhancements" section
  (scrollbar/selection/hover-glow styling) alone — it wasn't part of the theme-selection machinery,
  it's general sitewide chrome with hardcoded colors that'll look wrong until T142/audit tasks
  address it, which the spec explicitly said is expected/fine for this task.

## Verification

- `grep -rn "AppearancePicker\|useTheme\|appearance-theme\|appearance-mode\|appearance-font\|appearance-motion" src/` — zero matches.
- `grep -rn "data-theme\|data-font\|data-motion" src/` — only one incidental hit, a `MutationObserver`
  `attributeFilter` array in `src/hooks/useChartTheme.js` (out of this task's scope, harmless — those
  3 attribute names are now just inert entries in the filter list since nothing sets them anymore;
  `class` in the same filter still correctly fires on Matrix-mode toggling).
- `npm run lint` — 0 errors, 3 pre-existing warnings (unrelated files).
- `npm test` — 34 suites / 235 tests passed (down from 35 suites, as expected — `appearance.test.js`
  was deleted with its source file).
- `npm run build` — succeeded.
- `npm run dev` + curled `/`, `/reviews/movies`, `/vinyl`, `/workouts`, `/listening`, `/projects`,
  `/reviews/perfumes` — all HTTP 200, zero occurrences of `AppearancePicker`/`global-appearance-btn`/
  `🎨` in the rendered HTML, zero error-trace strings. Dev server compiled clean with no warnings.
- Matrix-mode preservation (the one behavior that had to be 100% intact): read
  `src/hooks/useKonamiCode.js` in full — it is entirely self-contained (its own `isActive` state, its
  own `matrix-active` sessionStorage key, its own keydown listener for the Konami sequence) with zero
  dependency on anything deleted in this task. `_app.js` still passes its return value into
  `MatrixLayout`'s `active` prop unchanged, and `_document.js`'s pre-paint script still restores the
  class from `sessionStorage` before hydration. Did not have browser-automation tooling available in
  this environment to literally simulate the arrow-key sequence, but every file in the activation/
  persistence chain was read and confirmed unaffected.
