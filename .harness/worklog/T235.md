# T235 — migrateWorkoutData.js delegates to shared workoutBackfill writes

## What changed
- `src/scripts/migrateWorkoutData.js` no longer defines its own
  `storeWorkoutInDynamoDB` / `storeExerciseInDynamoDB` or issues `PutCommand`s
  directly. It now imports `storeWorkoutInDynamoDB` from `../lib/workoutBackfill.js`
  and `migrateWorkoutData()` calls it once per workout.
- Removed the now-unused inline `DynamoDBClient`/`DynamoDBDocumentClient` setup
  and the `calculateExerciseMetrics`/`calculateWorkoutMetrics` imports (metrics
  are computed inside the shared function now).
- Simplified the end-of-run summary to a single "N/total workouts processed"
  line, since the shared function doesn't expose per-exercise stored/skipped
  counts (matches the sibling `wdi-consolidate-audit` approach).
- Exported `migrateWorkoutData` and guarded the `main()` auto-invocation with
  `process.env.NODE_ENV !== 'test'` so the function is importable/testable
  without triggering a live Hevy fetch on import (Jest sets `NODE_ENV=test`).
- Added `src/scripts/migrateWorkoutData.test.js`, mocking `../lib/workoutBackfill`
  and asserting `storeWorkoutInDynamoDB` is called exactly once per workout with
  the raw Hevy workout object, plus a case verifying one workout's failure
  doesn't stop processing of the rest.

## Verify
- `npm run lint` — 0 errors (6 pre-existing warnings, unrelated to this change).
- `npm test` — 44 suites / 290 tests pass, including the new
  `migrateWorkoutData.test.js`.
- `npm run build` — succeeds.

## Note (not fixed — out of scope for this task)
Running `node src/scripts/migrateWorkoutData.js` directly hits a pre-existing
bug in `src/lib/workoutBackfill.js`: it imports `./dynamo` and `./constants`
without file extensions, which resolves fine under Next.js/webpack and under
Jest's resolver, but fails strict Node ESM resolution
(`ERR_MODULE_NOT_FOUND`) when the module chain is loaded by plain `node`.
Previously `migrateWorkoutData.js` was self-contained and never hit this path;
now that it imports `workoutBackfill.js`, it's the first raw-`node`-executed
script to expose the issue. Confirmed via `node --input-type=module -e
"import('./src/lib/workoutBackfill.js')"` → `Cannot find module
'.../src/lib/dynamo'`. Fixing it means editing `src/lib/workoutBackfill.js`,
which is outside this task's scope (`src/scripts/migrateWorkoutData.js` only).
Flagging for a human/follow-up task to add explicit `.js` extensions to the
imports in `workoutBackfill.js`.
