# T396 — single-newline soft-break fix for Markdown.js

2026-07-16

## What I did

Per T388's recorded decision, added `preserveSingleNewlines(src)` to
`src/components/Markdown.js`, exported alongside the existing
`preserveBlankLines`. It appends a CommonMark hard-break marker (two
trailing spaces) to any line immediately followed by another non-blank
line — turning a single typed Enter (a "soft break") into a visible line
break instead of the space CommonMark would otherwise collapse it to.
Blank lines are left completely untouched, so `preserveBlankLines`'s
existing gap-paragraph logic (T382) still owns real paragraph breaks.

Composed the two passes in block mode only:
`preserveBlankLines(preserveSingleNewlines(children))` — since
`preserveSingleNewlines` never rewrites a blank line, the two passes don't
interfere regardless of order. The `inline` branch (used by
`ReviewCard.js` for album track highlights) was left completely untouched.

No new package dependency — pure two-trailing-spaces CommonMark syntax, no
`remark-breaks`, no `rehype-raw`, no raw `<br>` HTML string in the source.

## Tests added (`src/components/Markdown.test.js`)

- Single newline (`'line one\nline two'`) renders one `<p>` containing a
  `<br>` between the two lines (block mode).
- Normal single-blank-line paragraph break (`'para one\n\npara two'`)
  still renders exactly two `<p>` elements, no injected `<br>`.
- The T382 blank-line-run gap-paragraph tests (`\n\n\n` and `\n\n\n\n`)
  still pass, confirmed unaffected by the new pass (found and preserved a
  subtlety: the gap-paragraph placeholder is a U+00A0 non-breaking space
  in the source, not a plain space — my new tests match that exactly, as
  the pre-existing tests already did).
- Inline mode (`'line one\nline two'`) still has no `<br>` — the inline
  branch is provably untouched.
