### Bug found + fixed during the LIVE run (no data lost)
The first `LIVE=1` attempt failed with a DynamoDB `ValidationException` on the very first row:
the migration's REMOVE `UpdateExpression` interpolated raw attribute names (`REMOVE source, ...`),
and `source` is a DynamoDB **reserved keyword**. DynamoDB rejects the whole expression atomically,
so **no data was mutated** (verified by re-running the read-only dry run — all 4 rows still intact).
The dry run and T365's unit tests never caught it because the dry run never sends an `UpdateCommand`
and the tests only exercised the pure `fieldsToStrip()` helper — the real REMOVE only executes on a
LIVE write, which was (correctly) deferred to this human-gated task.

Fixed in commit `c6a5083`: extracted a pure `buildRemoveExpression(fields)` helper that aliases every
field via `ExpressionAttributeNames` (`#f0`, `#f1`, …) so reserved keywords never appear literally,
wired `stripRow` through it, and added regression tests asserting the expression uses aliases and
never the literal `source`. `npm test` (482 pass), `npm run lint` (clean), `npm run build` (ok). The
re-run then completed all 4 rows successfully.

### Notes
- This is a `src/scripts/` ops-script change only — it is not part of the deployed Next.js bundle,
  so no production redeploy is required.
- Lesson for future DynamoDB REMOVE/SET migrations: always alias attribute names via
  `ExpressionAttributeNames`; several common field names (`source`, `name`, `status`, `date`, …) are
  reserved keywords and will be rejected if used literally.
