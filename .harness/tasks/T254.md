## Do

`src/pages/api/github/repos.js` (`GET /api/github/repos`) fetches the owner's repo list from GitHub's REST API, filters/sorts it, and maps each repo to a small client-facing shape (`name, fullName, description, url, language, stars, forks, lastPush, createdAt, isPrivate, topics`). GitHub's REST repos-list endpoint has no commit-count field at all (confirmed in `.harness/tasks/T228.md`'s research, which this task directly actions) — the only way to get one per repo is an extra REST call per repo: `GET https://api.github.com/repos/{full_name}/commits?per_page=1`, then reading the response's `Link` header for `rel="last"` and parsing its `page=` query param as a total-commit-count proxy (since with `per_page=1`, the last page number IS the total commit count). This task wires that up. Do NOT migrate to GitHub's GraphQL API — this app has zero GraphQL client/dependency today (checked `package.json`: only `@aws-sdk/*`, `next`, `react`, `chart.js`, `node-cache`), and introducing one is a bigger architectural change than this personal site's 'no new infra' house rule (root `CLAUDE.md`) warrants for this feature; the existing 4-hour `withApiCache` TTL (`src/lib/apiCache.js`) means this N+1 fan-out only actually runs on a cache miss (at most every ~4h), which is comfortably inside GitHub's 5000 req/hr authenticated ceiling for a personal-repo-list-sized fan-out.

**1. Add a small pure helper for parsing the Link header (easy to unit test in isolation):**

```js
// Given a GitHub `Link` response header value, return the total commit count
// implied by the rel="last" page number (per_page=1 means page number == count).
// Returns null if the header is missing/unparseable (caller decides the fallback).
export function parseCommitCountFromLinkHeader(linkHeader) {
  if (!linkHeader) return null;
  const lastMatch = linkHeader.match(/<[^>]*[?&]page=(\d+)[^>]*>;\s*rel="last"/);
  if (!lastMatch) return null;
  const parsed = Number.parseInt(lastMatch[1], 10);
  return Number.isNaN(parsed) ? null : parsed;
}
```

Export it from `repos.js` (named export alongside the default `handler`) so the test file can import and unit-test it directly, separate from the full handler flow.

**2. After the existing filter+sort step (the `activeRepos` array, currently built at the `.filter(...).sort(...).map(...)` chain around lines 63-79), fetch each active repo's commit count BEFORE the final `.map` that builds the client-facing shape** — only fan out for repos that survived the filter (not the raw unfiletered list), to minimize the number of extra calls:

```js
const reposWithCommitCounts = await Promise.all(
  activeReposPreMap.map(async (repo) => {
    try {
      const commitsUrl = `https://api.github.com/repos/${repo.full_name}/commits?per_page=1`;
      const commitsResponse = await fetch(commitsUrl, { headers });
      if (!commitsResponse.ok) {
        console.warn(`⚠️ [GitHub] commit-count fetch failed for ${repo.full_name}: ${commitsResponse.status}`);
        return { ...repo, commitCount: null };
      }
      const linkHeader = commitsResponse.headers.get('link');
      const fromLinkHeader = parseCommitCountFromLinkHeader(linkHeader);
      if (fromLinkHeader !== null) {
        return { ...repo, commitCount: fromLinkHeader };
      }
      // No Link header means there's only one page of results (0 or 1 commits) —
      // the response body's array length tells us which.
      const commitsBody = await commitsResponse.json();
      return { ...repo, commitCount: Array.isArray(commitsBody) ? commitsBody.length : null };
    } catch (err) {
      console.warn(`⚠️ [GitHub] commit-count fetch threw for ${repo.full_name}:`, err.message);
      return { ...repo, commitCount: null };
    }
  })
);
```

Use `reposWithCommitCounts` (not the original pre-commit-count array) as the input to the final `.map(repo => ({ ... }))` that builds the client-facing shape, and add `commitCount: repo.commitCount` to that mapped object, alongside the existing fields. A per-repo fetch failure must NOT fail the whole route — degrade that one repo's `commitCount` to `null` and continue (matches this route's existing defensive style elsewhere, e.g. the `GITHUB_TOKEN`-missing warning-and-continue at the top of the file).
