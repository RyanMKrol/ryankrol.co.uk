# T067 ‚Äî Accept metadata-backfill fields on all four review update APIs

## Do

Each review type's `add.js` API route already accepts optional source-metadata fields that its
sibling `update.js` route currently does NOT accept ‚Äî meaning an existing review can never have
its metadata backfilled/corrected via the update flow today. Fix all four:

- **`src/pages/api/reviews/books/update.js`**: currently accepts only
  `{ title, author, rating, overview, password, originalTitle, originalAuthor }`. Add support for
  the same optional fields `src/pages/api/reviews/books/add.js` accepts:
  `source, olid, coverId, coverUrl, volumeId, bookAuthors, firstPublishedYear, isbn, subjects,
  pageCount, publisher`. All optional ‚Äî persist whichever are present in the request body,
  alongside the existing required fields, unchanged validation for the required ones.
- **`src/pages/api/reviews/albums/update.js`**: currently accepts only
  `{ title, artist, rating, highlights, originalTitle, originalArtist, password }`. Add support
  for the optional `lastfm` object that `src/pages/api/reviews/albums/add.js` accepts, and derive
  `thumbnail` from it the same way `add.js` does
  (`lastfm?.images?.[lastfm.images.length - 1]?.['#text'] || ''`).
- **`src/pages/api/reviews/movies/update.js`**: currently accepts only
  `{ title, rating, gist, password, originalTitle }`. Add support for the same optional fields
  `src/pages/api/reviews/movies/add.js` accepts: `tmdbId, mediaType, posterPath, tmdbOverview,
  tmdbDate`.
- **`src/pages/api/reviews/tv/update.js`**: same as movies ‚Äî add `tmdbId, mediaType, posterPath,
  tmdbOverview, tmdbDate` support, matching `src/pages/api/reviews/tv/add.js`.

In every case: these are metadata fields ONLY. Do NOT change how `rating`, the review-text field
(`overview` for books, `highlights` for albums, `gist` for movies/tv ‚Äî this is the user's own
written review, not sourced from any external API), `title`/`author`/`artist`, or the
originalTitle/originalAuthor lookup keys are validated or handled ‚Äî this task only ADDS optional
metadata fields to what's already persisted, mirroring each type's own `add.js`.

## Done when

- All four `update.js` routes accept the additional optional fields listed above (matching their
  sibling `add.js` exactly) and persist them onto the `PutCommand` item when present in the
  request body.
- Calling each `update` endpoint WITHOUT any of the new optional fields still succeeds exactly as
  before (no new required fields were introduced) ‚Äî existing edit-page behavior is unaffected
  until T068-T071 start sending the new fields.
- A test (new or added to an existing test file for these routes, if one exists ‚Äî otherwise skip
  automated testing for these specific handlers if this repo has no existing API-route test
  pattern to follow, and note that in your build summary) confirms at least one of the four routes
  persists a new optional field when provided and omits it cleanly when absent.
