Both source ideas in this batch (card-decoration brainstorm + sort-options request) independently flagged commit count as a candidate — the brainstorm as a 'decorate the card' metadata idea, the sort request as a 'maybe' sortable dimension — and both flagged it as the SAME underlying data-availability question: if commit-count data gets fetched at all, it becomes available for both display and sort essentially for free.

Confirmed technical constraints (checked against GitHub's REST and GraphQL API docs): GitHub's REST repos-list endpoint (`/user/repos`, `/users/{username}/repos` — what `src/pages/api/github/repos.js` calls today) and the single-repo REST endpoint do NOT return a commit count field at all. Getting one requires either:
(a) An N+1 REST call per repo — `GET /repos/{owner}/{repo}/commits?per_page=1` and reading the `Link` header's `rel="last"` page number as a proxy for total commit count (a common but slightly hacky trick since GitHub's commits endpoint has no direct count field either) — this multiplies GitHub API calls by the repo count, against an already-tight budget (server-side rate limit is 20 req/60s per IP in this app's own `src/lib/rateLimit.js`, on top of GitHub's own 5000 req/hr authenticated ceiling), and would slow down (or require restructuring) the currently-single-request, single-page `withApiCache`-wrapped read in `repos.js`.
(b) Migrating to GitHub's GraphQL API (`https://api.github.com/graphql`), which can batch-fetch `defaultBranchRef.target.history.totalCount` for every repo in one query — much cheaper per-request, but this app is REST-only today (`fetch` calls with a bearer token, no GraphQL client, no query-building/parsing infra) — adopting GraphQL is a real new API surface and a small new dependency/pattern, not a drop-in tweak to the existing `repos.js`.

Given this repo's explicit house rule (root `CLAUDE.md`: 'Keep it simple... no new infra... unless explicitly asked') and that this is a personal site (not a product where commit-count is a meaningfully load-bearing metric), the recommended default is (a) below, but the tradeoff is a genuine judgment call the owner should make explicitly before any code gets written, since it's an architectural choice (new API surface + dependency), not a routine build task.

Separately, NOT part of this decision but worth recording so it isn't silently dropped: lines-of-code (the other metadata idea floated alongside commit count) was assessed and is NOT being pursued at all, no task authored for it — GitHub's API has no LOC endpoint; the closest built-in proxy, `/repos/{owner}/{repo}/languages`, returns BYTES per language, not source line counts, and a true LOC count would require cloning every repo and running a tool like `cloc`/`tokei` (infeasible inside this app's cached serverless read path) or a third-party service (new infra, same house-rule conflict, worse cost/benefit than even the commit-count GraphQL option since LOE is meaningfully higher for a vaguer payoff). This is a clean 'no' with a clear technical reason, not an open question — flagged here for visibility rather than given its own decision task.
