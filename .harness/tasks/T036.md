# T036 — Dashboard: show each task's worklog (+ audit) in its expanded body

## Do

In the local backlog dashboard (`.harness/dashboard/server.js`), surface each task's **worklog**
(and its audit record when present) in the task's expanded detail body, alongside the spec that's
already rendered there. Useful especially for done tasks — to read what the loop actually did — but
apply it to ANY task that has a worklog (failed/blocked tasks have worklogs too).

- The worklog lives at `.harness/worklog/<id>.md`; the loop's audit record (when it exists) lives
  at `.harness/worklog/<id>.audit.md`. These files are committed and readable straight from disk.
- Add read helpers mirroring the existing `readSpec`: e.g. `readWorklog(id)` reading
  `.harness/worklog/<id>.md` and `readAudit(id)` reading `.harness/worklog/<id>.audit.md`, each
  returning `''` when the file is absent (same try/catch pattern as `readSpec`). Resolve them from
  `HARNESS_DIR` (`.harness`), not `ROOT`.
- In `buildBacklog()`, include `worklogContent: readWorklog(t.id)` and
  `auditContent: readAudit(t.id)` on each task object (next to the existing `specContent`).
- In the client `task(t)` template, after the existing spec `<pre>`, render — only when non-empty —
  a labelled worklog block (a small heading like "Worklog" + a `<pre>` of `esc(t.worklogContent)`)
  and, separately, a labelled audit block ("Audit" + `<pre>` of `esc(t.auditContent)`). When a task
  has no worklog, render no worklog/audit block (the spec/`(no spec file)` behaviour is unchanged).
- Keep everything escaped via the existing `esc()` and styled with the existing `pre`/`kv` CSS
  (reuse them; no new dependencies — this stays a pure-Node, dep-free server).

## Done when

- Expanding a task that has `.harness/worklog/<id>.md` shows its worklog text in a labelled `<pre>`
  in the expanded body; if `.harness/worklog/<id>.audit.md` also exists, the audit record shows in a
  separate labelled `<pre>`. Both are HTML-escaped.
- A task with no worklog shows no worklog/audit block and still renders its spec exactly as before.
- This applies across buckets (done, failed, blocked, …) — not just done — wherever a worklog file
  exists.
- Manually verify: run `npm run harness:dashboard`, expand a done task that has a worklog + audit
  (e.g. `T001`), and confirm both the worklog and audit text appear under the spec.
- The dashboard remains dependency-free (no new npm packages); `npm run lint` (incl. the dashboard
  files) and `npm test` pass; the existing `.harness/dashboard/lib.test.js` still passes.
