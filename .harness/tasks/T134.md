## Do

`src/lib/workoutBackfill.js`'s `storeWorkoutInDynamoDB(workout)` (used by the live Hevy-sync backfill, `backfillWorkouts()`, which is auto-triggered on API cache misses per `triggerBackfillAsync`/`src/lib/workoutQueries.js` callers) writes a full workout item to the `Workouts` DynamoDB table with `ConditionExpression: 'attribute_not_exists(id)'`. When that condition fails (`ConditionalCheckFailedException` — a `Workouts` item with this `id` already exists), the function currently just logs '... already exists, skipping' and returns `false` unconditionally, WITHOUT checking whether the existing item is actually complete.

Root cause confirmed via a live read-only production audit (2026-07-02, via a scratch script using the same `docClient` config as `src/lib/dynamo.js`, not committed): there are currently 72 `Workouts` items (dated 2026-03-14 through 2026-06-30 — the most recent workout at the time, i.e. an ONGOING problem, not historical) that exist ONLY as a lean stub shape — `{id, title, start_time, description, exercise_count, created_at, updated_at}` — with NO `data_source`, NO `backfilled_at`, NO `exercises` field, and NO computed metrics (`totalVolume`, `totalWorkingSets`, etc.). Grepping `src/` for `exercise_count` returns zero matches — nothing in this codebase produces that shape, so something OUTSIDE this repo is writing these lean stubs directly to the `Workouts` table (and, per the same audit, ALSO to the `Exercises` table with full real per-set data but no `data_source`/`created_at`/`backfilled_at` metadata — confirmed for all 72 via the `workout_id-index` GSI). Because this app's own `storeWorkoutInDynamoDB` uses `attribute_not_exists(id)` and gives up unconditionally on conflict, once the external process's lean stub exists for a given workout id, THIS APP'S OWN Hevy backfill can never complete that workout — forever, on every future sync run — even though it has the real Hevy API data in hand and would happily write the full record if the id didn't already exist.

`src/scripts/backfillIncompleteWorkoutMetrics.js` (already built, a separate already-`done` backlog task) is a ONE-OFF script that repairs the 72 records that exist TODAY by deriving metrics from the `Exercises` table; a pending backlog task (gated, needs-human) will run it once against production. Neither of those prevents new lean stubs from permanently blocking this app's real data for future workouts, since whatever writes the lean stubs is still active. This task is the forward/ongoing fix: make the app's own live sync self-healing so it no longer needs a human to re-run a one-off script every time this recurs.

Fix `storeWorkoutInDynamoDB` in `src/lib/workoutBackfill.js`: when the initial workout `PutCommand` fails with `error.name === 'ConditionalCheckFailedException'`, do a `GetCommand` (`import { GetCommand } from '@aws-sdk/lib-dynamodb'`) for that `id` before giving up.
- If the existing item already has an `exercises` field (truthy): keep the CURRENT behavior exactly — log 'already exists, skipping', return `false`. Do not touch a genuinely complete record.
- If the existing item is missing `exercises` (same definition as `isIncomplete()` in `src/scripts/backfillIncompleteWorkoutMetrics.js`: `!workoutItem.exercises`): heal it in place with an `UpdateCommand` that SETs `exercises` plus the same computed-metrics fields this function already computes into `metrics` (`calculateWorkoutMetrics(workout)` — `totalVolume`, `totalWorkingSets`, `totalWarmupSets`, `uniqueExercises`, `durationMinutes`, `workoutType`, etc; mirror the field list `buildMetricsUpdate`/`writeMetricsUpdate` in `backfillIncompleteWorkoutMetrics.js` write, for consistency), log that it healed an incomplete record, then CONTINUE ON into the existing exercise-storing loop below (the one that already does `PutCommand` per exercise with `ConditionExpression: 'attribute_not_exists(exercise_id)'` and already tolerates `ConditionalCheckFailedException` per-exercise — leave that loop's behavior as-is, it safely no-ops on exercise rows that already exist). Do NOT `return false` early in this branch — after the update + exercise loop, `return true` so the caller (`backfillWorkouts()` in the same file) counts this as a successful store: its `newWorkouts` counter increments and its `consecutiveExisting` counter resets (both matter — `newWorkouts > 0` is what triggers `clearApiCache(...)` at the end of `backfillWorkouts()`, and resetting `consecutiveExisting` lets the backfill keep scanning further back in the same run instead of stopping early after 10 consecutive 'already exists').

Do not modify `src/scripts/migrateWorkoutData.js`, `src/scripts/auditWorkouts.js`, or `src/scripts/backfillIncompleteWorkoutMetrics.js` — this task is scoped to the live ongoing sync path only (`src/lib/workoutBackfill.js`).
