**Process:**
1. `npm run dev`, open the home page in a browser at a desktop viewport (~1400px) and compare
   directly against `renders/1a.png`: hero type sizes/weights, stat block colors and proportions,
   collection-wall grid density/spacing, latest-takes card styling, gym rail and vinyl shelf rail
   layout, footer.
2. Resize to a mobile viewport (~390px) and compare against `renders/4a.png`: 3-col stat grid, quick-
   link rows replacing the wall, ☰ nav behavior.
3. Also audit the shared components in isolation as they're used across the home page: `Pill`,
   `StatBlock`, `CoverTile`, `StarRating`, `PipMeter`, `Badge`, `SearchInput` — check border widths,
   radius, shadow, font choices (`Bricolage Grotesque` for display, `Nunito` for body, `Space Mono`
   for mono/labels) against the design tokens in the README.
4. For every gap you find (wrong color, wrong spacing, wrong font weight, wrong radius, misaligned
   elements, missing hover/active states the design implies, etc.), fix it directly in the component
   or page CSS/markup. Prefer fixing shared components (`Pill.js`, `StatBlock.js`, etc.) over
   page-specific overrides when the gap is systemic (e.g. if border-radius is wrong everywhere, fix
   the `--radius-card` token in `globals.css`, not each page individually).
5. Re-check after each fix — this is iterative, not a single pass.

## Done when

- The live home page (both breakpoints), nav, footer, and shared components visually match
  `renders/1a.png`/`renders/4a.png` closely — exact hex colors, correct fonts, correct border/radius/
  shadow treatment, correct spacing rhythm. Minor unavoidable differences (real data vs. mock data,
  browser font rendering) are fine; systemic mismatches (wrong colors, wrong fonts, wrong layout
  structure) are not.
- `npm run lint`, `npm test`, and `npm run build` all pass after your tweaks.
- No functional regression: nav links, search, sort, pagination (wherever already wired) still work
  after your CSS/markup changes.
- Document in your worklog which specific gaps you found and fixed (a short list), so future audits
  don't re-discover the same issues.
