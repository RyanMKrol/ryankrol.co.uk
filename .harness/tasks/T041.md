# T041 — Harness dashboard rework — rendering + interactions (server.js)

## Do

T040 reworked `.harness/dashboard/lib.js` to group tasks into four buckets: `ready`, `waiting`,
`needsHuman`, `done` (with `waiting-loop` tasks omitted entirely, `blocked` tasks folded into
`needsHuman`, and `failed` tasks folded into `done`, sorted not-reviewed-first-then-numeric-id).
Update `.harness/dashboard/server.js`'s client-side rendering (the `PAGE` template's embedded
`<script>`) to match the `local-jobs` sibling repo's dashboard Backlog page
(`~/Development/local-jobs/dashboard/app/backlog/page.tsx`) in structure and interaction, while
**keeping this repo's own existing color theme** (the `:root` CSS custom properties already
defined at the top of `PAGE`'s `<style>` block — do NOT adopt local-jobs' cream/orange "Sunny
8-bit" palette; only the layout/behavior should change, not the colors/fonts).

**IMPORTANT constraint:** this file's own header comment states it is "A standalone pure-Node
HTTP server (no npm deps...)". Preserve that — do not add `react-markdown` or any other npm
package. Where local-jobs renders the task spec via React Markdown, either hand-roll a minimal
formatter for the spec's actual shape (it only ever contains `##` headers and `-`/`*` bullet
lists — see any file under `.harness/tasks/*.md` for real examples) using plain string
replacement/regex, or keep the spec as clearly-labeled preformatted text if a minimal formatter
isn't worth the complexity — your call, but zero new dependencies either way.

Concrete changes to make:

1. **Always render all four sections**, even when a bucket is empty — show a "None." placeholder
   inside an empty section rather than hiding the section (current behavior hides empty sections
   via `ORDER.filter(...)`; remove that filtering).
2. **Live auto-refresh**: poll `/api/backlog` every 5000ms (`setInterval`) instead of fetching
   once on page load, re-rendering `#main`/`#counts` on each poll. Update the header copy to say
   "Auto-refreshes." (matching local-jobs' summary line), since it will now actually be true.
3. **Clickable dependency navigation**: in the expanded row body's "needs: ..." / "depends on: ..."
   list, each referenced task id should be a clickable element that scrolls to and expands that
   task's row wherever it currently lives (it may be in a different section), with a brief visual
   highlight (e.g. a temporary background-color flash using the existing `--acc` custom property)
   that fades after ~1.5s. This must work even if the target task is currently hidden inside a
   collapsed `<details>`-style container — expand it first, then scroll+highlight.
4. **Collapsible "Build log"**: wrap the existing worklog content display (currently an
   always-visible `<pre>`) in a native `<details><summary>Build log</summary>...</details>` so it's
   collapsed by default, matching local-jobs. Do the same for audit content if present
   (`<summary>Audit</summary>`).
5. **Bucket pill relabeling**: since T040 changed the bucket taxonomy, update the pill
   rendering to match — a task in the `needsHuman` bucket that has `blocked: true` should show a
   distinguishing pill (e.g. "⚠ blocked (loop gave up)") rather than the plain "needs human" pill,
   so the two different reasons a task needs a human aren't visually conflated. A task in `done`
   with `failed: true` keeps its existing red "failed" pill + Mark-failed/Undo-fail button
   (unchanged); a `done` task without `failed` gets a green "done" pill (unchanged).
6. Keep everything else from the current implementation that T040 didn't touch: the existing
   bulk "select all + mark done" action for the Needs-a-human section (local-jobs only has
   per-row Mark done, but there is no reason to remove a bulk action we already have and works),
   the existing Done-section Show All/Reviewed/Not-reviewed filter + bulk "mark reviewed" action,
   and the existing `/api/mark-*` endpoints (unchanged — this task is presentation/interaction
   only, no new server-side mutation endpoints).

  than disappearing).

  scrolls the page to and expands that task's row, with a brief visible highlight — verified by

  on click, not an always-visible block.

  `failed:blocked`) shows a visually distinct pill/label from a plain gate/needs-human task.
