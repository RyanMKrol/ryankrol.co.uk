This is the deploy checkpoint for the whole Collection redesign (T141-T170). Vercel's Git
integration is disconnected (see root `CLAUDE.md`'s "Deploying" section) — none of the work built
across T141-T170 has gone live yet, no matter how many commits landed on `main`. This task makes no
code changes; it is a pure operational run (mirrors T083's precedent — `scope: []`, diff should be
empty aside from the worklog).

1. `git fetch origin && git status` — confirm the local checkout's `main` matches `origin/main`
   exactly (no ahead/behind). If it doesn't, `git pull` first.
2. `npm run build` locally as a final pre-deploy sanity check (CI should already have passed this on
   every commit, but re-confirm before spending a Vercel build).
3. `vercel whoami` — confirm the CLI is authenticated. **If this fails (not logged in), STOP and
   record `failed:blocked`** with that exact reason — this is a genuine prerequisite this task
   cannot supply itself, a human needs to run `vercel login` once. Do not attempt any workaround.
4. Run the deploy non-interactively and capture its output (the production URL is the last line of
   stdout on success):
   ```sh
   vercel --prod --yes
   ```
5. Confirm the deployment actually reached a ready state — parse the CLI's own exit code (`vercel
   --prod --yes` exits non-zero on failure) AND independently verify by fetching the returned URL:
   ```sh
   curl -s -o /dev/null -w '%{http_code}' <the-production-url>
   ```
   must print `200`.
6. Objectively confirm the LIVE page is actually serving the Collection redesign, not a stale/old
   build — fetch the homepage HTML and grep for markers that only exist post-redesign:
   ```sh
   curl -s <the-production-url>/ | grep -o "Bricolage+Grotesque"
   curl -s <the-production-url>/ | grep -o "background-color:#FBF7EF"
   ```
   both must find a match. (These are the same two markers used to verify T142 locally — see
   `.harness/worklog/T142.md` for why they're the right signal: the font link and the anti-FOUC
   `<html>` background hardcode are both server-rendered and unique to the Collection tokens, so
   their presence is strong evidence this deploy actually picked up the redesign rather than serving
   a cached/stale build.)
