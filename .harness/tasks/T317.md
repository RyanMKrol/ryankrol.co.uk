## Overview
The header's horizontally-scrolling nav pill row (`.collection-nav-pills` in `src/components/Header.js:71-85`, styled at `src/styles/globals.css:874-897`) only responds to native horizontal-scroll input (trackpad swipe, or a mouse's horizontal/tilt scroll wheel). A plain Windows mouse has neither, so the row is completely unreachable past the visible pills on those machines — confirmed by the reporting user on their own Windows desktop. Add click-and-drag as a fallback scrolling mechanism: mousedown starts a drag, mousemove adjusts `scrollLeft` by the pointer delta, mouseup/mouseleave ends it. This only needs to apply at desktop widths — the same row is `display:none` under the mobile breakpoint (`globals.css:2307-2309`, replaced by `.nav-mobile-menu`), which already scrolls fine as a plain vertical list, so no touch/pointer handling is needed there.

## Do
In `src/components/Header.js`, add mouse-drag handling to the existing `<nav ref={navRef} className="collection-nav-pills...">` element (do not replace the existing `scroll`/`resize`/`ResizeObserver` effect that drives `showNavFade` via `shouldShowScrollFade` from `src/lib/scrollFade.js` — dragging changes `scrollLeft` the same way native scrolling does, so that effect keeps working unmodified as long as you mutate `navRef.current.scrollLeft` directly rather than bypassing it).

Implementation shape:
- On `onMouseDown` of the nav: record the starting pointer X and the nav's current `scrollLeft` in refs (not state, to avoid re-render churn on every mousemove), and set an `isDragging` ref to true. Also reset a `totalMovement` accumulator ref to 0.
- Attach `mousemove` and `mouseup` listeners on `window` (not just the nav element) for the duration of the drag, so the drag continues tracking even if the pointer leaves the nav's bounds mid-gesture, and always ends cleanly on mouseup wherever it happens. Remove these listeners on mouseup/unmount.
- On `mousemove` while dragging: compute `deltaX = pointerX - startX`, set `navRef.current.scrollLeft = startScrollLeft - deltaX`, and accumulate `totalMovement += Math.abs(deltaX-since-last-move)` (or track max absolute delta from start — either is fine, the point is knowing whether this gesture was a meaningful drag vs a stationary click).
- On `mouseup` (or the drag ending because the pointer left the window): clear `isDragging`, remove the window listeners. If `totalMovement` exceeded a small threshold (~5px), suppress the click-through so the drag doesn't also register as a pill navigation: the pills are `<Link href=...><Pill>...</Pill></Link>`, i.e. real anchor elements, so a mousedown->drag->mouseup sequence over a link can fire a native `click` and navigate. Set a ref flag (e.g. `suppressNextClick`) when the threshold is exceeded, and add an `onClickCapture` handler on the `<nav>` that calls `e.preventDefault(); e.stopPropagation()` and clears the flag when it's set, leaving normal clicks (movement under the threshold) to navigate exactly as today.
- Prevent the default native drag-image/text-selection behavior during the gesture (e.g. `e.preventDefault()` in `onMouseDown`, and/or `user-select: none` on `.collection-nav-pills` while a `dragging` class or inline style is applied) so dragging over the pill text doesn't select it or start a native link-drag ghost image.
- Cosmetic but expected: swap the cursor to `grab` at rest and `grabbing` while dragging (CSS on `.collection-nav-pills`, e.g. via a `dragging` class toggled in state/ref-driven style, or a simple `cursor: grab` always plus `:active { cursor: grabbing }` if that's sufficient without JS).

In `src/components/Header.test.js`, add RTL coverage: simulate `fireEvent.mouseDown` -> `fireEvent.mouseMove` (with movement past the click threshold) -> `fireEvent.mouseUp` on the nav pill row and assert `scrollLeft` was updated accordingly (you'll need to stub/set `scrollWidth`/`clientWidth`/`scrollLeft` on the nav DOM node the way `Header.test.js` likely already does for the existing fade-mask tests — check the existing test file for the pattern before adding a new one) and that no navigation occurred (e.g. the mocked `next/router` push/route wasn't triggered, or the anchor's default wasn't allowed to run). Also add a small-movement case asserting a plain click (movement under the threshold) still navigates normally.

Per this repo's visual-check convention (root `CLAUDE.md`), a new interactive state needs a `FLOWS` entry in `scripts/_visual-harness.mjs`: add one that simulates the drag (Playwright can dispatch `mouse.down()` / `mouse.move()` / `mouse.up()` at the nav pill row on a page where the row visibly overflows, e.g. by shrinking the viewport or using a page whose nav already overflows at default width) and screenshots the post-drag scroll position, so a scoped visual-check run can confirm the row actually moved.

## Done when
Dragging with the mouse (mousedown + move + mouseup) inside `.collection-nav-pills` scrolls the row left/right proportionally to the drag distance, on both left and right drag directions, and stops cleanly on mouseup even if the pointer left the row first. A drag exceeding ~5px of total movement does not also trigger navigation to whichever pill the pointer ends up over; a plain click (no/negligible movement) still navigates normally. The existing right-edge scroll-fade mask (`showNavFade` / `shouldShowScrollFade`) continues to update correctly as the row is dragged, unmodified. `npm run lint`, `npm test`, and `npm run build` all pass; the new Header.test.js coverage exercises both the drag-scroll and drag-does-not-navigate / click-still-navigates behaviors. A new `FLOWS` entry exists in `scripts/_visual-harness.mjs` covering the drag gesture, and `node scripts/visual-check.mjs` runs it successfully.
