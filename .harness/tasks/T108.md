## Do

In `src/pages/api/github/repos.js`, the `GET /api/github/repos` handler fetches the owner's repos from the GitHub API and returns them for the public-facing `/projects` page (`src/pages/projects/index.js`, which renders every repo in `data.repos` with no client-side filtering — it trusts the API to only return what should be public).

When `GITHUB_TOKEN` is set (the normal production case), the handler calls the *authenticated* `GET /user/repos?affiliation=owner` GitHub endpoint (see line 41), which — unlike the public `/users/{username}/repos` fallback — includes the owner's PRIVATE repos too. Each raw GitHub repo object already carries a `private` boolean, which the handler maps through to the response as `isPrivate: repo.private` (line 83), but nothing in the pipeline actually filters private repos OUT. So today, any private repo owned by the account is returned by the API and rendered as a public-facing project card — a real information leak on a public personal site.

Fix: exclude private repos from the response. The cleanest place is the existing `.filter()` call at lines 65-68:
```js
.filter(repo => {
  // Filter out forks; archived repos are included
  return !repo.fork && repo.pushed_at;
})
```
Add `&& !repo.private` to that same predicate (and update the comment to mention private repos are excluded too), so private repos never make it into `activeRepos` / the `repos` array in the JSON response, regardless of which upstream endpoint fetched them (this also makes the code correct if `GITHUB_TOKEN` is absent and the public-only fallback endpoint is used, since that endpoint should never return `private: true` items anyway — the filter is a defense-in-depth guarantee either way).

Leave `isPrivate: repo.private` in the `.map()` (line 83) as-is — it's now always `false` for anything that survives the filter, harmless to keep, and cheap to leave in case a future feature wants it.

No changes needed to `src/pages/projects/index.js` — once the API never returns private repos, the existing render-everything-in-`data.repos`  behavior is correct.
