## Overview
Owner follow-up on idea #3 (pr-pill-size-round2-spacing). The owner was asked whether they wanted a T331-style dev chooser page again for round 2 of PR-pill sizing, and instead said: 'The style I'm mainly concerned with is that the styling on the workouts page doesn't look right on the workouts page. When I click into an individual workout and look at the cards, it actually looks fine... If we could align the two to be what the individual workout page styling is, I'd be fine with that.' Investigation confirmed this is NOT a `.pr-badge` CSS problem — both pages render the exact same shared `<PrBadge>` component (src/components/Badge.js:39-46) applying the identical `.pr-badge` class (src/styles/globals.css:2776-2782: gap 0.2rem, padding 0.1rem 0.45rem, font-size 0.7rem — T333's shipped state, unchanged). The discrepancy is entirely CONTEXT: on /workouts/[id].js (lines 225-274), the exercise name is a separate `<h3 className="workout-exercise-card-title">` (font-size 1.125rem, bold, font-family var(--font-display), margin-bottom 1rem — src/styles/globals.css:2289-2295) fully decoupled from the pills; the pills instead sit inside `.workout-set-row-badges` (src/styles/globals.css:2322-2329: `display:flex; align-items:center; gap:0.4rem; justify-content:flex-end; padding: 0 0.75rem`), itself inside a padded, tinted `.workout-set-row` (src/styles/globals.css:2303-2311: `padding: 0.5rem 0.75rem; background-color: var(--color-set-row-bg)`) — next to a short lowercase set label ('set 1'/'warmup'), NOT next to the exercise name at all. By contrast, on /workouts/index.js (lines 242-248), inside `.workout-session-exercise-item` (src/styles/globals.css:1998-2006: `font-family: var(--font-mono); font-size: 0.85rem; line-height: 1.6rem; white-space: nowrap`), the exercise `{name}` is a bare text node and `<PrBadge>` is placed immediately, inline, directly abutting the name with zero wrapper/gap/padding in a dense nowrap mono text line — no flex container, no separation, no breathing room. That flush, unpadded, dense-text placement — not the pill's own size — is why the identical pill 'reads' heavier on the list page. This is a clear, mechanical, well-defined CSS/markup context difference, so per the owner's explicit preference ('if we could align the two... I'd be fine with that') this ships as a single direct fix, NOT a chooser — no dev comparison page, no needs-human variant pick.
