This IS a genuinely separate, structural bug from container-audit-1 (not merely the same root cause manifesting differently): CSS Grid's implicit minimum track size for a bare `1fr` track is `minmax(auto, 1fr)`, where `auto` resolves to the largest MIN-CONTENT size of any item placed in that track — NOT `0` — and the sizing algorithm will NOT shrink a track below that minimum even if it means the grid exceeds its container's available width. The one unguarded long-text risk found in `.project-card`'s children: `.project-card-name` (globals.css lines 1877-1883, the repo name link rendered from `repo.name` in `src/pages/projects/index.js:117-120`) has no `overflow-wrap`, `word-break`, or `white-space` rule — a GitHub repo name is commonly a long unbroken kebab-case string (e.g. `ryankrol-co-uk-collection-redesign`) with few/no natural wrap points at typical card widths, so it can force that grid track (and therefore the row's cards) wider than its fair 1/3 share, pushing the 3rd column past the container's right edge — exactly Idea 2's screenshot symptom.
