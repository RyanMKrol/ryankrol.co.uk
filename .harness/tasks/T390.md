## Overview
src/pages/reviews/perfumes/index.js is the only one of the five review-type index pages (movies/tv/books/albums/perfumes) that doesn't paginate its filtered list — it renders the full `filteredPerfumes` array straight into `MasonryColumns` with no page state and no `<Pagination>`. Every sibling (e.g. src/pages/reviews/books/index.js) follows the same recipe: import `Pagination` from '../../../components/Pagination' and `paginate` from '../../../lib/pagination', define `const PAGE_SIZE = 12`, hold a `currentPage` state, compute `const { items: pagedItems, page, pageCount } = paginate(filteredX, currentPage, PAGE_SIZE)`, feed the paged slice into `MasonryColumns`, render `<Pagination currentPage={page} totalPages={pageCount} onPageChange={setCurrentPage} />` below the grid, and reset `currentPage` to 1 in a dedicated `useEffect` whenever the search/sort (and here, ownership) filters change. Separately, the page currently shows a 'Found X perfumes' count (lines 133-137) only when a search term or non-'all' ownership filter is active, and has no empty-state at all when the filtered list is empty (MasonryColumns just renders empty column divs). Both changes are combined into one task because they land in the exact same render region of the same small file and interact directly: the empty-state card must key off the filtered (pre-pagination) list being empty and must render INSTEAD OF the grid + pager (a Pagination component showing '1' of '1' pages over zero items, or an empty grid with a dangling pager, would look broken).
