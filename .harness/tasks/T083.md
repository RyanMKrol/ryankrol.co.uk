# T083 — Run the incomplete-workout-metrics backfill against production

## Do

T082 built `src/scripts/backfillIncompleteWorkoutMetrics.js` (dry-run by default, real writes
behind a `LIVE=1` flag) and its npm script. This task is a pure operational run — it makes no code
changes, so its diff should be empty aside from the worklog. Using the AWS credentials already
available in this checkout's `.env.local` (root `CLAUDE.md`'s standard `vercel env pull
--environment production` convention):

1. Before touching anything, do a full `ScanCommand` of the `Workouts` table and save the raw
   result (e.g. to a scratch file, not committed) as a BEFORE snapshot. From it, compute the exact
   set of item ids missing the `exercises` field — this is the objective "incomplete" set (at time
   of writing, 72 items dated 2026-03-14 through 2026-06-30; the live count may differ by the time
   this runs since new incomplete workouts keep appearing until T134 ships its forward-fix).
2. Run `npm run workout:backfill-metrics` (dry-run, no `LIVE=1`) and confirm the set of workout ids
   it reports it will update EXACTLY matches the incomplete set computed in step 1. If they don't
   match, or the script throws/exits non-zero, STOP — record `failed:blocked` with the discrepancy;
   do not proceed to a live run.
3. Re-run live: `LIVE=1 npm run workout:backfill-metrics`.
4. Scan the `Workouts` table again (AFTER snapshot) and verify, for every item, ALL of the
   following — this is the objective pass/fail bar, not a subjective spot-check:
   - Every item that was in the incomplete set now has non-null `exercises`, `totalVolume`,
     `totalWorkingSets`, `uniqueExercises`, `durationMinutes`, and `workoutType` fields.
   - For every item that was ALREADY complete in the BEFORE snapshot (not in the incomplete set),
     its record is byte-identical between BEFORE and AFTER — the script must never modify an
     already-complete workout.
   - For each now-repaired item, cross-check its new `exercises`/`totalVolume`/`totalWorkingSets`
     against a fresh, independent query of the `Exercises` table for that `workout_id` (via the
     `workout_id-index` GSI) run through `calculateWorkoutMetrics`/`calculateExerciseMetrics`
     (`src/lib/workoutMetrics.js`) directly — the two must agree, confirming the write matches what
     the real underlying exercise data actually supports, not just "some non-null value got written".

If any check in step 4 fails, STOP — do not attempt to hand-fix data; record `failed:blocked` with
the specific mismatch(es) found so a human can investigate before any retry.

## Done when

The live backfill has been run against production `Workouts` (or, if the BEFORE scan finds zero
incomplete items, the run is skipped as a no-op — record that finding instead), AND every check in
step 4 passes with zero mismatches. Record the final incomplete-count/repaired-count/verified
summary in this task's worklog as the evidence. (This unblocks the workouts-list and
programme-stats symptoms described in T134/T135's originating idea — repairing the 72
currently-broken records is the other half of that fix, alongside T134's forward-fix preventing new
ones.)
