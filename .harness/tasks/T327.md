Capping at STORE time (not render time) means both the candidate radio-list render (~line 135, `candidates.map`) and the existing status label (~line 110, `${row.candidates.length} candidate(s) found`) automatically reflect the capped count with no other code change — do not add a separate `.slice()` at the render call site, it would make the label say '5 candidates found' while only showing 3.

2. **Add a page-level 'Apply all selections' button.** Add `const [applyingAll, setApplyingAll] = useState(false);` alongside the existing `page`/`rowState` state. Add a `confirmAll` handler:
```js
const confirmAll = async () => {
  if (applyingAll) return;
  const toApply = pageItems.filter((item) => {
    const row = rowState[item.id];
    return row && row.status === 'found' && row.selectedIndex !== null && row.selectedIndex !== undefined;
  });
  if (toApply.length === 0) return;
  setApplyingAll(true);
  for (const item of toApply) {
    // eslint-disable-next-line no-await-in-loop
    await confirmRow(item);
  }
  setApplyingAll(false);
};
```
Sequencing decision (checked src/lib/backfillQueue.js and grepped src/pages/api/reviews/*/update.js): the existing SEARCH phase runs through `runBackfillQueue` with a 1500ms `minSpacingMs` specifically because it hits externally-rate-limited metadata APIs (TMDB/Google Books/Last.fm, guarded by src/lib/rateLimit.js). The SAVE path each `onConfirm` calls (e.g. `POST /api/reviews/movies/update`) is this site's own internal, password-gated Dynamo-write route and is NOT wrapped in `checkRateLimit` anywhere under `src/pages/api/reviews/`. So reusing `runBackfillQueue` itself (built for a `searchFn` that returns candidates or throws, with retry-after handling) is a poor fit for `confirmRow` (which already catches its own errors internally, sets row status to `'found'` + an error message, and does not rethrow) — do NOT route saves through `runBackfillQueue`. Instead use the plain sequential `for...of` loop above: it still runs saves ONE AT A TIME (never bursts them in parallel, matching the site's general non-bursting discipline) without inventing artificial delay for a call that isn't externally rate-limited. Reuse the EXISTING `confirmRow` function as-is (do not duplicate its save/error logic) — it already sets `status: 'saving'` then `'saved'` or `'found'`+error per row, so per-row progress is already visible via the existing `statusLabel()` function with no new UI needed for that. Because `confirmRow` swallows its own errors and returns normally, one row's save failure does NOT stop the loop — the `for` loop continues to the next selected row automatically. This is the partial-failure behavior: after `confirmAll` finishes, any failed row is back in `status: 'found'` with `row.error` set and its selection retained (so the existing per-row error message shows, and that row can be retried either via its own 'Confirm' button or by clicking 'Apply all selections' again — rows already `'saved'` are excluded from `toApply` so they're never re-submitted).

Render the button as a new `.bbl-page-actions` div, placed as the FIRST child inside the top-level `<div className="bulk-backfill-list">`, before `{pageItems.map(...)}`:
