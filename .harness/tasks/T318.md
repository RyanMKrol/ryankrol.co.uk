## Overview
On /reviews/perfumes, a perfume card's Longevity and Projection progress bars (the filled track beneath 'LONGEVITY'/'Light <-> All-day' and 'PROJECTION'/'Skin scent <-> Room-filling') have gone invisible -- the headings/labels render, but the filled bar underneath is blank space. This is a regression, confirmed via `git log -p --follow -- src/components/perfumeVariants/Variant6Hybrid.js`: commit 0708eac ('T301: add hover/focus tooltips to perfume card Longevity/Projection bars', 2026-07-06) wrapped each bar's track div in a new `<Tooltip>` (now Variant6Hybrid.js:83-90 for longevity, :99-106 for projection); before that commit the track/fill divs were unwrapped direct children and rendered fine.

Root cause (pinned down, not hypothesis): `Tooltip`'s root element (src/components/Tooltip.js:3) is `<span className="tooltip-wrapper">`. `.tooltip-wrapper` (src/styles/globals.css:3192-3195) is `position: relative; display: inline-flex;` with NO explicit `width` -- it shrink-to-fits its content. The wrapped child, `.perfume-v4-scale-track` (globals.css:3582-3588), is styled `width: 100%`; as a flex item this becomes a percentage flex-basis resolved against the flex container's own indefinite/auto width -- a circular sizing case browsers collapse to ~0 width. That's exactly the symptom: heading/label text (outside the Tooltip) is unaffected, only the bar collapses.

Confirmed this is the ONLY place the bug can occur: `Tooltip` is used in exactly one other spot app-wide, src/pages/projects/index.js:237-253, wrapping intrinsically-sized `<span>` content (star count / commit count) -- never a `width:100%` block -- so that usage never triggers it.

Use the same class name on both, e.g. `className="perfume-v4-scale-tooltip"`.
2. In `src/styles/globals.css`, add a new rule near the existing `.perfume-v4-scale-track`/`.perfume-v4-scale-fill` rules (around line 3582) targeting that class:
```css
.perfume-v4-scale-tooltip {
  display: flex;
  width: 100%;
}
```
This overrides ONLY this usage's wrapper (not the shared `.tooltip-wrapper` base rule) to stretch to the full width of the `.perfume-v4-scale` container it sits in, so the `width:100%` track child inside it resolves against a definite width again and its fill renders at the correct proportion.

Do NOT modify the base `.tooltip-wrapper` rule (globals.css:3192-3195) itself -- the only other Tooltip usage in the app (src/pages/projects/index.js:237-253, star/commit count pills) depends on it staying `inline-flex` with no explicit width (shrink-to-fit around small text/icon content); changing the shared rule risks visibly breaking that unrelated usage. Do NOT change `.perfume-v4-scale-track`/`.perfume-v4-scale-fill` (globals.css:3582-3594) -- those rules are already correct and were not the actual defect.
