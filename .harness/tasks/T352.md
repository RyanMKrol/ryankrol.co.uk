## Overview
The owner confirmed GOOGLE_BOOKS_API_KEY IS correctly set in Vercel's Production environment, so this is not a missing-key issue. Failures are intermittent and bursty (most rows failed one night, only 4-5 failed the next day) — consistent with Google's own upstream quota/rate-limiting kicking in under bursty read patterns, not a hard configuration gap. Currently src/pages/api/books/search.js wraps ANY failure from searchGoogleBooks() (including a real upstream 429) into a generic `500 {message: 'Error searching books', error: error.message}` response (search.js lines ~28-31), losing the actual upstream HTTP status and any diagnostic detail. Because src/lib/backfillQueue.js's retry logic only retries an item when the thrown error carries a numeric `.retryAfterSeconds` — which src/pages/reviews/books/backfill.js's handleSearch() (around lines 28-45) only sets when THIS APP's own route returns a 429 — an upstream Google 429 flattened to a 500 is never retried and is reported as a permanent, unhelpful error for every affected row.
