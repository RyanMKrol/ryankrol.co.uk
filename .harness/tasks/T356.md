## Overview
Give Hot Takes the same edit+delete capability every review type already has: a list page of edit links, a single detail page combining an edit form and a delete button, and two new password-gated API routes. Hot Takes' data model ({id, text, date}) is much simpler than a review, so this is a small, well-precedented full-stack task copying the existing perfumes edit flow.

## Do
1) Create src/pages/api/hot-takes/update.js (POST, mirroring src/pages/api/reviews/perfumes/update.js's structure but adapted to hot takes' simpler shape): parse { id, text, password } from req.body; gate on password !== process.env.RYANKROL_SITE_KEY -> 401; validate text via validateHotTakeText from '../../../lib/hotTakes' -> 400 with message 'A non-empty take under 500 characters is required' if invalid; GetCommand the existing item by id to preserve its original `date` field (id never changes for hot takes since there's no derived-key recompute like perfumes has); PutCommand the updated item back as { id, text, date: preservedDate } into DYNAMO_TABLES.HOT_TAKES_TABLE; call clearApiCache('api-hot-takes'); respond 200 with { message, hotTake }. 2) Create src/pages/api/hot-takes/delete.js (DELETE, mirroring src/pages/api/reviews/perfumes/delete.js): parse { id, password } from req.body; gate on password; validate id present -> 400; DeleteCommand by { id } from DYNAMO_TABLES.HOT_TAKES_TABLE; clearApiCache('api-hot-takes'); respond 200. 3) Create src/pages/hot-takes/edit.js (mirroring src/pages/reviews/perfumes/edit.js): fetch GET /api/hot-takes, sort newest-first via sortByDateDesc from '../../lib/hotTakes' (already exported), render a list where each hot take's text (truncate/wrap as needed) links to `/hot-takes/edit/${encodeURIComponent(take.id)}`. 4) Create src/pages/hot-takes/edit/[id].js (mirroring src/pages/reviews/perfumes/edit/[id].js's combined edit+delete-on-one-page shape): read id from router.query, fetch GET /api/hot-takes and find the matching item by id, pre-fill a form with a single Take field using MarkdownEditor (className="collection-form-textarea", same pattern as hot-takes-4-markdown's add.js change — do not reintroduce a plain textarea here) plus a Password field; on submit POST to /api/hot-takes/update with { id, text, password }; include a Delete button (collection-form-button-danger, with a confirm() prompt matching perfumes' 'Are you sure you want to delete this review? This action cannot be undone.' wording adapted to 'take') that DELETEs to /api/hot-takes/delete with { id, password }; on success of either action, redirect to /hot-takes/edit after a short delay (setTimeout ~2000ms), matching the perfumes edit page's UX. 5) No changes needed to src/pages/hot-takes/index.js (the public list stays read-only, no edit links added there) or src/lib/hotTakes.js (validateHotTakeText and sortByDateDesc already exist and are reused as-is).

## Done when
Visiting /hot-takes/edit while logged into no special auth (page itself is public, like other *_/edit.js list pages) shows every hot take newest-first, each linking to /hot-takes/edit/<id>. Visiting that detail page shows the take's current text pre-filled in a MarkdownEditor and a password field. Submitting with the correct RYANKROL_SITE_KEY password successfully updates the take's text via POST /api/hot-takes/update (verified against DynamoDB or the API response) while preserving its original date, and redirects back to /hot-takes/edit. Clicking Delete (after confirming) with the correct password removes the take via DELETE /api/hot-takes/delete and redirects back to /hot-takes/edit. Submitting either action with a wrong/missing password is rejected with a 401 and shows an inline error, matching the perfumes edit page's error-handling pattern. A visual check (scripts/visual-check.mjs) of the new hot-takes-edit list page and hot-takes-edit-detail page renders correctly with no unstyled/broken elements — scripts/_visual-harness.mjs's PAGES/FLOWS and fixtures must be updated to cover the two new pages per this project's visual-verification convention (a new page/interactive state requires a new PAGES/FLOWS entry in the same change).
