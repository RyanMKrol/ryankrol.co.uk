# T093 — Fix dashboard auto-refresh wiping checkbox selections

## Do

`.harness/dashboard/server.js`'s `setInterval(refreshNow, 5000)` (`server.js:283`) re-fetches
`/api/backlog` and does a wholesale `document.getElementById('main').innerHTML = ...` replace
every 5 seconds (inside the `render` function, `server.js:256`). Since checkbox ticks (the
bulk-select checkboxes for "Mark N reviewed"/"Mark N done") are pure client-side DOM state, this
wipes any in-progress selection before the user can act on it.

The owner asked specifically to mirror how the `local-jobs` sibling repo's React-based dashboard
avoids this same problem (`~/Development/local-jobs/dashboard/app/backlog/page.tsx`). That
dashboard keeps selection in a `useState<Set<string>>` (`const [selected, setSelected] =
useState<Set<string>>(new Set())`, `page.tsx:259`) declared independently of its data-polling
state — the poll (`usePoll`) only updates the fetched task data, never touches `selected`, and each
row's checkbox reads its checked-ness from `selected.has(t.id)` when the row list re-renders. So
selections inherently survive a data refresh because they're driven by a persisted set of ids, not
by raw DOM/browser checkbox state.

Translate this same pattern into our plain-JS (non-React) dashboard: maintain a persisted
selection set as a plain JS variable declared OUTSIDE the `render`/`refreshNow` functions (e.g.
`let selectedIds = new Set();` at module/script scope), update it from the existing checkbox
`onchange` handlers (`toggle...`-style functions already in `server.js`) exactly as today, and when
generating each task row's checkbox HTML during a re-render, set its `checked` attribute based on
`selectedIds.has(t.id)` rather than relying on the DOM's own retained checkbox state (which is
what gets destroyed by the `innerHTML` replace). This way, a full re-render on every poll still
happens (matching local-jobs' architecture, where re-rendering IS how React always works, not
something to avoid), but the freshly-generated markup's checkboxes come back pre-checked according
to the persisted set.

## Done when

- A persisted selection set (independent of the polled task data) drives each checkbox's checked
  state on every render, matching the mechanism in `local-jobs`' `page.tsx:259` translated to plain
  JS/DOM.
- Manually verify: open the dashboard, tick a "select for bulk review/mark-done" checkbox, wait
  through at least two 5-second poll cycles (10+ seconds) without touching anything else, and
  confirm the checkbox is STILL ticked afterward (not cleared by the intervening refreshes).
- The existing "select all" / bulk-action buttons ("Mark N reviewed", "Mark N done") still work
  correctly against whatever is currently selected in the persisted set.
- Selecting a checkbox, then having the underlying data actually change (e.g. a task gets marked
  done between polls) still refreshes the row's OTHER content correctly — only the checked state is
  preserved, not stale task data.
