# T027 — Replace review sort dropdown with segmented sort buttons

## Do

Replace the sort `<select>` dropdown on the review list pages with a row of segmented
**sort buttons**, matching the existing filter-button look (the workouts All/Push/Pull/Legs
filter, the date-range buttons). Build a single shared, reusable component
`src/components/SortButtons.js` and use it on all four review list pages
(`src/pages/reviews/{movies,tv,books,albums}/index.js`), replacing each page's
`.sort-container` + `<select className="sort-select">` block. Add the segmented-button styles
to `src/styles/globals.css` (you may keep or remove the now-unused `.sort-select`/`.sort-label`
rules — don't leave dead selectors referenced by markup that no longer exists).

**Button model (agreed):**
- One **mutually-exclusive field button per sort field**. Exactly one field is active at a time.
- Each button is a **2-state toggle: ascending ↔ descending** — there is **NO off/disabled
  state** (clicking the active button only flips its direction; it never deactivates).
- The **active** button is visually highlighted and shows a **direction arrow** (↑/↓) indicating
  the current direction. Inactive buttons show no arrow (or a neutral one).
- Clicking a **different** field switches the active field to that field at its **default
  direction** (see mapping). Clicking the **active** field **flips** its direction.
- **Initial state on load = Date, descending (newest first)** — preserves today's default order.

**Fields are configurable per page** (this is why it must be a shared, prop-driven component):
- movies, tv, books → **Date, Title, Score** (3 buttons).
- albums → **Date, Title, Artist, Score** (4 buttons — albums has an extra Artist sort today).

So `SortButtons` should accept the list of fields to render (e.g. a `fields` prop / array of
`{ key, label }`), the current `sortBy`, and an `onChange(newSortBy)` callback — it does NOT
own the sort itself; it just drives the page's existing `sortBy` state. The pages keep their
existing comparator logic in the `useEffect` (the `sortBy` string values already exist).

**Direction ↔ existing `sortBy` value mapping** (do not invent new comparator keys — reuse the
ones already handled in each page's sort `useEffect`):

| Field | default-direction value (arrow) | flipped value (arrow) |
|---|---|---|
| Date | `date` — newest first (↓) | `date-desc` — oldest first (↑) |
| Title | `title` — A→Z (↑) | `title-desc` — Z→A (↓) |
| Artist (albums only) | `artist` — A→Z (↑) | `artist-desc` — Z→A (↓) |
| Score | `score` — highest first (↓) | `score-desc` — lowest first (↑) |

Pick a consistent arrow convention and apply it uniformly (e.g. ↓ = "primary/biggest first",
↑ = "reversed"); the exact arrow glyph is your call as long as the active field always shows its
current direction.

## Done when

- `src/components/SortButtons.js` exists as a single reusable component, driven by a per-page
  field list, and is used on **all four** pages (`movies`, `tv`, `books`, `albums`) in place of
  the old `<select className="sort-select">`. No review list page still renders a sort `<select>`.
- albums renders **4** field buttons (Date/Title/Artist/Score); movies, tv, books render **3**
  (Date/Title/Score). Exactly one button is active at any time.
- Clicking a non-active field activates it at its default direction; clicking the active field
  flips ascending↔descending; the active button is highlighted and shows the current-direction
  arrow. There is no way to reach an "unsorted/off" state.
- On first load each page shows Date-newest-first (Date button active, descending) — the same
  order as before this change.
- Every existing sort the dropdown offered is still reachable via the buttons, reusing the same
  `sortBy` comparator values (`date`/`date-desc`/`title`/`title-desc`/`artist`/`artist-desc`/
  `score`/`score-desc`) — no comparator behaviour changes.
- Buttons are styled consistently with the existing segmented filter buttons (theme-driven via
  the CSS custom properties in `globals.css`; no hardcoded colours), and look correct under both
  the default and `matrix-active` themes.
- Manually verify against the running app: load `/reviews/movies` and `/reviews/albums`, confirm
  clicking each field button re-sorts the visible cards and the active/arrow state tracks it.
- `npm run lint`, `npm test`, and `npm run build` all pass; docs updated if any convention changed.
