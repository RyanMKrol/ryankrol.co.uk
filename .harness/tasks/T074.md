# T074 — Perfume "Application notes" field — backend

## Do

Add a new field `applicationSpots` to the perfume review record: an array containing zero or more
of the exact strings `"Wrists"`, `"Elbows"`, `"Clavicles"`, `"Beard"`, `"Back of neck"`,
`"Behind ears"` (a fixed six-item list — do not make this configurable/extensible).

Update `src/pages/api/reviews/perfumes/add.js` and `src/pages/api/reviews/perfumes/update.js` to
accept `applicationSpots` from the request body, validate it, and persist it on the
`PerfumeRatings` item. It is OPTIONAL — a request that omits it must still succeed.

Add a validator to `src/lib/perfumes.js`, mirroring the existing `validatePerfumeRating` style:
`validateApplicationSpots(value)` — true only for an array where every element is one of the six
exact strings above (an empty array is valid). Export it alongside the existing exports.

**Note on the existing `notes` field:** this idea repurposes what the `notes` field MEANS (from
"fragrance notes" commentary to general application-notes commentary) but does NOT rename or
remove the underlying `notes` field/column itself — it stays as the free-text companion to the new
`applicationSpots` checkboxes. This backend task does not need to touch the existing `notes`
field's validation or handling at all; the meaning change is purely a frontend labeling/placeholder
change, handled in the dependent task T075. This task only adds `applicationSpots`.

## Done when

- `src/lib/perfumes.js` exports `validateApplicationSpots(value)`, true only for an array whose
  elements are each one of `"Wrists"`, `"Elbows"`, `"Clavicles"`, `"Beard"`, `"Back of neck"`,
  `"Behind ears"` — an empty array passes, an array containing any other string fails, and a
  non-array value fails.
- A new or updated `src/lib/perfumes.test.js` asserts: `[]` passes, `["Wrists","Behind ears"]`
  passes, `["Wrists","NotASpot"]` fails, a non-array fails. `npm test -- perfumes.test.js` passes.
- `POST /api/reviews/perfumes/add` accepts `applicationSpots` and persists it on the created item
  when provided; omitting it still succeeds. An invalid `applicationSpots` value is rejected with
  HTTP 400.
- `POST /api/reviews/perfumes/update` accepts and persists `applicationSpots` the same way, with
  the same validation and optionality.
