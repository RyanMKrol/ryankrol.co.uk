## Overview
Per T388's recorded decision, add a scoped preprocessing step to the shared Markdown component so a single typed Enter (a CommonMark "soft break") renders as a visible line break in block mode, instead of collapsing to a space — without adopting a site-wide `remark-breaks` plugin.

## Do
In `src/components/Markdown.js`, add a new pure preprocessing function (e.g. `preserveSingleNewlines(src)`, exported alongside the existing `preserveBlankLines`) that, given the raw markdown string, appends a CommonMark hard-break marker (two trailing spaces, i.e. `'  '`) to the end of any line that is immediately followed by another non-blank line (a "soft break" that CommonMark would otherwise collapse to a space). Do NOT touch lines that are already blank — a blank line starts a real paragraph break and must be left for `preserveBlankLines`/ReactMarkdown to handle exactly as today. Do NOT introduce a raw `<br>` HTML tag string — `ReactMarkdown` does not render raw HTML unless `rehype-raw` is added, and this task must NOT add that or any other new package dependency; the two-trailing-spaces hard-break syntax is pure CommonMark and needs nothing new. Apply this new function together with the existing `preserveBlankLines`, in block mode only (the `<ReactMarkdown remarkPlugins={[remarkGfm]}>{preserveBlankLines(children)}</ReactMarkdown>` line, currently `Markdown.js:60`) — compose the two functions (e.g. `preserveBlankLines(preserveSingleNewlines(children))`, or the reverse) so neither pass corrupts or double-counts the other's output; verify this by test (see Done when). Do NOT change the `inline` branch (`Markdown.js:47-58`, used by `ReviewCard.js` for album track highlights) at all — it must render single newlines exactly as it does today, untouched. Do NOT add `remark-breaks`, `rehype-raw`, or any new package dependency.

## Done when
`src/components/Markdown.test.js` gains new tests (in addition to ALL existing tests staying green) asserting:
1. `<Markdown>{'line one\nline two'}</Markdown>` (a single newline, no blank line) renders `line one` and `line two` as visibly separate lines within the same paragraph — e.g. assert a `<br>` element exists between the two text nodes inside the single `<p>`, or equivalent DOM proof that a visible break occurred (NOT collapsed into one space-joined line).
2. The existing blank-line-run behaviour from T382 (`preserveBlankLines`) is unaffected — the existing gap-paragraph tests for `'para one\n\n\npara two'` and `'para one\n\n\n\npara two'` still pass unmodified, and a normal single-blank-line paragraph break (`'para one\n\npara two'`) still renders as exactly two `<p>` elements with no injected `<br>`.
3. Inline mode (`<Markdown inline>{'line one\nline two'}</Markdown>`) is unaffected by this change — a single newline in inline mode still behaves exactly as before, with no new `<br>` introduced there.

No new package dependency is added (`package.json` unchanged). `npm run lint`, `npm test`, and `npm run build` all stay green.
