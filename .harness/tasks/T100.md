## Do

Create `src/lib/rateLimit.js`, a new pure-ish helper module for server-side rate limiting on API routes, following the code style of the existing `src/lib/apiCache.js` (JSDoc comments, `console.log`/`console.warn` visibility, no external deps beyond what's already in package.json).

Export two functions:

1. `checkRateLimit(key, { windowMs = 60_000, max = 20 } = {})` — a fixed-window counter keyed by an arbitrary string `key`. Implementation: keep a module-level `const buckets = new Map();` mapping `key -> { count, resetAt }` (epoch ms timestamps). On each call:
   - If there's no entry for `key`, or `Date.now() >= entry.resetAt`, create/replace the entry as `{ count: 1, resetAt: Date.now() + windowMs }` and return `{ allowed: true, retryAfterSeconds: 0 }`.
   - Else if `entry.count >= max`, return `{ allowed: false, retryAfterSeconds: Math.max(1, Math.ceil((entry.resetAt - Date.now()) / 1000)) }` WITHOUT incrementing (a blocked request shouldn't extend the window).
   - Else increment `entry.count` and return `{ allowed: true, retryAfterSeconds: 0 }`.
   Add a top-of-file comment explaining the per-instance limitation described in `design` above (copy/adapt that paragraph).

2. `getClientIp(req)` — reads `req.headers['x-forwarded-for']` (Vercel sets this; may be a comma-separated list — take the first entry, trimmed), falling back to `req.socket?.remoteAddress`, falling back to the literal string `'unknown'`.

Add a `console.log` on block (mirroring the emoji-prefixed logging convention elsewhere in `src/lib`/`src/pages/api`, e.g. `🚦 [RateLimit] Blocked key=... retryAfterSeconds=...`) so blocked requests are visible in Vercel logs — this is the codebase's established visibility story (see root CLAUDE.md 'Be generous with console.log').
