# T066 — Shared metadata-backfill lookup+confirm modal component

## Do

Build a reusable React component (e.g. `src/components/MetadataBackfillModal.js`, exact name your
call) that all four review edit pages (books, albums, movies, tv — T068-T071) will use to offer a
"Backfill from API" button. This component itself does not know about any specific review type —
it's parameterised by its caller:

- Props should include at minimum: a search function/query (the caller provides how to fetch
  candidates — e.g. a URL to call or a function returning a promise of results), a way to render
  each candidate result (the caller provides what a result "looks like" — title/cover/year/etc.,
  since this varies per type), and an `onConfirm(selectedResult)` callback invoked when the user
  picks a candidate and confirms.
- Behavior: a button that, when clicked, runs the search and opens a modal showing the candidate
  result(s) (handle zero, one, and multiple results). The user selects one candidate and clicks a
  confirm action; the modal then calls `onConfirm` with the selected candidate's data and closes.
  A cancel/close action is also available without calling `onConfirm`.
- This is presentational infrastructure only — it does not itself call any review's `update` API;
  the caller (T068-T071) is responsible for taking `onConfirm`'s payload and applying it to that
  page's form state / submitting it.

You may look at `src/components/BookSearch.js` for an existing (books-only, add-flow) example of a
similar search-then-select pattern, but this new component should be generic across review types,
not books-specific.

## Done when

- A new component exists under `src/components/` implementing the button → search → modal →
  select → confirm/cancel flow described above, accepting a caller-supplied search mechanism and
  result-rendering, and calling `onConfirm(selectedResult)` on confirm.
- The component renders sensibly for zero results ("no matches found"), a single result, and
  multiple results (a selectable list).
- No review-type-specific logic (books/albums/movies/tv field names, specific API endpoints) is
  hardcoded inside this component — it must be reusable by all four of T068-T071 without
  modification.
