# T080 — Run the Open-Library-to-Google-Books book migration against production

## Do

T079 built `src/scripts/migrateOpenLibraryBooksToGoogleBooks.js` (dry-run by default, real writes
behind a `LIVE=1` flag) and its npm script. This task is a pure operational run — it makes no code
changes, so its diff should be empty aside from the worklog. Using the AWS credentials already
available in this checkout's `.env.local` (root `CLAUDE.md`'s standard `vercel env pull
--environment production` convention):

1. Before touching anything, do a full `ScanCommand` of `BookRatingsV3` and save the raw result
   (e.g. to a scratch file, not committed) as a BEFORE snapshot — this is what makes step 4's
   verification possible.
2. Run `npm run books:migrate-openlibrary` (dry-run, no `LIVE=1`) and capture its output: the set
   of record titles it says it will update ("matched") vs. leave alone ("no match — left as
   openlibrary"). If the script throws an exception or exits non-zero on the dry run, STOP —
   record `failed:blocked` with the error; do not proceed to a live run.
3. Re-run live: `LIVE=1 npm run books:migrate-openlibrary`.
4. Scan `BookRatingsV3` again (AFTER snapshot) and verify, for every record, ALL of the following —
   this is the objective pass/fail bar, not a subjective "looks reasonable" call:
   - Every record the dry run listed as "matched" now has `source: 'googlebooks'`, a `volumeId`,
     and no `olid`/`coverId` fields.
   - Every record the dry run listed as "no match" is byte-identical to its BEFORE snapshot
     (still `source: 'openlibrary'`, nothing changed).
   - For EVERY record regardless of match status, `title`, `author`, `rating`, and `overview` are
     byte-identical between the BEFORE and AFTER snapshots (the script must never touch the user's
     own authored review content, only source-metadata fields).
   - No record present in the BEFORE snapshot is missing from the AFTER snapshot (nothing was
     deleted).

If any check in step 4 fails, STOP — do not attempt to hand-fix data; record `failed:blocked` with
the specific mismatch(es) found so a human can investigate before any retry.

## Done when

The live migration has been run against production `BookRatingsV3` (or, if the BEFORE scan finds
zero `source === 'openlibrary'` records to begin with, the run is skipped as a no-op — record that
finding instead), AND every check in step 4 passes with zero mismatches. Record the final
matched/no-match/verified counts in this task's worklog as the evidence.
