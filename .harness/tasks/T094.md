# T094 — Per-spot spray count for Application spots — backend

## Do

T074 added `applicationSpots` to the perfume review as a flat array of ticked spot labels (e.g.
`["Wrists", "Night"]`). Change this to also carry a per-spot spray count: replace the shape with
an array of `{spot, sprays}` objects, e.g. `[{"spot": "Wrists", "sprays": 2}, {"spot": "Night",
"sprays": 1}]`. A spot not present in the array means it wasn't applied (0 sprays) — this replaces
the old flat-string-array shape entirely, it does not add a second parallel field.

Update `src/lib/perfumes.js`'s `validateApplicationSpots(value)` to validate the NEW shape: true
only for an array where every element is an object with `spot` (one of the six exact strings
`"Wrists"`, `"Elbows"`, `"Clavicles"`, `"Beard"`, `"Back of neck"`, `"Behind ears"`) and `sprays`
(a positive integer — reject 0 or negative; a spot with 0 sprays should simply be omitted from the
array, not included with `sprays: 0`). An empty array remains valid (no spots applied).

Update `src/pages/api/reviews/perfumes/add.js` and `update.js` to accept, validate, and persist
this new `applicationSpots` shape (they already accept/persist the field from T074 — just the
validator and the shape of what's considered valid changes here). Both routes remain optional for
this field.

## Done when

- `validateApplicationSpots(value)` in `src/lib/perfumes.js` now validates the `{spot, sprays}[]`
  shape: true for `[]`, true for `[{"spot":"Wrists","sprays":2}]`, false for a `sprays` of 0 or
  negative, false for an unrecognised `spot` string, false for the OLD flat-string shape
  (`["Wrists"]` should now fail validation, since the shape has changed).
- `src/lib/perfumes.test.js` is updated to test the new shape (replacing/updating any assertions
  that tested the old flat-string-array shape). `npm test -- perfumes.test.js` passes.
- `POST /api/reviews/perfumes/add` and `POST /api/reviews/perfumes/update` both accept and persist
  the new `{spot, sprays}[]` shape for `applicationSpots`, rejecting an invalid shape with HTTP 400,
  while still treating the field as optional overall.
