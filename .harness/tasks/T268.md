## Do

`.harness/consolidate-ideas.mjs` is the Stage-3 consolidation script `/convert-ideas` runs once per sweep (via `.harness/consolidate-ideas.sh`, under the loop's shared lock) to allocate real task ids for every per-idea agent's `.harness/.pending-tasks/<slug>.json` file and merge them into `.harness/TASKS.json`. Today it has no awareness of the deploy-task convention documented in `.harness/CLAUDE.md` (~lines 15-50): "at most one pending deploy task titled exactly 'Deploy pending site changes to production'; every new site-touching task's id gets added to its `dependsOn`, or a fresh one is authored if none is pending." That convention is currently something each authoring pass has to remember and has already been violated once (the T171/T193/T188 double-deploy-task incident, caught by the owner, not by tooling).

Add automatic enforcement right after step 5 ("Build final task objects" — the `newTaskObjects` array, built by the `.map()` call ending at `.harness/consolidate-ideas.mjs:145` with `return obj; });`) and before step 6 ("Merge into TASKS.json" — `tasksDoc.tasks.push(...newTaskObjects);` at `.harness/consolidate-ideas.mjs:148`):

1. 2. If there are zero site-touching new tasks, do nothing further — skip straight to the existing merge step.
3. Otherwise, look for an existing task in `tasksDoc.tasks` (the tasks already on disk, read at the top of this script into the `tasksDoc` variable) whose `title` is exactly `"Deploy pending site changes to production"` and whose `status` is `"pending"`.
   - If found: push every site-touching new task's real id (`a.realId`) into its `dependsOn` array, deduping against ids already present (union, don't duplicate).
   - If NOT found (none pending — the most recent one, if any, already flipped to `done`, or none has ever existed): allocate a NEW task id via the SAME `allocId()` function already used for every other task in this pass (so it participates in `NEW_MD_FILES` / gets committed by the shell wrapper automatically), and construct it with the exact same shape T193/T216/T238 use — read `.harness/tasks/T193.md` for the canonical spec body to reuse as a template (adjust only the `dependsOn` prose to name the site-touching tasks from THIS sweep), and read `.harness/TASKS.json`'s T238 entry for the exact field values to copy: `gate: null`, `scope: []`, `facets: {"layer":"harness","workType":"config","risk":["cross-cutting"]}`, `title: "Deploy pending site changes to production"`, `dependsOn` = the site-touching new tasks' real ids, `expectsTest: false`, `tags: []` (or whatever T238 uses). Record what happened (created a new deploy task with id X depending on [ids], OR updated existing pending deploy task Y's dependsOn to add [ids], OR "no site-touching tasks this sweep, no action") into the `summary` object written in the existing step 8, and fold one short line about it into `suggestedCommitMessage`.
