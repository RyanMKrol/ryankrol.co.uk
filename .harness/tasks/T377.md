## Overview
The shared Markdown editor's toolbar (Bold / Italic / List / Link) is add-only: clicking Bold on already-bold text wraps it again (**text** -> ****text****) instead of un-bolding it. Make Bold, Italic, and List TOGGLE — a second click on already-formatted text removes the formatting — so every consumer of the shared component (book review add/edit, top-of-mind edit, hot-takes add/edit) benefits from one fix.

## Do
Edit `src/components/MarkdownEditor.js` (the shared component; do NOT touch the consumer pages — the fix belongs in the one shared component so all of them get it). Change the toolbar behaviour so Bold, Italic, and List toggle formatting off when it is already present, rather than always adding another layer:

- **Bold** (`**`) and **Italic** (`*`): on click, detect whether the current selection is already wrapped in the marker and, if so, REMOVE it; otherwise wrap as today. Handle BOTH forms of 'already wrapped': the markers sitting immediately OUTSIDE the selection (e.g. `**[test]**`) and the markers INCLUDED in the selection (e.g. `[**test**]`) — a toggle-off must recognise and strip both. Ensure a bold `**` run is not mistaken for italic `*` (italic detection must not fire on a `**`-wrapped selection, and vice-versa).
- **List** (`- ` line prefix): if the selected line(s) already start with `- `, REMOVE the prefix; otherwise add it as today (mirror the existing `insertLinePrefix` line-handling — operate on the line the selection starts on, consistent with current behaviour).
- **Link**: leave as INSERT-ONLY — a click still wraps the selection as `[selection](url)`, unchanged. (Link toggling is out of scope for this task per the owner decision; do not attempt to detect/unwrap existing links.)

The two pure helpers `wrapSelection` (line 3) and `insertLinePrefix` (line 12) are the natural place for the detect-and-strip logic (add a toggling variant, or a companion unwrap helper, keeping them pure so they stay unit-testable). Preserve the existing focus/selection-restore and scroll-position-restore behaviour in `applyChange` (a toggle-off must still restore focus and a sensible selection, and must not break the long-textarea scroll-restore path the existing test asserts).

A first click on unformatted text must STILL wrap exactly as before — the existing 'wraps the selection' assertions must keep passing; you are ADDING toggle-off, not replacing toggle-on.
