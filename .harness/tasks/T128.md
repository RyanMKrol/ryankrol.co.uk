## Do

Add server-side de-duplication of near-identical Last.fm album search results, used by the album add/edit flows and the vinyl add flow (all go through `GET /api/lastfm/album-search`, which calls `mapAlbumSearchResult` per result in `src/lib/lastfm.js`).

Context: Last.fm's `album.search` API already returns results sorted by relevance (per Last.fm's own API documentation) and offers no additional sort/relevance query params to tune — there is nothing to "turn on" there. The real problem (the owner reported it with a screenshot of searching "808s and Heartbreaks") is that Last.fm's crowd-tagged database contains multiple near-duplicate entries for what is really the same album, e.g. "808s and Heartbreaks", "Kanye West 808s And Heartbreaks", "808s And Heartbreaks (feat. Kid Cudi)", "808s and Heartbreaks ft Lil Wayne", "808s & Heartbreak" — all surfaced as separate rows with nothing collapsing them, making it hard to tell which one is "the right one".

Add a new pure function `dedupeAlbumResults(results)` to `src/lib/lastfm.js` (alongside the existing `mapAlbumSearchResult`/`mapAlbumInfo`) that takes an array of ALREADY-MAPPED result objects (the shape `mapAlbumSearchResult` returns: `{ title, artist, mbid, url, image }`) and collapses near-duplicates:

- Build a normalized key per result: lowercase the title, strip a leading artist-name prefix if the title starts with the artist name (case-insensitive), strip trailing/parenthetical qualifiers like "(feat. X)", "feat X", "ft X", "ft. X", strip punctuation (`&`, `.`, `,`, `'`), collapse "&" and "and" to the same token, collapse repeated whitespace, trim. Combine with a normalized artist (lowercase, trimmed, punctuation-stripped) to form the group key.
- Within each group of results sharing a normalized key, keep exactly ONE representative: prefer the first result in the group that has a non-empty `mbid` (Last.fm's canonical MusicBrainz ID — a stronger identity signal than the fuzzy name match); if none in the group has an `mbid`, keep the first result in the group (preserving Last.fm's original relevance order).
- Preserve the relative order of the surviving representatives exactly as they first appeared in the input — do not re-sort, only remove the later duplicates.
- Do not mutate the input array; return a new array.

Wire it into `src/pages/api/lastfm/album-search.js`: it currently ends with `const raw = Array.isArray(albums) ? albums : [albums]; return raw.map(mapAlbumSearchResult);` (around line 40-43). Import `dedupeAlbumResults` from `../../../lib/lastfm` and change the final line to `return dedupeAlbumResults(raw.map(mapAlbumSearchResult));`. This affects every caller of this endpoint (album add `src/pages/reviews/albums/add.js`, album edit `src/pages/reviews/albums/edit/[id].js`'s backfill modal, and vinyl add `src/pages/vinyl/add.js` — all go through `src/components/LastfmAlbumSearch.js` / this same API route), with no caller-side changes needed.

Add unit tests in `src/lib/lastfm.test.js` (existing co-located file — follow its existing `describe`/`it` + plain-object-in/out style) for `dedupeAlbumResults` covering: the "808s and Heartbreaks" family of near-duplicate titles collapsing to one result; an mbid-bearing duplicate being preferred as the survivor over a duplicate without an mbid in the same group; genuinely distinct albums (different titles/artists) all surviving unchanged; and that surviving order matches first-appearance order in the input.
