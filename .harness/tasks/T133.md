## Do

This is a needs-human task by explicit owner decision: it modifies the harness's own core loop mechanics (`.harness/loop.sh`'s push behaviour), and the owner's stated policy is that harness-modifying-harness changes always require a human in the loop rather than being built by the unattended autonomous loop itself. This task is gated out of `.harness/loop.sh`'s own task selection by `gate: "needs-human"` (see `HARNESS.md` §2 — SELECT skips needs-human tasks). A human (optionally pairing with an interactive Claude Code session they are directly supervising in this checkout, NOT the unattended `loop.sh`/`supervise.sh` run) should make this change directly on `main`.

Background: on 2026-07-02, a burst of ~10 task commits within under an hour (T090–T099) caused Vercel's Hobby-tier build rate limit to silently reject builds after commit `7a166ec` (~00:18 UTC) — confirmed via `gh api repos/RyanMKrol/ryankrol.co.uk/commits/<sha>/status`, which showed a Vercel status check with `state=failure` and `target_url` containing `upgradeToPro=build-rate-limit`. Production then served a ~10h-stale build with no alert anywhere — GitHub Actions CI (`ci.yml`) only runs lint/test/build, never an actual Vercel deploy, so a fully green harness loop gave zero signal that prod had stopped updating. It was manually fixed via `npx vercel@latest --prod --yes`. The owner explicitly chose to throttle/cooldown pushes (NOT to add a post-push Vercel-deploy-status check — that alternative was considered and rejected in favour of this approach).

**What to build:** add a minimum cooldown between successive pushes to `origin/main` from `.harness/loop.sh`, so a burst of task completions can't fire pushes (and therefore Vercel deploys) faster than Vercel's Hobby-tier build rate limit allows.

Current push call sites in `.harness/loop.sh` (verify exact line numbers against the current file — they may have shifted since this was written):
- `mark_done()` — around line 214 — pushes the `status=done` bookkeeping commit after CI goes green.
- `reconcile_overlays()` — around line 252 — pushes when an owner overlay (human-done/manual-fail) is reconciled into `TASKS.json`.
- `block_task()` — around line 740 — pushes a `failed:blocked` marker.
- the main per-task integration push — around line 835 — `git -C "$ROOT" push origin "HEAD:$MAIN_BRANCH"` — this is the one that lands each task's actual code commit and is the primary driver of the T090–T099 burst.

All four currently call `git ... push origin "HEAD:$MAIN_BRANCH"` directly and independently (grep `.harness/loop.sh` for `push origin` to find every call site — there may be more than these four). Introduce ONE shared helper (e.g. `throttled_push()`) that every call site routes through, and have it enforce a minimum wall-clock interval since the previous successful push before it runs `git push`, sleeping out the remainder of the cooldown if a push is attempted too soon after the last one.

Design points to settle while building (deliberately not over-specified — use judgement and record the choice, e.g. in the commit message):
- Add a new knob to `.harness/harness.env` (mirroring existing knobs like `RL_BACKOFF_MIN`/`RL_BACKOFF_MAX`), e.g. `PUSH_COOLDOWN_SECONDS`, with a sensible default in the low hundreds of seconds (e.g. 300s/5min is a reasonable starting point — enough to stop a fast run of small tasks from bursting many pushes within a minute; err on the side of a bit more than seems necessary since the failure mode is silent). Read it in `loop.sh` with a fallback default so the harness still works if the var is unset.
- Persist the last-push timestamp somewhere that survives a loop restart — `supervise.sh` relaunches `loop.sh` on a ~5h15m cadence, and the loop also restarts after a rate-limit backoff sleep — an in-memory shell variable alone resets the cooldown clock on every restart, which defeats the point if a restart lands mid-burst. A small gitignored state file (e.g. `.harness/worklog/.last-push-ts`, alongside the existing gitignored `worklog/.failures.buf` scratch state) read/written by the helper is the simplest robust approach.
- The cooldown must NOT apply to the very first push of a fresh run (no prior timestamp on disk) — only throttle back-to-back pushes.
- Keep the change behaviourally inert when nothing is bursting: a project completing one task every few hours should never notice a cooldown; it should only matter during a rapid multi-task run.
- Do NOT touch the CI-gate/revert logic (`wait_ci_green`, red-CI revert) — this is purely about pacing WHEN a `git push` happens, not what happens after it.
- Do NOT build a post-push Vercel-deploy-status check (that was option (b) in the original idea and the owner explicitly chose this cooldown approach instead).
