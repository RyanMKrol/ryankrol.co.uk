## Overview
T324 (this session, done) made the WHOLE `.collection-review-controls` block (search+sort+pills together) always wrap onto its own row below the page title, sitewide. This follow-on is about the INTERNAL structure of that block: today SearchInput, SortButtons, and any pill row are all undifferentiated flex-wrap siblings inside `.collection-review-controls`, so a pill row can end up sharing a line with search/sort or wrapping unpredictably. The ask: search+sort always on one row, and any pill row always on its own row directly beneath.

## Do
In src/styles/globals.css, add a rule targeting `.collection-review-controls > .collection-pill-group` (the wrapper class every pill row already renders inside this block — confirmed via src/components/PillGroup.js, which always wraps its Pills in `<div className="collection-pill-group">`, and via src/pages/projects/index.js's inline tag-pill block at ~line 168 which uses the identical literal className) that sets `flex-basis: 100%;` so the pill row is forced to claim the full row width and drops to its own line under `.collection-review-controls`'s existing `flex-wrap: wrap` (declared at line ~452-458), leaving SearchInput (`.collection-search-input`) and SortButtons (`.sort-buttons-container`) to share the first row by default flex layout since neither has a forced flex-basis. This is a CSS-only change — do not touch any page's JSX. Confirmed via grep that exactly 4 of the 8 pages consuming `.collection-review-controls` currently render a `.collection-pill-group` inside it: reviews/perfumes (ownership pills), vinyl (view-toggle pills), projects (tag pills), and workouts (Push/Pull/Legs filter pills). Note workouts' `.collection-review-controls` contains ONLY the PillGroup — no SearchInput/SortButtons in that block (its DateRangeFilter lives in a separate `.search-container`/`.workout-filters` block outside `.collection-review-controls` entirely) — so the new rule is a harmless no-op there (a lone flex-basis:100% child changes nothing visually). reviews/{movies,tv,books,albums} render no pill row in this block at all and are unaffected by the new selector. Do not add `order` unless the spec review decides DOM-order independence is worth the added complexity — current DOM order in every page already places the pill row last, so `flex-basis: 100%` alone is sufficient without an `order` property.
