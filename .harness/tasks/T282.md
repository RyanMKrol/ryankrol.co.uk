## Do

Write a one-off migration script that converts existing `PerfumeRatings` rows from the old `considerTravelSize`/`considerFullBottle` booleans to the new `ownership` string field added by the dependency task (`perfume-ownership-field-1`, which adds `validateOwnership`/`VALID_OWNERSHIP` to `src/lib/perfumes.js` and changes the API round-trip). Model this script closely on the existing `src/scripts/migrateOpenLibraryBooksToGoogleBooks.js` (read it first) — same shape: `require('dotenv').config({ path: '.env.local' })`, a hardcoded `TABLE_NAME`, a `DynamoDBClient`/`DynamoDBDocumentClient` built the same way, **dry-run by default** gated on `IS_LIVE = process.env.LIVE === '1'` (prints what it would change, writes nothing unless `LIVE=1`), pure helper functions exported via `module.exports` for testing, and a `main()`/`if (require.main === module)` entry point.

Only production `PerfumeRatings` table exists (per `DYNAMO_TABLES.PERFUME_RATINGS_TABLE` in `src/lib/constants.js` — read the actual table name from there, don't hardcode a guess) and per prior investigation has only ~2 rows total, so this is deliberately small and simple — no batching/pagination needed, a single `ScanCommand` is enough.

1. **Derivation rule** (pure function, exported for tests): given a row's `considerTravelSize`/`considerFullBottle` booleans, compute the `ownership` value with this priority: if `considerFullBottle` is truthy → `'Full bottle'`; else if `considerTravelSize` is truthy → `'Travel size'`; else → `'Sample'` (the default/fallback when neither flag was set — treat a plain sample as the base case, since every reviewed perfume started as at least a sample). Name this function `deriveOwnership(row)`.
2. **Scan** the `PerfumeRatings` table for rows that have `ownership === undefined` AND (`considerTravelSize !== undefined` OR `considerFullBottle !== undefined`) — i.e. rows still on the old shape.
3. For each such row, compute `ownership` via `deriveOwnership`, and log what would change (old booleans → derived ownership) whether or not `LIVE=1`.
4. When `LIVE=1`, apply an `UpdateCommand` per row: `SET ownership = :ownership REMOVE considerTravelSize, considerFullBottle` (use `ExpressionAttributeNames` if needed to avoid reserved-word collisions, matching the pattern in `migrateOpenLibraryBooksToGoogleBooks.js`'s `updateReview`), keyed on the row's `id` (the `PerfumeRatings` partition key — NOT `title`/`author` like the Books migration, that script's composite key doesn't apply here).
5. Add an npm script entry to `package.json`'s `"scripts"` block, alongside the existing `"perfume:create-table"` line: `"perfume:migrate-ownership": "node src/scripts/migratePerfumeOwnership.js"`.
6. Add `src/scripts/migratePerfumeOwnership.test.js` testing the pure `deriveOwnership` function only (not the AWS calls) — mirror `migrateOpenLibraryBooksToGoogleBooks.test.js`'s style of testing exported pure helpers in isolation. Cover all three priority cases (`considerFullBottle: true` → `'Full bottle'` regardless of `considerTravelSize`; `considerTravelSize: true, considerFullBottle: false` → `'Travel size'`; both false/undefined → `'Sample'`).

Do not run the migration live yourself as part of this task (no real AWS credentials guaranteed in the build environment) — the code must be correct and dry-run-by-default; a human runs `LIVE=1 npm run perfume:migrate-ownership` afterward against production.

## Done when

- `src/scripts/migratePerfumeOwnership.js` exists, defaults to dry-run (no writes unless `LIVE=1`), reads the real `PerfumeRatings` table name from `DYNAMO_TABLES` (`src/lib/constants.js`), and exports `deriveOwnership` (plus its other helpers) for testing.
- `deriveOwnership` implements the stated priority (full bottle > travel size > sample) and is covered by `src/scripts/migratePerfumeOwnership.test.js` for all three cases.

- Running `node src/scripts/migratePerfumeOwnership.js` locally (dry run, no `LIVE` env set) against a table with a row shaped like `{ id: 'x', considerTravelSize: true, considerFullBottle: false }` prints the row it would change and the derived `ownership` value without making any AWS write calls.
