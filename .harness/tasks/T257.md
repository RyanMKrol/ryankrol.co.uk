## Do

Build two small pure/lib pieces and one new shared UI component, all generic and type-agnostic (no movie/tv/book/album-specific knowledge baked in) ‚Äî they will be consumed by four follow-up per-type pages (movies/tv/books/albums), which already depend on this task.

1. `src/lib/backfillQueue.js` ‚Äî export a function, e.g. `runBackfillQueue(items, searchFn, options)`, that:
   - Takes an array of arbitrary items, an async `searchFn(item) => Promise<candidates>` , and options `{ minSpacingMs = 1500, onItemStart, onItemSettled }`.
   - Processes items ONE AT A TIME (never concurrently) in array order, waiting at least `minSpacingMs` between the START of one `searchFn` call and the start of the next (do not just wait between promise *resolutions* ‚Äî spacing is about not bursting requests, not about total wall-clock throughput).
   - Calls `onItemStart(item)` right before starting a lookup, and `onItemSettled(item, { candidates, error })` when it resolves or rejects.
   - If `searchFn` rejects with an error shaped like a rate-limit response (assume the caller's `searchFn` throws an `Error` whose `.retryAfterSeconds` property is set when the caller detected a 429 ‚Äî document this contract in a comment so the per-type pages know to attach it), wait `max(minSpacingMs, retryAfterSeconds*1000)` and retry that SAME item exactly once before giving up and reporting it as an error via `onItemSettled`.
   - Return a controller object `{ cancel() }` that stops processing further items (in-flight request is allowed to finish, but no further items start) ‚Äî needed so a page unmount / page-change during an in-flight bulk run doesn't leak activity into a new page.
   - This is the single place that encodes 'be a good API customer' ‚Äî see this task's `design` field for the full rationale. Do not hardcode any particular page size assumption into this file; it must work correctly for any array length.

2. `src/lib/backfillEligibility.js` ‚Äî pure predicate functions, one per type, each taking a review item (as already returned by that type's existing `GET /api/reviews/<type>` route) and returning `true` if it still needs a metadata backfill:
   - `needsMovieBackfill(item)` ‚Üí `true` when `item.tmdbId` is missing/falsy.
   - `needsTvBackfill(item)` ‚Üí same field, `item.tmdbId` (TV records share the movies shape per CLAUDE.md's data model).
   - `needsBookBackfill(item)` ‚Üí `true` when the item has NEITHER `item.volumeId` (Google Books) NOR `item.olid` (Open Library) set ‚Äî i.e. no rich source metadata of any kind has ever been attached.
   - `needsAlbumBackfill(item)` ‚Üí `true` when `item.lastfm` is missing OR `item.lastfm.mbid` is falsy.
   Keep each function a one-line, easily-testable pure predicate.

3. `src/components/BulkBackfillList.js` ‚Äî a new, generic, dense list-view component (NOT a modal, and NOT built on top of `MetadataBackfillModal` ‚Äî that component is a single full-screen blocking dialog for one item at a time, unsuited to running N lookups inline at once; this is a deliberately separate, denser sibling). Props (mirror `MetadataBackfillModal`'s type-agnostic caller-supplies-everything shape):
   - `items` ‚Äî array of already-FILTERED (by the caller, using `backfillEligibility.js`) domain items needing backfill, each assumed to expose a stable `item.id`.
   - `pageSize` (default 15) ‚Äî paginate `items` using the existing `paginate()` helper from `src/lib/pagination.js` (do not reinvent pagination math); render page controls using the existing `src/components/Pagination.js` component for visual/behavioural consistency with the rest of the site.
   - `onSearch: async (item) => candidates[]` ‚Äî run through `backfillQueue.js`'s `runBackfillQueue` internally, scoped to only the CURRENT page's items (changing page enqueues a fresh queue run for the new page's items and cancels any prior in-flight queue via its controller).
   - `renderItemLabel: (item) => ReactNode` ‚Äî how to render the original item's identity (e.g. title/author/artist) in its row.
   - `renderCandidate: (candidate) => ReactNode` ‚Äî renders one candidate (same contract/shape as `MetadataBackfillModal`'s `renderResult` prop ‚Äî callers can reuse the same JSX they already wrote for the per-item modal).
   - `getCandidateKey: (candidate, index) => key` (optional, same contract as the modal's `getResultKey`).
   - `onConfirm: async (item, candidate) => void` ‚Äî caller-supplied save action (e.g. POST to the type's existing `/update` API); the component awaits it and marks that row 'saved' on success or shows an inline error and lets the owner retry that single row without affecting others.
   - Per-row UI (per the owner's explicit UX answer: a compact MULTI-candidate picker, not a single auto-confirmed top match): each row shows the original item's label, a status (queued / searching / N candidates found / no matches / saved / error), and once candidates arrive, a compact selectable list of a few candidates (radio-style selection, matching `MetadataBackfillModal`'s existing `.mbm-result-item` interaction pattern but laid out inline in the row rather than in an overlay) plus a per-row 'Confirm' button (disabled until a candidate is selected, shows a saving spinner state, disabled again once saved).
   - The component itself does not know about passwords, TMDB/Google Books/Last.fm specifics, or DynamoDB ‚Äî all of that stays in the per-type pages that consume it.

Write tests for `backfillQueue.js` (sequential ordering, spacing enforcement using fake timers, the retry-once-then-give-up-on-429 behavior, and `cancel()` stopping further items) and `backfillEligibility.js` (each predicate, both the missing-field and present-field cases) in their co-located `*.test.js` files. `BulkBackfillList.js` itself does not need a test (component/integration tests are a known pre-existing gap in this repo, not a bar this task should be first to clear).

## Done when

`npm test` passes and includes real coverage for `backfillQueue.js` (sequential execution ‚Äî never two searches in flight at once; the configured minimum spacing is actually enforced between request starts; a searchFn rejection carrying `.retryAfterSeconds` causes exactly one retry after waiting at least that long, then reports failure if it rejects again; `cancel()` prevents any further items from starting) and `backfillEligibility.js` (all four predicates, one true case + one false case each). `npm run lint` and `npm run build` both succeed. `src/components/BulkBackfillList.js` exists, is generic (grep it ‚Äî no hardcoded reference to `tmdb`, `googlebooks`, `lastfm`, or any DynamoDB table name anywhere in the file), uses `paginate()` from `src/lib/pagination.js` and renders `src/components/Pagination.js` for page controls, and its rows render a multi-candidate picker (not a single auto-confirm button) per the props contract described in `## Do`.
