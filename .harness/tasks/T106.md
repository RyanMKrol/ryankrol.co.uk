## Do

In this repo (`ryankrol.co.uk`, a Next.js site with an autonomous build harness under `.harness/`), the harness's local dashboard (`.harness/dashboard/server.js`, started via `npm run harness:dashboard`) lets the owner checkbox-select multiple backlog tasks and click a bulk "Mark N done" / "Mark N reviewed" button. Today this is wasteful: `POST /api/mark-done-bulk` and `POST /api/mark-reviewed-bulk` (server.js, current handler around what's now lines ~293-306, inside the `if (req.method === 'POST' && MUTATIONS.includes(url))` block) do:
```js
if (url === '/api/mark-done-bulk' || url === '/api/mark-reviewed-bulk') {
  const ids = Array.isArray(body.ids) ? body.ids : [];
  if (!ids.length) return sendJSON(res, 400, { ok: false, error: 'no ids' });
  const script = url === '/api/mark-done-bulk' ? 'mark-done.sh' : 'mark-reviewed.sh';
  const results = [];
  for (const id of ids) results.push({ id, ...(await runScript(script, [id])) });
  return sendJSON(res, 200, { ok: results.every((r) => r.ok), results });
}
```
— i.e. it shells out to `mark-done.sh` (or `mark-reviewed.sh`) once PER id via `runScript()` (defined a bit above, `execFile('bash', [path.join(HARNESS_DIR, script), ...args], ...)`). Each script invocation does its own full git cycle — acquire the harness's mkdir-based repo lock (`acquire_lock`/`release_lock`, sourced from `loop.sh` via `LOOP_SOURCE_ONLY=1 source "$HERE/loop.sh"`), write one id into the JSON overlay (`.harness/human-done.json` for mark-done, `.harness/reviews.json` for mark-reviewed), `git add`, `git commit -m "human-done: $id [skip ci]"` (or `reviewed: $id [skip ci]`), then a push-with-rebase-retry loop (`git fetch` / `git rebase origin/<MAIN_BRANCH>` / `git push`, 3 attempts). So selecting N tasks and clicking the bulk button today produces N separate commits and N separate pushes on `main` — this is confirmed root cause in the idea and bloats the harness's own git history for what is conceptually one owner action.

Read `.harness/mark-done.sh` and `.harness/mark-reviewed.sh` in full first — they are nearly identical siblings (same lock-acquire / jq-overlay-write / commit / push-retry shape, one on `human-done.json` with a `needs-human`-gate validation step, one on `reviews.json` with no such validation). Both currently accept exactly ONE task id as `$1` (plus an optional leading `--undo`).

**The fix — make each script's git plumbing (commit + push) run exactly ONCE per invocation, regardless of how many ids it's given, then make the bulk HTTP handlers pass the whole id list through in one call instead of looping:**

1. **`.harness/mark-done.sh`**: change its argument handling from a single `<TNNN>` to one-or-more ids: `mark-done.sh [--undo] <TNNN> [TNNN...]`. Keep `--undo` as a single leading flag applying to every id in the call (don't support mixing mark and undo in one invocation — that's out of scope). Validation must be **fail-fast and atomic**: before writing anything, loop over every id and check (same checks as today, per id): it matches `^T[0-9]+$`, it exists in `TASKS.json` (`tj -e --arg id "$id" '.tasks[]|select(.id==$id)'`), and its `gate` is `needs-human`. If ANY id fails validation, `die` immediately with that id named in the message — do this before touching `human-done.json` or acquiring the lock, so a bad id in a batch never partially applies. Once all ids validate, apply the overlay update for every id (same `jq '.[$id] = {done:true, at:$at}'` shape as today, just looped across ids into the same tmp file before it's moved into place — or an equivalent single-pass jq using all ids; either is fine as long as the RESULT is one JSON write). Then do the git portion exactly ONCE: one `acquire_lock`/`release_lock` (unchanged), one `git add`, one `git diff --cached --quiet` no-op check, one `git commit`, one push-retry loop (reuse the existing 3-attempt fetch/rebase/push loop verbatim — don't change its retry mechanics). Commit message: for a single id keep the EXACT existing format (`human-done: T123 [skip ci]` / `human-done: clear T123 [skip ci]`) so single-id usage is a no-op diff in behavior; for 2+ ids, join them with `, ` (e.g. `human-done: T123, T124, T125 [skip ci]`, or `human-done: clear T123, T124 [skip ci]` for `--undo`). Keep printing the existing per-id status line (`marked $id done` / `cleared human-done for $id`) for each id so CLI output stays informative — only the git operations are consolidated, not the console output.
2. **`.harness/mark-reviewed.sh`**: apply the exact same shape — multi-id args, one-pass overlay write across all ids, one commit (`reviewed: T123, T124 [skip ci]` / `reviewed: clear ... [skip ci]`), one push-retry. This script has no `needs-human` gate check (per its own comment, `reviewed` is a purely owner-facing annotation any real task id can get) — keep that; validation here is just "id matches `^T[0-9]+$` and exists in TASKS.json", fail-fast across all ids before any write, same as above.
3. **`.harness/dashboard/server.js`**: replace the `for (const id of ids) results.push({ id, ...(await runScript(script, [id])) });` loop with a SINGLE call: `const outcome = await runScript(script, ids);` (passing the whole `ids` array through as argv — `runScript`'s signature already takes an args array, no change needed there), then build the `results` array by mapping every id to that one shared outcome: `const results = ids.map((id) => ({ id, ...outcome }));`. This deliberately preserves the existing JSON response shape (`{ ok, results: [{id, ok, error|output}, ...] }`) that the dashboard's frontend JS already consumes unchanged (`markDoneBulk`/`markReviewedBulk` in the same file read `r.ok` and `(r.results||[]).filter(x=>!x.ok).map(x=>x.id+': '+x.error)`) — no frontend JS changes are needed or wanted. Update the stale comment on the line above this block (`// Bulk endpoints: loop the matching script over ids (each its own commit).`) to describe the new one-call/one-commit behavior instead.
4. **`.harness/README.md`**: the file table has one-line descriptions for `mark-done.sh` and `mark-reviewed.sh` (currently: "owner CLI: mark a `needs-human` task done → `human-done.json` (loop reads it; unblocks dependents)" and "owner CLI: mark a task reviewed → `reviews.json` (dashboard-only annotation; the loop ignores it)"). Tweak both to note they now accept multiple ids batched into a single commit, so the docs stay accurate (this repo's convention is docs land in the same commit as the behavior they describe).
