# T099 — Fix perfume slider thumbs not sitting flush at track start

## Do

The three `type="range"` sliders on the perfume forms (Rating in
`src/pages/reviews/perfumes/add.js:194`, and Longevity/Projection in
`src/components/PerfumeCharacteristics.js:17,38`) currently have **zero custom CSS** — confirmed
via a repo-wide check, there is no `input[type="range"]` styling anywhere in
`src/styles/globals.css` or inline in these components. They render with the browser's fully
native default appearance. This is the root cause of the reported bug: a native range input's
thumb is centered ON the value position by default, so at the minimum value the thumb sits inset
by roughly half its own width from the track's visual left edge, rather than flush against it —
this is standard browser behavior for an unstyled range input, not a bug in existing CSS (there is
none to have introduced the bug).

Add custom styling for these range inputs (a shared CSS rule/class in `globals.css`, e.g. targeting
`input[type="range"]` broadly or a shared class applied to all three sliders — your call on which,
but prefer ONE shared rule over three separate copies since all three sliders need the same fix)
so the thumb sits flush with the track's true start when the value is at its minimum, using the
site's existing theme CSS custom properties for colors (per this repo's theming convention — do
not hardcode colors here) rather than browser defaults. This will likely require styling the
thumb via `::-webkit-slider-thumb`/`::-moz-range-thumb` and the track via
`::-webkit-slider-runnable-track`/`::-moz-range-track` (removing default browser appearance via
`-webkit-appearance: none` etc. as needed) to get precise control over thumb positioning at the
track's edges.

This applies only to the production perfume forms listed above — do not touch anything under
`src/pages/v1/**`/`v2/**`/`v3/**` (those UI-rewrite variant trees are being deleted entirely by a
separate task, T092, so any sliders there are moot).
