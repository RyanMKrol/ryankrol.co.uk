The Projects page (`src/pages/projects/index.js`, rendered at `/projects`) lists the owner's GitHub repos as cards, sourced from `GET /api/github/repos` (`src/pages/api/github/repos.js`). That route's filter step deliberately keeps archived repos in the list (see the comment at `repos.js:73`, `// Filter out forks and private repos; archived repos are included`) but currently gives the frontend no way to tell which ones are archived, because the raw GitHub REST API response's native `archived: true|false` boolean is dropped during mapping. Add a small visual badge on each project card for repos where GitHub reports `archived === true`.

**1. Thread the field through the API route (`src/pages/api/github/repos.js`):**

In the `.map(repo => ({ ... }))` block at `repos.js:80-92`, add an `archived` key alongside the existing ones (`name`, `fullName`, `description`, `url`, `language`, `stars`, `forks`, `lastPush`, `createdAt`, `isPrivate`, `topics`). Normalize to a real boolean since the field may be absent in some mocked/edge-case payloads:

```js
archived: Boolean(repo.archived),
```

No other change is needed in this file — this is a pure passthrough of a field GitHub's API already returns on every repo object; no new HTTP call, no cache-key change (the cache key in `generateCacheKey('github-repos', { username: GITHUB_USERNAME })` is unaffected by this).

**2. Update the existing test (`src/pages/api/github/repos.test.js`):**

This file already mocks a `publicRepo` fixture (lines 22-35) that is fed through `handler`. Add a new test case (alongside the existing `'excludes private repos from the response'` test at line 80) that asserts the new field is threaded through correctly. Suggested shape: create a repo fixture identical to `publicRepo` but with `archived: true` (e.g. `{ ...publicRepo, name: 'archived-repo', full_name: 'testuser/archived-repo', html_url: 'https://github.com/testuser/archived-repo', archived: true }`), run it through `handler` alongside the existing non-archived `publicRepo` (which has no `archived` key at all, so it should map to `false`), and assert: the archived fixture's mapped entry has `archived === true`, and the plain `publicRepo` entry (no `archived` field on the raw fixture) has `archived === false` (not `undefined`) — this is exactly what `Boolean(repo.archived)` is for.

**3. Add the badge to the Projects page (`src/pages/projects/index.js`):**

This page already imports `Badge` from `../../components/Badge` (`index.js:4`) and uses it for GitHub topic pills inside `.project-card-tags` (around `index.js:253-261`), passing `accentColor="var(--accent-projects)"` `variant="soft"` `mono={false}` for that lowercase-Nunito plain-text treatment. Reuse the same `Badge` component for the archived indicator, but give it a visually distinct, muted/neutral color (do NOT reuse `--accent-projects`, which is already the page's active accent for topic pills/pill-group filters — an archived flag is a secondary status note, not a topic) — use the existing `--color-ink-mute` CSS custom property (defined in `src/styles/globals.css`, already used elsewhere for muted/secondary text) as the badge's `accentColor`.

Add the badge inside the existing `.project-card-meta-row` block (`index.js:231-251`), which already conditionally renders the language dot, 'updated ... ago' text, and fork count as sibling `<span>`s in a flex row with wrapping — add a new conditional entry to that same row, after the existing three, so it participates in the same wrap/gap layout with no new CSS needed:

```jsx
{repo.archived && (
  <Badge accentColor="var(--color-ink-mute)" variant="soft" mono={false}>
    archived
  </Badge>
)}
```

Use lowercase `archived` text to match this page's existing lowercase convention (page title, 'updated ... ago' meta text, topic pills are all lowercase). Do not add any new CSS — `.project-card-meta-row` is already `display:flex; align-items:center; gap:1rem; flex-wrap:wrap` (`globals.css:1809-1817`) and `Badge`'s own `.collection-badge`/`.collection-badge-plain` classes (`globals.css:2626-2643`) already render as a self-contained inline-flex pill, so it will drop into the row correctly with no layout changes.

Do not touch the sort/filter/search logic, the tag pill-group, or any other part of the page — this is additive only.
