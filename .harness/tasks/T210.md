## Do

Fix the Projects page grid (`/projects`, rendered by `src/pages/projects/index.js`) so cards no longer stretch to match their row's tallest neighbour, AND so a short card lets the following card visually flow up into the gap — real masonry/Pinterest-style reflow, not just per-card auto-height.

**Root cause (confirmed):** `.projects-grid` (`src/styles/globals.css:1738-1743`) is `display:grid; grid-template-columns: repeat(3, minmax(0,1fr)); gap:1.25rem`, with no `align-items` override, so CSS Grid's default `align-items:stretch` stretches every card in a row to match that row's tallest card (e.g. a short "Basket" card stretches to match its taller neighbours, leaving visible empty space at its own bottom). `.project-card` is at `src/styles/globals.css:1751-1760`; the collapse-to-1-col mobile breakpoint is the `@media (max-width: 900px)` block at `src/styles/globals.css:1745-1749`.

**Decision already made (do not re-litigate) — use CSS multi-column (`column-count`), NOT `align-items:start`-only, NOT `grid-template-rows: masonry`, NOT a JS-measured masonry approach:**
- `align-items: start` alone (keeping `display:grid`) only solves problem (1) — a card still occupies a FIXED grid cell (its own row/column), so a short card's neighbour below it (e.g. "Sprout" per the owner's example) cannot shift up into the freed space. The owner explicitly asked for the reflow, not just auto-height, so this alone is insufficient.
- `grid-template-rows: masonry` is real masonry but is currently Firefox-only (not in Chrome/Safari) — not viable for a public personal site with an unknown-browser audience.
- A JS-measured masonry approach (measure each card's rendered height, greedily place into whichever column currently has the least total height, e.g. via a small library or a custom `useLayoutEffect`) is the most faithful match to "Sprout shifts up" but is real added complexity for a personal-site projects page — a new client-side layout pass, resize-handling, and (per root `CLAUDE.md`) this repo's explicit "keep it simple... no new infra unless explicitly asked" convention argues against it here.
- **CSS multi-column (`column-count: 3` on `.projects-grid`) is the pragmatic choice and is what this task implements.** It requires zero new dependencies, is one CSS rule change, and gives genuine masonry-style behaviour for free: with `column-fill: balance` (the CSS default, which applies here since `.projects-grid` has no fixed height), the browser distributes cards across columns so total column heights stay roughly even, and a card immediately continues into the next available space in its column — i.e. a short card genuinely lets the following content shift up, satisfying the owner's actual complaint.

**On the reading-order caveat (checked against the actual page, not assumed):** `src/pages/projects/index.js:85` renders the subtitle "what I've been building on github · sorted by recent activity" — so recency ordering IS a real, visible claim on the page, not a hypothetical concern. However, CSS multi-column does not scramble order arbitrarily: cards are placed into columns in strict DOM order, filling column 1 top-to-bottom until the balance point, then continuing into column 2, then column 3 — i.e. within each column, cards still read top-to-bottom by recency (newest at the top of column 1). This is the same convention as a newspaper column or a Pinterest board and reads as reasonable for a personal projects showcase — it is not the same as "unsorted," it's a different (very common) traversal convention: read down a column first, then move to the next. Given the alternative correct-order-preserving option (JS-measured masonry) is disproportionate complexity/new-infra for this repo's stated philosophy, and `grid-template-rows: masonry` isn't broadly supported, accept this column-major reading order as the tradeoff for a working, dependency-free masonry effect. Do not attempt to preserve strict row-major reading order as part of this task.

**Implementation steps** (all in `src/styles/globals.css`, within the existing `/* ===== Projects page (T158) ===== */` block, lines ~1736-1804):
1. Replace `.projects-grid`'s `display:grid; grid-template-columns: repeat(3, minmax(0,1fr)); gap:1.25rem;` with a multi-column layout: `column-count: 3; column-gap: 1.25rem;` (keep `margin-bottom: 2rem;` as-is). Multi-column containers don't support `row-gap`/`gap` for vertical spacing between stacked items the way grid does — vertical spacing between cards within a column must come from a margin on `.project-card` instead (see step 3).
2. Update the `@media (max-width: 900px)` block (currently sets `grid-template-columns: 1fr`) to instead set `column-count: 1;` so the mobile/narrow-viewport behaviour (single column) is unchanged.
3. On `.project-card` (`src/styles/globals.css:1751-1760`): add `break-inside: avoid;` (prevents a single card's content from being split across two columns — a real risk without it) and `margin-bottom: 1.25rem;` (replaces the vertical spacing the old grid `gap` provided, matching the same 1.25rem value). Leave every other `.project-card` property (`display:flex`, `flex-direction:column`, `padding`, `border`, `border-radius`, `background-color`, the internal `gap:0.75rem` between a card's own children) untouched — the card's internal layout is not the problem, only its stretching/positioning within the outer container.
4. Do not touch `.project-card-top`, `.project-card-name`, `.project-card-stars`, `.project-card-description`, `.project-card-meta-row`, or any tag/badge styling — none of that needs to change.
5. Do not touch `src/pages/projects/index.js` — the fix is CSS-only, no JSX/markup changes needed.
6. Per root `CLAUDE.md`'s Definition of Done: this is a pure CSS layout change with no data-model, architecture, or page-behaviour implications, so no `CLAUDE.md`/`README.md` update is expected as a result of this specific change — but if you notice either doc describing this layout in a way that's now stale, fix it in the same commit.
