# T081 — Fix production crash: WorkoutCard assumes `exercises` always exists

## Do

**This fixes a live production crash on `https://www.ryankrol.co.uk/workouts` — highest priority.**

`src/components/WorkoutCard.js` unconditionally accesses `workout.exercises.length`,
`workout.exercises.map(...)`, and passes `workout.exercises` into `getTotalVolume(exercises)`/
`getTotalSets(exercises)` (both of which call `exercises.forEach(...)`), with no guard for
`exercises` being absent. Confirmed via a live production DynamoDB query: **72 of 367** real
`Workouts` table items currently have NO `exercises` field at all (they only have
`{start_time, end_time, id, title, exercise_count, description, created_at, updated_at}` — a
different, leaner shape than the ~295 items that DO have the full computed-metrics shape
`{..., durationMinutes, exercises, totalVolume, totalWarmupSets, totalWorkingSets,
uniqueExercises, workoutType}`). Rendering any of these 72 workouts throws
`TypeError: Cannot read properties of undefined (reading 'length')` inside `WorkoutCard`, which
Next.js surfaces as a fatal "Application error: a client-side exception has occurred" — blanking
the ENTIRE page (all workouts, not just the broken one), since React throws during render and
there's no error boundary isolating individual cards.

Fix `WorkoutCard.js` to render gracefully when `workout.exercises` is missing/undefined: treat it
as an empty array for the stat line (Exercises: 0, Sets: 0, Volume: 0kg) and don't render the
per-exercise breakdown section at all in that case (rather than crashing). Do NOT attempt to
compute or guess exercise data that isn't present — this task is a defensive rendering fix, not a
data-completeness fix (see the separate T082/T083 for backfilling the missing data itself).

A concrete real reproduction case (verified live moments ago): workout id
`5c4249f9-bd32-44ff-b7a5-b3edf05303f6` (title "Legs", `start_time: 2026-06-30T09:38:10+00:00`) is
one of the 72 incomplete records in production right now — though by the time this task is built,
newer/different workouts may be the incomplete ones instead, so don't hardcode this specific id;
build a component-level guard that handles ANY workout missing `exercises`, and verify by
constructing a test/fixture workout object shaped like the incomplete records
(`{id, title, start_time, end_time}`, no `exercises` key) and confirming `WorkoutCard` renders it
without throwing.

## Done when

- `WorkoutCard`'s `getTotalVolume`/`getTotalSets` helpers and its JSX no longer assume `exercises`
  exists — passing a workout object with no `exercises` key at all (e.g.
  `{id: 'x', title: 'Legs', start_time: '...', end_time: '...'}`) must not throw.
- When `exercises` is missing, the rendered card shows the workout's title/date/duration as normal,
  shows "Exercises: 0", "Sets: 0", "Volume: 0kg" (or equivalent zero-state), and does not attempt
  to render an exercise breakdown list.
- When `exercises` IS present (the normal/existing case), rendering is unchanged from current
  behavior — no regression for the ~295 complete records.
- Manually verify in a running dev server (`npm run dev`): temporarily render a `WorkoutCard` (or a
  small scratch page) with a workout object shaped exactly like the incomplete production records
  (no `exercises` key) and confirm it renders without a console error, then remove the scratch
  verification code before finishing — this repo has no component-test harness yet
  (`expectsTest: false`), so this manual check is the acceptance bar.
