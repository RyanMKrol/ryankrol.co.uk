## Overview
The album metadata backfill page re-selects albums that already have full Last.fm metadata because needsAlbumBackfill gates on lastfm.mbid, which Last.fm frequently returns empty and the write path stores as ''. Change the sentinel to a reliably-present field (lastfm.url) so enriched albums stop reappearing forever.

## Do
In src/lib/backfillEligibility.js, change needsAlbumBackfill from `(item) => !item.lastfm || !item.lastfm.mbid` to gate on lastfm.url instead: `(item) => !item.lastfm || !item.lastfm.url`, and update the doc comment accordingly (it currently says 'missing Last.fm metadata (no mbid)'). Then update the needsAlbumBackfill describe block in src/lib/backfillEligibility.test.js so it reflects the new sentinel: replace the mbid-based cases with url-based cases. Do NOT touch the movies/tv/books eligibility functions or their tests, and do NOT change any write path (add.js/update.js/backfill.js) or perform any data migration — the stored data is already correct, only the filter is wrong.

## Done when
needsAlbumBackfill returns false for an enriched album that has a lastfm object with a present url but an EMPTY mbid (e.g. { lastfm: { mbid: '', url: 'https://last.fm/...' } }); returns true for a truly un-enriched album with no lastfm object at all (e.g. { title: 'X' }); and returns true for an album that has a lastfm object but empty/absent url (e.g. { lastfm: { url: '' } }). The updated needsAlbumBackfill tests in src/lib/backfillEligibility.test.js assert exactly these three cases, and no test still asserts the old mbid-based behaviour for albums.
