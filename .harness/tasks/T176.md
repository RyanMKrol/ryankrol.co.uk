Root cause (confirmed by reading .harness/dashboard/server.js in full): there are TWO separate, independent 'expanded' states in this page, and only one of them is preserved across the 5s poll. (1) The outer per-task row expansion is a CSS class: `toggle(id)` (around line 212-215) adds/removes the `open` CLASS on `#task-<id>` (`.task.open .body { display:block }`, line 164), and `window.__openIds` (declared line 182) records which task ids are expanded; `render()` (lines 283-310) rebuilds `#main`'s `innerHTML` wholesale every poll and then, at line 304, re-applies `classList.add('open')` to every id in `__openIds` — so the row itself correctly stays open across a refresh. (2) Nested INSIDE that row's `.body`, when a task has a worklog or audit file, `task()` (lines 261-282) emits `<details class="buildlog"><summary>Build log</summary><pre>...</pre></details>` (worklog: line 279, audit: line 280) — a native HTML `<details>` element with its OWN independent open/closed state (the browser toggles its `open` IDL attribute natively when you click the `<summary>`; there is no JS handler on it at all today). Because `render()` replaces `#main.innerHTML` wholesale every 5s, EVERY `<details>` element is a brand-new DOM node with no `open` attribute, and nothing in the codebase tracks or reapplies `<details>` open state — `__openIds`/line 304 only ever touches the `.task` div's CSS class, never the nested `<details>` elements' native `open` attribute. So: the row expansion (spec/scope/depends-on) survives a refresh, but the build log/audit `<details>` the owner actually wants to keep reading always snaps shut on the very next 5s poll, regardless of `__openIds`. Fix: track nested `<details class="buildlog">` open state the same way `__openIds` tracks row state, then reapply it after each render. Concretely, in the PAGE template's <script> block in .harness/dashboard/server.js: (a) near `window.__openIds = new Set();` (line 182) add a second set, e.g. `window.__openLogIds = new Set();`, plus a handler `window.onLogToggle=function(el){ if(el.open) window.__openLogIds.add(el.id); else window.__openLogIds.delete(el.id); };` that responds to the browser's native `toggle` event on a `<details>` element; (b) in `task()`, give each buildlog/audit `<details>` a stable, task-and-kind-scoped id and wire the native toggle event to the new handler, e.g. change line 279 to `'<details class="buildlog" id="log-'+t.id+'-build" ontoggle="onLogToggle(this)"><summary>Build log</summary><pre>'+esc(t.worklogContent)+'</pre></details>'` and line 280 similarly with `id="log-'+t.id+'-audit"'`; (c) in `render()`, alongside the existing `window.__openIds.forEach(...)` reapply loop (line 304), add a second loop that reapplies the native open state to any `<details>` whose id is in `__openLogIds`, e.g. `window.__openLogIds.forEach(id=>{ var el=document.getElementById(id); if(el) el.open = true; });`. This mirrors the existing `__openIds` pattern (a JS-side Set surviving the innerHTML replace, reapplied post-render) but targets the native `<details open>` attribute instead of a CSS class, since that's the actual mechanism controlling build-log visibility. Do not remove or change the existing `__openIds`/`toggle(id)` row-level mechanism — it already works correctly and must keep working alongside this addition. Keep the fix confined to the PAGE template string inside server.js; no other file needs to change (there is no lib.js involvement — lib.js only computes backlog data, not the dashboard's client-side rendering).
