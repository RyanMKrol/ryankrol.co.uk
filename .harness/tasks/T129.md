TMDB and Google Books are different: both already return relevance-sorted results with no code change needed, confirmed via each provider's own API documentation (checked July 2026):

- TMDB `/search/movie` and `/search/tv` (called from `src/pages/api/tmdb/search.js`) do not support a `sort_by` parameter at all. TMDB's own documentation and support forum confirm the search endpoints are sorted by relevance while being boosted by popularity, with no way to change that via a query param (`sort_by` is only available on the different `/discover/*` endpoints, which this app doesn't use). So there is no missed relevance param on TMDB's side — the code already gets TMDB's best-effort relevance order.
- Google Books `/books/v1/volumes` (called from `src/pages/api/books/search.js`'s `searchGoogleBooks`) supports an `orderBy` param with values `relevance` (the default) and `newest`. The code never sets `orderBy`, so it already defaults to `relevance` — again, nothing missed.

So for these two providers, there is no analogous "flip a parameter" or "add a dedup pass" fix that's obviously correct and low-risk the way the Last.fm one was. The remaining option is a different, riskier kind of change: re-ranking results ON TOP OF what each provider already decided is most relevant — e.g. boosting exact-title matches to the top, sorting by TMDB's `popularity` field (not currently even kept by `mapTmdbResult` in `src/lib/tmdb.js`), or de-duping Google Books results that represent multiple editions/ISBNs of the same underlying book (not currently deduped by `mapGoogleBooksResult` in `src/lib/googlebooks.js`). Overriding the order the upstream API already decided is a judgment call about whether a homemade heuristic would actually beat TMDB's/Google's own ranking, and it's easy to imagine specific cases (like the owner's "Annihilation" edit-page example, where a legitimate-looking "Annihilation (2025-10-23)" result and several differently-titled films like "Mortal Kombat: Annihilation" rank ahead of the 2018 film) getting worse rather than better without more examples of what "right" looks like.
