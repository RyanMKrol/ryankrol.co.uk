# T079 — Write a migration script: re-fetch Open-Library-sourced books from Google Books

## Do

Existing book reviews in `BookRatingsV3` that were originally sourced via Open Library carry
`source: 'openlibrary'`, `olid`, and/or `coverId` fields (no `volumeId`, no Google-Books-specific
fields). The owner wants these migrated to Google Books data instead. Write a new script
`src/scripts/migrateOpenLibraryBooksToGoogleBooks.js`, following this repo's existing migration
script conventions (see `src/scripts/migrateAlbumsToV2.js` for the general style — direct
`docClient`/AWS SDK usage, `.env.local` credentials, clear `console.log` progress output — though
this script updates records IN PLACE in `BookRatingsV3`, it does not migrate between tables).

**This script MUST be dry-run by default** (prints what it would change, writes nothing) and only
perform real writes when explicitly invoked with a live flag (e.g. `LIVE=1 node
src/scripts/migrateOpenLibraryBooksToGoogleBooks.js`) — this is real user data being changed based
on an automated, potentially-ambiguous title/author match against an external API, so it must
never write silently. For each book review with `source === 'openlibrary'`:
1. Search Google Books using the review's `title` + `author` (the same query shape
   `src/pages/api/books/search.js`'s `searchGoogleBooks` function uses, or reuse that function
   directly if practical — see `src/lib/googlebooks.js`'s `mapGoogleBooksResult` for the result
   shape: `{ source, volumeId, title, authors, firstPublishedYear, coverUrl, isbn, subjects,
   pageCount, publisher }`).
2. If a confident single top match is found (e.g. the top result's title matches closely — use
   your judgement on a reasonable match heuristic, log any ambiguous/no-match case instead of
   guessing), update the review record: set `source: 'googlebooks'`, `volumeId` (from the match),
   `coverUrl`, `bookAuthors` (from the match's `authors` array), `firstPublishedYear`, `isbn`,
   `subjects`, `pageCount`, `publisher` — and REMOVE the now-inapplicable `olid`/`coverId` fields
   (Open-Library-specific identifiers that don't apply to a Google Books record).
3. If no confident match is found, leave that record untouched and log it clearly as "no match —
   left as openlibrary" in the script's output, so the owner can review/handle those manually.
4. Never touch `title`, `author`, `rating`, or `overview` (the user's own review text) — only the
   source-metadata fields listed above.

Add an npm script for it in `package.json`, e.g. `"books:migrate-openlibrary": "node
src/scripts/migrateOpenLibraryBooksToGoogleBooks.js"`, matching this repo's existing script-naming
convention (see `"perfume:create-table"`, `"workout:migrate-data"`).

**Verification for THIS task must use mocked/fixture data, never the real production DynamoDB
table or real Google Books API calls** — write a unit test (or a small standalone test harness
script, whichever fits this repo's existing test conventions better) that feeds the migration
logic a few fixture "book review" objects and a mocked Google Books search response, and asserts
the correct dry-run output and the correct field-mapping logic, without ever touching real AWS
credentials or the live Google Books API. The actual live run against production is a separate,
manual, human-gated step — NOT part of this task's "Done when" (see T080).

## Done when

- `src/scripts/migrateOpenLibraryBooksToGoogleBooks.js` exists, defaults to dry-run (no writes
  without `LIVE=1`), and implements the search/match/update/skip logic described above.
- A test using mocked/fixture data (not real AWS/Google Books calls) confirms: a fixture record
  with a confident match gets the correct field mapping computed (source/volumeId/coverUrl/
  bookAuthors/firstPublishedYear/isbn/subjects/pageCount/publisher set, olid/coverId cleared); a
  fixture record with no good match is left alone and logged as unmatched; `title`/`rating`/
  `overview` are never present in the computed update for any fixture. `npm test` passes.
- `package.json` has a new npm script to run this migration, following the existing naming
  convention.
- This task's own verification made zero real AWS or Google Books API calls.
