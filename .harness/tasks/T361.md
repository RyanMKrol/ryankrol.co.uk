## Overview
This repo's book search currently only calls Google Books (src/pages/api/books/search.js -> searchGoogleBooks -> mapGoogleBooksResult in src/lib/googlebooks.js). A prior scoped trial (reverted, uncommitted) proved a Hardcover.app GraphQL search works well as a replacement provider: POST https://api.hardcover.app/v1/graphql with an `authorization: Bearer <HARDCOVER_API_TOKEN>` header (token already sits in .env.local), query shape `search(query: $q, query_type: "Book", per_page: 20) { results }` returning `data.data.search.results.hits[]`, each `hit.document` carrying `id`, `title`, `author_names[]`, `release_year`, `image.url` (cover — present directly on the search hit, no separate lookup needed), `isbns[]`, `genres[]`, `pages`. There is no `publisher` field in Hardcover's Book search index. This unit makes Hardcover the search backend for /api/books/search.js, mirroring the existing Google Books code shape exactly (same withApiCache wrapping, same rate-limit gate, same error/429 propagation contract that src/lib/backfillQueue.js's retry logic depends on via `.retryAfterSeconds`).

## Do
Create src/lib/hardcover.js exporting `mapHardcoverResult(hit)` that normalises one search hit's `.document` into the exact same common shape src/lib/googlebooks.js's mapGoogleBooksResult already returns: `{ source: 'hardcover', volumeId, title, authors, firstPublishedYear, coverUrl, isbn, subjects, pageCount, publisher: [] }` — mapping `id`->volumeId, `title`->title, `author_names`->authors (default []), `release_year`->firstPublishedYear, `image.url`->coverUrl (null if absent), `isbns`->isbn (default []), `genres`->subjects (default []), `pages`->pageCount (null if not a number), `publisher` always `[]` since Hardcover's index has no such field. Add src/lib/hardcover.test.js covering: a well-formed hit maps every field correctly; missing/null `image`, `author_names`, `isbns`, `genres`, `pages` each degrade to the same defaults mapGoogleBooksResult uses (null/[] as appropriate) rather than throwing. Then rewrite src/pages/api/books/search.js: replace the body of `searchGoogleBooks` with a new `searchHardcover(title, author)` that POSTs the GraphQL query above with `Content-Type: application/json` and the Bearer token from `process.env.HARDCOVER_API_TOKEN`, folds `title`+`author` into a single query string the same way the existing function does (e.g. `${title} ${author}`.trim() since Hardcover's search is a single free-text field, not separate title/author query params), extracts `data.data.search.results.hits`, and maps each hit through `mapHardcoverResult`. Keep every existing behavioural contract identical: same `withApiCache`/`generateCacheKey('book-search', {title, author})` wrapping, same `checkRateLimit`/`getClientIp` gate at the top, same 400 on missing `title`, same console.log/console.error style (swap the '📚 [GoogleBooks]'/'❌ [GoogleBooks]' prefixes for '📚 [Hardcover]'/'❌ [Hardcover]'), same 429-detection-and-Retry-After-header contract on upstream failure (Hardcover's docs cite a 60 req/min limit — treat any non-2xx with a 429 status the same way the Google Books path already does, propagating `Retry-After`). If `HARDCOVER_API_TOKEN` is unset, log a console.warn (mirroring the existing GOOGLE_BOOKS_API_KEY warn) and let the request fail naturally with a clear error rather than silently falling back to Google Books (fallback behavior, if any, is a separate decision — this unit assumes replacement, not dual-provider). Update CLAUDE.md's Key env vars table entry for HARDCOVER_API_TOKEN to drop the 'Trial provider, books Bulk Backfill only' / 'not yet wired into BookSearch.js' caveat (superseded by this + the next unit), and update CLAUDE.md's file map row for src/pages/api/books/search.js to describe it as calling Hardcover instead of Google Books. Do not delete src/lib/googlebooks.js or GOOGLE_BOOKS_API_KEY in this unit — that removal-vs-fallback decision is explicitly deferred to the needs-human unit in this same idea.

## Done when
GET /api/books/search?title=<X>&author=<Y> returns results sourced from Hardcover's GraphQL API (verified via a manual local `npm run dev` request against a known title), each result shaped identically to the old Google Books response (same field names/types BookSearch.js and backfill.js already consume), `npm test` passes including the new src/lib/hardcover.test.js covering well-formed and degraded/missing-field hits, `npm run lint` and `npm run build` are clean, and CLAUDE.md's env var table + file map row for this route reflect the new Hardcover-backed behaviour.
