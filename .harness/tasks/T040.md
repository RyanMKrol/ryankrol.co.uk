# T040 — Harness dashboard rework — bucket grouping + sort order (data layer)

## Do

Rework `computeBacklog`/`summarize` in `.harness/dashboard/lib.js` to group tasks into exactly
**four** display buckets instead of the current seven (`ready`, `waiting-human`, `waiting-loop`,
`needs-human`, `failed`, `blocked`, `done`), matching the `local-jobs` sibling repo's dashboard
Backlog page (`~/Development/local-jobs/dashboard/app/backlog/page.tsx`) as the reference model:

- **`ready`** — unchanged: the existing `ready` bucket logic (not done, ungated, all deps done).
- **`waiting`** — the existing `waiting-human` bucket logic UNCHANGED (a task whose unmet-dependency
  closure transitively contains a needs-human/gate/failed blocker — keep this repo's existing
  recursive `blockedByHuman` check exactly as-is; it is already more thorough than local-jobs'
  immediate-dependency-only check, so do not weaken it). The existing `waiting-loop` bucket
  (unmet deps that are all themselves buildable) must be **entirely omitted from every bucket** —
  do not display these tasks anywhere. (This matches local-jobs' own explicit design: a task
  blocked only by another buildable task is "noise... the loop will get to its blocker on its
  own" — it's neither ready nor waiting from the owner's point of view.)
- **`needsHuman`** — the union of the current `needs-human` bucket AND the current `blocked`
  bucket (a task the loop gave up on after repeated failures, flagged `failed:blocked` in its
  worklog). Both need the owner's attention, so both belong in this one section, but each task
  must carry a flag so the UI (T041) can render a distinguishing pill for a loop-given-up task
  vs. a gate/needs-human task (e.g. keep the existing `blocked: true` field on the task object so
  T041 can check it).
- **`done`** — the union of the current `done` bucket AND the current `failed` bucket (an
  owner-marked false success, terminal). Sort this bucket's array **not-reviewed items first,
  then by numeric task id** (parse the numeric part of `T123` → `123`) — this is the fix for a
  separate, related backlog idea about sort order; implement it here, in this bucket's ordering,
  rather than as a separate task. A `failed` task must keep its existing `failed: true`/`reviewed`
  fields so T041 can render its distinct pill.

Update `.harness/dashboard/lib.js`'s `summarize()` to report counts for the new four-bucket model
(`ready`, `waiting`, `needsHuman`, `done` — rename/consolidate from the current
`{ready, 'waiting-human', 'needs-human', failed, blocked, 'waiting-loop', done}` shape). Keep
`isDone`/`deriveTask` as-is unless a field needs adding to support the above (e.g. keeping
`blocked`/`failed`/`reviewed` flags available on each task object — these already exist, just
make sure none are dropped by the bucket-collapsing change).

- `computeBacklog(tasks, opts)` returns each task tagged with one of exactly four bucket values:
