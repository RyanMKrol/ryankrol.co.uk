## Overview
Idea #2 wants existing Google-Books/Open-Library-sourced metadata wiped from BookRatingsV4 once Hardcover is the sole provider, so old-provider data doesn't linger and so those books become newly eligible for a clean Hardcover-sourced re-backfill (clearing volumeId and olid flips needsBookBackfill(item) back to true per src/lib/backfillEligibility.js). This must mirror the dry-run/LIVE=1 discipline of src/scripts/stripBackfillEditedDates.js exactly, since the live pass is a real, destructive write across production DynamoDB data with no undo.

**Scope of THIS task: the migration script, its unit tests, its (read-only) dry run, and the docs — NOT the destructive production run.** The actual `LIVE=1` execution against production is an irreversible, no-undo data deletion that must be triggered deliberately by a human, not run autonomously by the build loop; it is tracked as its own needs-human follow-up (T370, which dependsOn this task). So this task is DONE when the script exists, is correct, its dry run works read-only, its tests pass, and the docs are updated — the live wipe is explicitly out of scope here and its absence must NOT be treated as an unmet criterion. (History: an earlier authoring of this task put the LIVE run in `## Done when` while the Overview said to defer it — a self-contradiction that made the task unbuildable and blocked it; this rewrite removes the live execution from the acceptance criteria and hands it to T370.)

## Do
Create src/scripts/stripOldBookProviderMetadata.js following src/scripts/stripBackfillEditedDates.js's exact structure and conventions: dotenv-loads .env.local, builds a DynamoDBDocumentClient the same way, defaults to a dry run that only logs, and only performs real writes when `LIVE=1` is set in the environment. Scan DYNAMO_TABLES.BOOK_RATINGS_TABLE (from src/lib/constants.js) for every row that has any of these fields present: `source`, `coverUrl`, `volumeId`, `bookAuthors`, `firstPublishedYear`, `isbn`, `subjects`, `pageCount`, `publisher`, plus the legacy Open-Library-era `olid`/`coverId` fields (per root CLAUDE.md's Data model notes on historical rows). For each such row, build an UpdateCommand with a `REMOVE` UpdateExpression naming exactly the fields present on that row (don't blindly REMOVE fields that were never set — mirror stripBackfillEditedDates.js's targeted approach). Log each row's title and which fields it's stripping, both in dry-run (no write) and live (write + confirmation) modes, exactly matching the existing script's console.log style/emoji conventions. The `LIVE=1` code path must be written and correct — this task builds the whole script — but you do NOT execute `LIVE=1` here (that's T370, a human step); running the default read-only dry run to prove it works is expected and safe. Add src/scripts/stripOldBookProviderMetadata.test.js unit-testing the pure decision logic (which fields a given row shape would have stripped, and that a row with none of the target fields is left untouched) the same way stripBackfillEditedDates.js exports `shouldStrip` for testability — export an equivalent pure helper (e.g. `fieldsToStrip(item)`) from this new script. Update README.md's BookRatingsV4 schema section to note that `source`/`coverUrl`/`volumeId`/etc. are Hardcover-only now (once hardcover-migration-a/b/c have shipped) and that legacy Google-Books/Open-Library values are removed from every row by this migration once its live pass (T370) is run.
