## Overview
src/pages/reviews/books/backfill.js's handleSearch currently sends `provider: 'googlebooks'` as an explicit query param to /api/books/search (a leftover from the earlier scoped trial's provider-branching design). Since hardcover-migration-a removed provider branching from the API route entirely (Hardcover is now the only backend), this stale param must be removed. Separately, the idea explicitly asks this bulk sweep be a 'good API consumer' of Hardcover's documented 60 req/min limit — src/lib/backfillQueue.js (used internally by BulkBackfillList, which this page already uses) already paces one search at a time with a `minSpacingMs` option (default 1500ms) plus automatic 429 backoff/retry via `.retryAfterSeconds` — so this is a tuning task on an existing mechanism, not new pacing infrastructure.

## Do
In src/pages/reviews/books/backfill.js's handleSearch, remove the now-meaningless `provider: 'googlebooks'` entry from the URLSearchParams passed to /api/books/search (the route no longer branches on it after hardcover-migration-a). Confirm the 429-detection block (reading `Retry-After` into `err.retryAfterSeconds`) still matches the contract hardcover-migration-a's rewritten route emits on a Hardcover 429 — no change needed if the contract was preserved, but verify by reading the updated search.js. For pacing: BulkBackfillList.js doesn't currently expose a way to override backfillQueue's default `minSpacingMs` (1500ms) — thread a `minSpacingMs` prop through BulkBackfillList.js (defaulting to the current 1500ms if not passed, so no other caller changes behavior) down into its `runBackfillQueue(...)` call, then pass an explicit `minSpacingMs={1200}` (or similar value comfortably under Hardcover's 60-req/min ceiling — 1000ms is the bare minimum for 60/min, so use one with headroom, e.g. 1100-1200ms) from backfill.js. Document the choice with a one-line comment in backfill.js noting Hardcover's 60 req/min limit is the reason for the explicit value.
