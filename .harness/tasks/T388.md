## Overview
The user reports that on the site's shared Markdown renderer (src/components/Markdown.js), a single newline (one Enter) between two lines of text produces no visible line break — it takes a full blank line (two newlines) to get one line of visible spacing. This is standard CommonMark 'soft break' behaviour (a single newline collapses to a space; only a blank line starts a new paragraph), but it mismatches the user's plain-text/textarea mental model where one Enter should mean one line break. This is DIFFERENT from T382 (already shipped): T382 made runs of 2+ *blank* lines render proportionally more vertical gap via a `preserveBlankLines` preprocessing step in block mode only — it deliberately left single-newline-as-space untouched, and this task should not undo or duplicate that. There are three genuinely different ways to fix the single-newline complaint, each with real trade-offs and different blast radius, so which one to build is an owner decision, not something to guess:

(a) **Adopt `remark-breaks`** (a remark plugin) in src/components/Markdown.js's block-mode ReactMarkdown so a single newline renders as a `<br>`, matching 'one Enter = one line break'. Trade-off: this changes how EVERY multi-line piece of prose wraps, site-wide, across every one of the ~25 <Markdown> call sites (movie/TV/book/album/perfume review bodies, hot takes, /tom's Top-of-mind note, perfume descriptions) — including any existing review text that was written expecting CommonMark's collapse-to-space behaviour, which could visually break line-wrapping/paragraph flow in already-published reviews. Broadest blast radius, but also the most literal fix of the complaint and needs no per-site changes.

(b) **A narrower, Markdown.js-scoped preprocessing patch**, structurally similar to T382's `preserveBlankLines` helper: detect single newlines that are NOT already blank-line paragraph breaks and convert them into an explicit break (e.g. a trailing double-space + newline, or a raw `<br›`, before handing the string to ReactMarkdown), in block mode only, leaving inline mode (ReviewCard's track-highlights usage) exactly as-is per the existing INLINE_ELEMENTS/unwrapDisallowed convention. Trade-off: much smaller/more surgical blast radius (one component, one clearly-scoped preprocessing function, easy to unit test like `preserveBlankLines`), but still changes rendering behaviour for all existing content using block mode, and needs careful interaction with `preserveBlankLines`'s existing blank-line-run logic so the two preprocessing passes compose correctly rather than double-counting gaps.

(c) **Leave rendering as CommonMark-standard and instead fix the /tom editor's UX/help text** (src/pages/tom.js, and possibly MarkdownEditor.js's toolbar) to set the correct expectation — e.g. inline help copy near the textarea explaining 'press Enter twice for a new line' — with no rendering change anywhere. Trade-off: zero risk to existing published content and zero blast radius outside /tom, but doesn't actually change the behaviour the user called 'wrong'; it only better explains it, which may or may not satisfy the original complaint.

The owner needs to pick exactly one of (a)/(b)/(c) (or an owner-specified variant) before a buildable implementation task can be shaped, because scope, files touched, and expected diff size are entirely different for each.
