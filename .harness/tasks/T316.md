## Overview
T314 (commit 66e73e1) fixed the ARCHIVED tag's corner-radius mismatch on /projects cards but introduced a regression: it changed `.project-card-archived-tag`'s `top`/`left` from `0` to `var(--border-width)` (2px), reasoning this 'matches the effective behaviour it already had.' T314's own worklog (.harness/worklog/T314.md) records the getBoundingClientRect() measurements that disprove that reasoning: with `top: 0; left: 0`, the tag's measured origin was already offset from the card's outer origin by exactly one --border-width, because an absolutely-positioned child's top/left:0 resolves against its positioned ancestor's padding box (whose origin is the *inner* face of the ancestor's border) — not the ancestor's outer border-box origin that getBoundingClientRect() reports for the card. So `top: 0; left: 0` already sat the tag flush against the inside of the card's border with zero extra gap; the only real defect (which T314 correctly fixed) was the corner *radius* not accounting for that implicit inset. T314's added explicit `top: var(--border-width); left: var(--border-width)` stacks a second, redundant 2px on top of the implicit one, pushing the tag inward and leaving a visible ~2px white gap between the border and the tag along the straight top/left edges (see idea 8's screenshots: RatingsPlotter-Site, HuelPicker cards) — the corner curvature itself (fixed via calc(var(--radius-card) - var(--border-width)) on border-top-left-radius) is unaffected by this and should NOT be touched.

## Do
In src/styles/globals.css, in the `.project-card-archived-tag` rule (currently ~line 1805-1818), change `top: var(--border-width); left: var(--border-width);` back to `top: 0; left: 0;`. Leave `border-top-left-radius: calc(var(--radius-card) - var(--border-width))` and `border-bottom-right-radius: var(--radius-card)` exactly as T314 left them — those are correct and unrelated to this bug. Do not touch `.project-card`'s own border/radius/position rules. If you want to double-check the box-model reasoning empirically before committing, you can reproduce T314's own method: run the dev server, open /projects, inject an `.project-card-archived-tag` element into a `.project-card` via devtools (or use a Playwright script) if no repo in the live data happens to be archived, and compare `getBoundingClientRect()` of `.project-card` vs `.project-card-archived-tag` at top:0/left:0 vs top:2px/left:2px to confirm which produces zero gap against the border's inner edge.
